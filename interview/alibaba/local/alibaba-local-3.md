### 面试中的问题：

#### （1）自我介绍

**答题思路**：用 1 分钟讲清「你是谁 + 技术亮点 + 为什么匹配」。建议结构：教育背景 → 最拿得出手的 1~2 个项目及你的角色和量化产出 → 熟悉的技术栈 → 想来该岗位的原因。多用与 JD 相关的关键词，避免流水账。

#### （2）选择前端的原因

**答题思路**：给出真诚且有说服力的理由，避免「因为前端简单」。可从三方面组织：**成就感**（代码效果直观可见、能直接影响用户体验）、**技术契合**（喜欢工程化、交互设计与用户体验的结合）、**发展空间**（前端边界在扩展，如 Node、跨端、可视化、AI 应用）。最好结合一个让你入坑的具体经历。

#### （3）学习前端的方法

**答题思路**：体现**主动性和体系化**。可讲：以官方文档 + 源码为主打基础、通过做项目在实战中查漏补缺、遇到问题深挖原理而非只求跑通、关注社区（GitHub、技术博客）跟进新特性。举一个「通过某项目/某源码学到了什么」的具体例子会更有说服力。

#### （4）React文档的学习体会

**答题思路**：这题在考察你对 React 理解的深度。可结合新版文档（react.dev）的核心理念展开：

+ **纯函数与不可变性**：组件渲染应是纯函数，不在渲染中产生副作用、不直接修改 state/props。
+ **StrictMode 双调用**：开发模式下 React 会故意双调用渲染和部分函数，帮你暴露副作用问题。
+ **useState 原理**：state 是快照，同一次渲染中值不变；连续 `setState` 传值会被覆盖，应传更新函数 `setX(x => x+1)`。

结合自己踩过的坑来讲会更真实。

#### （5）YAML文档生成工具的实现

**答题思路**：这是对你项目的追问，按工具的实现链路讲清楚。核心是**解析与代码生成**：用 YAML 解析库把文档解析成 JS 对象（AST/数据结构）→ 遍历数据结构，套用模板（如 `docx`/`officegen` 库或字符串模板）→ 输出目标产物（Word / 接口代码）。**动机**要点明：接口定义作为单一数据源，自动生成文档和代码，减少前后端手动同步、避免不一致。

#### （6）YAML文档在前端开发中的应用

**答题思路**：讲清 YAML 作为**接口契约（Single Source of Truth）**带来的效率提升。基于统一的接口定义（类似 OpenAPI/Swagger）可自动生成：TS 类型定义、请求函数（axios 封装）、Mock 数据。好处是**前后端约定统一、类型安全、改接口只改一处**，减少手写样板代码和低级错误。

#### （7）后端如何利用YAML文档

**答题思路**：说明同一份 YAML 契约在后端侧的价值。后端可据此生成：路由/Controller 骨架、参数校验（DTO）、数据库模型、以及对外的接口文档。前后端**共用一份定义**，保证请求/响应结构一致，接口一改双方同步，减少联调时的对不上问题。

#### （8）前后端接口打通的成熟方法

目前主流是**基于契约（Schema）的协作**：

+ **OpenAPI / Swagger**：定义接口规范，配合代码生成器产出前端请求代码和 TS 类型、后端骨架。
+ **Apifox / YApi**：接口管理 + Mock + 自测一体化，前端可先基于 Mock 并行开发。
+ **GraphQL**：由前端按需查询，强类型 Schema 天然是契约。

常见问题是**接口变更不同步**，解决靠契约作为单一数据源 + 自动生成 + Mock 解耦前后端进度。

#### （9）实习项目经验分享

**答题思路**：用 STAR 结构讲一个有产出的项目：项目背景和你的角色 → 负责的核心模块 → 遇到的技术难点及你的解法 → 量化成果（性能、效率、复用率提升）。突出你**独立解决问题**和**在团队协作中的贡献**，不要泛泛而谈。

#### （10）移动端适配方案

常见适配方案：

+ **rem + 动态根字号**：JS 或媒体查询根据屏宽设置 `html` 的 `font-size`，元素用 `rem`（如 `lib-flexible + postcss-pxtorem`）。
+ **vw/vh**：直接按视口百分比布局，无需 JS。
+ **flex 弹性布局**：配合百分比适配不同屏宽。
+ **响应式（媒体查询）**：不同断点不同样式。

Tailwind CSS 提供原子类和内置响应式断点前缀（`sm:`/`md:`），用工具类快速写适配样式，减少手写 CSS 和命名负担。

#### （11）样式污染的避免方法

CSS 全局作用域易导致命名冲突和污染，常见隔离方案：

+ **CSS Modules**：编译时给类名加哈希后缀，实现局部作用域。
+ **Scoped CSS**（Vue `scoped`）：通过给元素加唯一属性选择器隔离。
+ **CSS-in-JS**（styled-components）：样式随组件生成唯一类名。
+ **BEM 命名规范**：靠约定手动避免冲突。
+ **Shadow DOM**：Web Components 的天然样式隔离。

#### （12）CSS预处理器的使用

预处理器（Sass/Less）提供变量、嵌套、mixin、函数等能力，提升 CSS 可维护性。关于 **`@include` 与 `@extend` 的区别**：

+ **`@include`（mixin）**：把 mixin 的样式**复制**到每个调用处，可传参，灵活；缺点是有重复代码。
+ **`@extend`**：让选择器**共享**同一组样式，Sass 会把它们合并到一个选择器组，输出体积小；但会改变选择器结构，跨模块使用时可能产生意外的选择器耦合。

需传参或逻辑用 `@include`，纯静态共享用 `@extend`。

#### （13）Monorepo和changesets发包流程

**Monorepo**：一个仓库管理多个包，共享依赖和工具链（常用 pnpm workspace / Turborepo）。

**changesets 发包流程**：

**（1）** 开发者改动后执行 `changeset add`，选择受影响的包和版本类型（major/minor/patch）并写变更说明，生成一个 changeset 文件。

**（2）** `changeset version` 消费这些文件，自动升级各包版本号、更新 CHANGELOG。

**（3）** `changeset publish` 发布到 npm。

CI/CD 中常见**安全问题**：npm token / GitHub token 泄露风险，应存 Secrets、用最小权限，并注意 CI 环境防止拉取到恶意依赖。

#### （14）有意思的项目经历

**答题思路**：挑一个**有技术挑战或独特性**的项目，讲清「为什么有意思」——是攻克了某个难题、用了新技术、还是解决了实际痛点。按「背景 → 挑战 → 你的方案与思考 → 结果与收获」组织，重点展现你的**技术热情和钻研能力**。

#### （15）两栏布局的实现方法

以「左侧固定、右侧自适应」为例：

+ **Flex**：父级 `display: flex`，左侧固定宽，右侧 `flex: 1`。
+ **Grid**：`grid-template-columns: 200px 1fr`。
+ **float + margin**：左侧 `float: left` 定宽，右侧 `margin-left` 留出宽度。
+ **absolute + margin**：左侧绝对定位，右侧 `margin-left`。

现代开发首选 **Flex / Grid**，简洁且易维护。

```css
.container { display: flex; }
.left { width: 200px; }
.right { flex: 1; }
```

#### （16）数组去重的实现方式

+ **Set（最常用）**：`[...new Set(arr)]`，简洁高效，但无法去重引用不同的对象。
+ **filter + indexOf**：`arr.filter((v, i) => arr.indexOf(v) === i)`，`O(n²)`。
+ **reduce + includes**：遍历累加不重复项。
+ **Map / 对象记录**：用哈希表记录已出现值，`O(n)`，适合大数据量。

```javascript
const unique = arr => [...new Set(arr)];
// 需按对象某字段去重时
const uniqueBy = (arr, key) => {
  const map = new Map();
  return arr.filter(item => !map.has(item[key]) && map.set(item[key], 1));
};
```
