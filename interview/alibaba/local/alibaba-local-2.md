### 面试中的问题：

#### （1）自我介绍

**答题思路**：用 1 分钟讲清「你是谁 + 技术亮点 + 为什么匹配」。建议结构：教育背景 → 最拿得出手的 1~2 个项目及你的角色和量化产出 → 熟悉的技术栈 → 想来该岗位的原因。多用与 JD 相关的关键词，避免流水账。

#### （2）vite的打包原理

Vite 分**开发**和**生产**两套机制：

+ **开发环境**：不打包，利用浏览器原生 **ESM**，按需编译请求到的模块，冷启动极快；用 **esbuild**（Go 编写）做依赖预构建，把 CommonJS 转成 ESM 并合并零散模块。
+ **生产环境**：为兼容性和性能，用 **Rollup** 做打包、Tree-shaking、代码分割和压缩。

#### （3）和webpack区别

+ **开发启动**：Webpack 先打包整个 bundle 再启动，项目越大越慢；Vite 基于原生 ESM 按需编译，秒级启动。
+ **热更新**：Webpack 需重新构建相关 chunk；Vite 只让失效模块重新请求，HMR 与项目体积无关。
+ **底层**：Vite 依赖预构建用 esbuild、生产用 Rollup；Webpack 全流程用自身的 JS 编译。Webpack 生态更成熟、配置更灵活。

#### （4）vite热更新原理

Vite 基于 ESM 实现 HMR：启动时建立 **WebSocket** 连接，并用文件系统监听（chokidar）监视文件变化。文件改动后，服务端计算受影响的模块边界，通过 WebSocket 推送更新消息，客户端按需重新请求**该模块**并替换，而不刷新整个页面、不重建 bundle，因此速度与项目大小无关。

#### （5）是用的http协议还是websocket协议

热更新**消息通知**用 **WebSocket**（服务端需主动推送变更，HTTP 无法服务端主动发起）。而具体**更新后的模块代码**仍是客户端通过 **HTTP** 请求拉取的。二者配合：WebSocket 负责「告诉你哪变了」，HTTP 负责「把新代码取回来」。

#### （6）http和websocket区别

+ **通信方向**：HTTP 是请求-响应、单向、客户端发起；WebSocket 建立后**全双工**，双方都能主动发消息。
+ **连接**：HTTP 短连接（每次请求重新建立）；WebSocket 长连接，一次握手后持续。
+ **握手**：WebSocket 借用 HTTP 发起 `Upgrade` 请求升级协议，之后走 `ws://`/`wss://`。
+ **场景**：实时聊天、推送、协同编辑等用 WebSocket。

#### （7）除了应用层还有哪些层

以 **TCP/IP 四层模型**为例，自顶向下：

+ **应用层**：HTTP、DNS、FTP 等（对应 OSI 应用/表示/会话层）。
+ **传输层**：TCP、UDP，端到端进程通信。
+ **网络层**：IP，负责寻址和路由。
+ **网络接口层（数据链路 + 物理）**：MAC 寻址、比特流传输。

OSI 则细分为七层：物理、数据链路、网络、传输、会话、表示、应用。

#### （8）分别都有什么协议

+ **应用层**：HTTP/HTTPS、DNS、FTP、SMTP、WebSocket。
+ **传输层**：TCP（可靠）、UDP（高效）。
+ **网络层**：IP、ICMP（ping）、ARP（IP 转 MAC）。
+ **数据链路/物理层**：以太网协议、PPP 等。

#### （9）SSL/TLS协议是什么

SSL/TLS 是位于**传输层之上、应用层之下**的安全协议，为通信提供**加密、完整性校验和身份认证**。TLS 是 SSL 的继任标准（现主流为 TLS 1.2/1.3）。

核心思路：用**非对称加密**（RSA/ECDHE）在握手阶段安全协商出一个**对称密钥**，之后数据用对称加密传输（兼顾安全和性能），并用证书验证服务端身份、用消息认证码保证数据未被篡改。

#### （10）https是怎么工作的

**（1）TCP 三次握手**建立连接。

**（2）TLS 握手**：客户端发送支持的加密套件和随机数；服务端返回**证书**（含公钥）和随机数；客户端验证证书合法性，生成 `pre-master`，用服务端公钥加密发回；双方基于三个随机数生成相同的**会话对称密钥**。

**（3）加密通信**：之后所有 HTTP 数据用对称密钥加密传输。

这样兼顾了非对称加密的安全握手和对称加密的高效传输，防窃听、防篡改、防伪造。

#### （11）vue的底层原理

Vue 把 `template` 通过**编译器**编译成 **render 函数**，render 执行生成 **虚拟 DOM（VNode）**。数据被**响应式**处理（Vue2 用 `Object.defineProperty`，Vue3 用 `Proxy`），组件渲染时会作为依赖被收集；数据变化时通知对应的渲染 watcher 重新执行 render，生成新 VNode，再通过 **diff** 对比新旧 VNode，最小化地更新真实 DOM。

#### （12）vue2和vue3响应式区别

+ **Vue2**：用 `Object.defineProperty` 递归劫持每个属性的 getter/setter。缺陷是**无法检测新增/删除属性**（需 `Vue.set`）、无法直接监听数组索引和 `length`，且初始化时要深度遍历。
+ **Vue3**：用 **`Proxy`** 代理整个对象，能拦截属性新增删除、数组索引变化，**惰性**代理嵌套对象（用到才递归），性能和能力都更好。

#### （13）vue2defineProperty怎么对数组进行处理的

`Object.defineProperty` 无法监听数组下标赋值和 `length` 变化，所以 Vue2 **重写了数组的 7 个变异方法**（`push`、`pop`、`shift`、`unshift`、`splice`、`sort`、`reverse`）：创建一个继承自 `Array.prototype` 的拦截原型，覆盖这些方法，调用时先执行原方法，再手动做依赖派发（`ob.dep.notify()`）触发更新。因此直接 `arr[0] = x` 或改 `length` 不会触发响应式。

#### （14）data()为什么是函数而不是对象

因为组件会被**复用**（多次实例化）。如果 `data` 是对象，所有实例会**共享同一个引用**，改一个会影响全部。写成函数后，每次创建实例都调用它返回一个**全新的对象**，保证各组件实例的数据相互独立。根实例因为只有一个，用对象也可以。

#### （15）vue的路由模式

+ **hash 模式**：URL 带 `#`，靠 `hashchange` 事件监听变化。改变 hash 不触发页面刷新、不向服务器发请求，兼容性好，无需服务端配置。
+ **history 模式**：URL 干净，用 `history.pushState/replaceState` 修改历史记录，靠 `popstate` 监听。需**服务端配置**将所有路由回退到 `index.html`，否则刷新会 404。

#### （16）diff算法和时间复杂度

Vue 的 diff 是**同层比较**：只对比同一层级的新旧节点，不跨层，把标准树 diff 的 `O(n³)` 优化到 **`O(n)`**。

对同层子节点，Vue2 采用**双端比较**：新旧列表各设头尾指针，按「旧头-新头、旧尾-新尾、旧头-新尾、旧尾-新头」四种方式两两比对并移动指针，命中则复用，配合 `key` 减少 DOM 移动。

#### （17）vue3diff算法的改进

+ **静态提升**：静态节点只创建一次，diff 时跳过。
+ **PatchFlag（靶向更新）**：编译时标记动态节点及其动态类型（class/style/text），运行时只对比被标记的部分，跳过静态内容。
+ **最长递增子序列**：处理带 key 的乱序列表时，用最长递增子序列算法求出**不需移动**的节点，最小化 DOM 移动次数。

#### （19）promise和async/await的区别

二者都用于处理异步，`async/await` 是 Promise 的**语法糖**：

+ **写法**：Promise 用 `.then/.catch` 链式，回调多了易嵌套；`async/await` 用同步写法写异步，更易读。
+ **错误处理**：Promise 用 `.catch`；`async/await` 用 `try/catch`。
+ **本质**：`await` 会等待 Promise resolve，`async` 函数返回的仍是 Promise。底层是基于 Generator + 自动执行器实现的。

#### （20）async和await的原理

`async/await` 本质是 **Generator + 自动执行器**的语法糖。`async` 函数被编译为一个生成器函数，每个 `await` 对应一次 `yield`（暂停），等待其后的 Promise 完成；有一个自动执行器（类似 co 库）不断调用 `.next()`，把 Promise 的结果回填、推动函数继续执行，直到全部完成后 resolve `async` 返回的 Promise。所以 `await` 后面的代码相当于放进了 `.then` 回调，属于微任务。

#### （21）promise的链式调用怎么实现的

关键是每次 `.then()` **返回一个新的 Promise**，而不是原来的实例，这样才能继续 `.then`。

+ 若回调返回**普通值**，新 Promise 直接以该值 resolve。
+ 若回调返回一个 **Promise**，则新 Promise 会等待它的状态并「展开」其结果，实现串行。

这样每个 `.then` 的返回值就成了下一个 `.then` 的输入，形成链条。

#### （22）promise.all之类是在类上面的吗

是的。`Promise.all`、`Promise.race`、`Promise.allSettled`、`Promise.any`、`Promise.resolve`、`Promise.reject` 都是 **`Promise` 构造函数的静态方法**（挂在类本身上），通过 `Promise.all(...)` 调用，用于处理**一组** Promise。

+ `all`：全部成功才成功，一个失败即失败。
+ `race`：以最先敲定（成功或失败）的为准。
+ `allSettled`：等所有都敲定，返回每个的状态。
+ `any`：任一成功即成功，全失败才失败。

#### （23）有没有什么在promise实例上的方法

有，实例方法定义在 `Promise.prototype` 上，通过实例调用：

+ **`.then(onFulfilled, onRejected)`**：注册成功/失败回调。
+ **`.catch(onRejected)`**：捕获错误，等价于 `.then(null, fn)`。
+ **`.finally(fn)`**：无论成功失败都执行，常用于收尾（关 loading）。

#### （24）说说浏览器缓存

分为**强缓存**和**协商缓存**：

+ **强缓存**：不发请求直接用本地缓存。由响应头 `Cache-Control`（如 `max-age`，优先级高）和 `Expires`（绝对时间，已过时）控制。
+ **协商缓存**：强缓存失效后，带标识向服务器询问是否可用。用 `Last-Modified/If-Modified-Since` 或 `ETag/If-None-Match`，未变则返回 304 用本地缓存。

命中顺序：先强缓存，未命中再走协商缓存。

#### （25）缓存的一些状态码

+ **200（from memory/disk cache）**：命中**强缓存**，直接读内存或磁盘缓存，不发请求。
+ **304 Not Modified**：命中**协商缓存**，服务器告知资源未变，用本地缓存，响应体为空，节省带宽。
+ **200（正常）**：缓存未命中或首次请求，服务器返回完整资源。

#### （26）etag和last-modify优先级 为什么

**ETag 优先级更高**。当两者同时存在时，服务器优先校验 ETag。原因是 `Last-Modified` 有局限：

+ 只能精确到**秒**，一秒内的多次修改无法识别。
+ 文件内容没变、只是重新保存或时间戳变了，也会被判定为「已修改」。

ETag 是根据文件**内容**生成的唯一标识，能精确反映内容是否真的变化，更可靠。

#### （27）last-modify缺点

+ **精度只到秒**：一秒内的多次改动无法被感知。
+ **误判**：文件被编辑后又还原、或只是修改时间变了而内容没变，也会被认为已修改，导致缓存失效、重新传输。
+ **分布式不一致**：多台服务器上同一文件的修改时间可能不同，造成缓存判断混乱。

这些正是引入 ETag（基于内容哈希）的原因。

#### （28）数组和链表区别 优缺点

+ **存储**：数组连续内存，链表靠指针链接、内存分散。
+ **访问**：数组支持随机访问 `O(1)`；链表只能顺序遍历 `O(n)`。
+ **增删**：数组中间增删需移动元素 `O(n)`；链表已知位置时改指针即可 `O(1)`。
+ **空间**：数组可能预留浪费或需扩容搬移；链表每个节点多存指针，有额外开销。

随机读多用数组，频繁增删用链表。

#### （29）知道哪些排序算法

+ **`O(n²)`**：冒泡、选择、插入（简单、数据量小时可用）。
+ **`O(n log n)`**：快速排序（原地、平均最快）、归并排序（稳定、需额外空间）、堆排序（原地）。
+ **非比较排序**：计数排序、桶排序、基数排序，特定条件下可达 `O(n)`。

其中**稳定**的有：冒泡、插入、归并；**不稳定**的有：选择、快排、堆排序。

#### （30）说说归并排序 稳定性和时间复杂度

**思路**：分治。把数组不断二分到单个元素，再两两**有序合并**。

+ **时间复杂度**：无论好坏均为 **`O(n log n)`**（`log n` 层，每层合并 `O(n)`）。
+ **空间复杂度**：`O(n)`，需要额外数组存合并结果。
+ **稳定性**：**稳定**，合并时相等元素让左半部分先入即可保持相对顺序。

```javascript
function mergeSort(arr) {
  if (arr.length <= 1) return arr;
  const mid = arr.length >> 1;
  const left = mergeSort(arr.slice(0, mid));
  const right = mergeSort(arr.slice(mid));
  const res = [];
  let i = 0, j = 0;
  while (i < left.length && j < right.length) {
    // <= 保证稳定性
    res.push(left[i] <= right[j] ? left[i++] : right[j++]);
  }
  return res.concat(left.slice(i)).concat(right.slice(j));
}
```
