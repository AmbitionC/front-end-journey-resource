### 面试中的问题：

#### （1）手写快速排序算法

**答题思路**：选一个基准值 pivot，把数组分成「比它小」和「比它大」两部分，递归排序两部分再拼接。平均时间复杂度 **O(n log n)**，最坏（近乎有序且固定取端点）退化为 **O(n²)**，可用随机基准或三数取中缓解。

```javascript
function quickSort(arr) {
  if (arr.length <= 1) return arr;
  const pivot = arr[arr.length - 1];
  const left = [], right = [];
  for (let i = 0; i < arr.length - 1; i++) {
    (arr[i] < pivot ? left : right).push(arr[i]);
  }
  return [...quickSort(left), pivot, ...quickSort(right)];
}
```

（原地分区版可用 Hoare/Lomuto 双指针，空间从 O(n) 降到 O(log n)。）

#### （2）敏感词过滤的实现与性能优化

敏感词过滤是**多模式串匹配**问题，KMP 只适合单模式串，工程上更常用 **Trie 树（前缀树）** 或 **DFA/AC 自动机**：把所有敏感词构建成一棵树，扫描文本时逐字符沿树走，一次遍历即可命中所有词。

+ **AC 自动机**：Trie + fail 指针，失配时跳到最长后缀节点，文本只需遍历一遍，复杂度 O(n)。
+ **优化**：预编译成 DFA、跳过无关字符、处理变形绕过（大小写、全半角、插入干扰符、拼音谐音）。

#### （3）KMP算法的原理

KMP 用于在主串中查找模式串，核心是**失配时不回退主串指针**，利用已匹配信息让模式串指针跳到合适位置。

+ **next 数组（部分匹配表）**：记录模式串每个位置的**最长公共前后缀**长度，失配时模式串指针跳到 `next[j]` 继续匹配。
+ **复杂度**：预处理 O(m) + 匹配 O(n)，整体 **O(n+m)**，优于暴力的 O(n·m)。

#### （5）Node.js的应用与理解

Node.js 是基于 V8 引擎的 JavaScript 运行时，采用**单线程 + 事件循环 + 非阻塞 I/O**（底层 libuv 用线程池处理文件/DNS 等阻塞操作）。

+ **擅长**：I/O 密集、高并发场景，如 API 网关、BFF 中间层、实时通信（WebSocket）。
+ **不擅长**：CPU 密集计算会阻塞事件循环，可用 `worker_threads` 或 `cluster` 多进程分担。

#### （6）Vue框架的工作原理

+ **响应式**：Vue2 用 `Object.defineProperty`、Vue3 用 `Proxy` 劫持数据，getter 时收集依赖、setter 时派发更新。
+ **渲染**：模板编译成 render 函数生成虚拟 DOM（VNode），数据变化后 diff 新旧 VNode，最小化 patch 真实 DOM。
+ **组件化**：单文件组件封装模板/逻辑/样式，通过 props/events 通信，实现复用与解耦。

#### （7）虚拟DOM的diff算法实现

虚拟 DOM 是用 JS 对象描述真实 DOM，diff 目的是找出最小变更，避免频繁直接操作 DOM。

+ **同层比较**：只对比同一层级节点，不跨层，把复杂度从 O(n³) 降到 **O(n)**。
+ **key 复用**：通过 key 标识节点身份，快速判断可复用；Vue2 用**双端比较**（新旧头尾四指针），Vue3 引入**最长递增子序列**优化节点移动。

#### （8）Vuex的作用与优势

Vuex 是 Vue 的集中式状态管理，把多组件共享的状态抽到统一 store，解决跨层级、兄弟组件通信难的问题。

+ **核心概念**：state（状态）、getters（派生）、mutations（同步修改，唯一改 state 的途径）、actions（异步）、modules（模块化）。
+ **优势**：状态可追踪、单向数据流清晰，配合 devtools 可做时间旅行调试。

#### （9）缺乏状态管理可能引发的问题

+ **逐层透传**：父子多层传递 props（prop drilling），中间组件被迫承接无关数据。
+ **通信困难**：兄弟/远亲组件只能靠事件总线或全局变量，耦合高、难维护。
+ **状态分散**：同一份数据多处副本易不一致，出 bug 难定位、难调试。

#### （10）前端学习之路的起点

**答题思路**：讲清入门契机 + 学习路径 + 体系化过程（HTML/CSS/JS 基础 → 框架 → 工程化 → 深入原理），体现主动性和持续学习能力，可用一两个自驱项目佐证成长。

#### （11）前端性能优化的策略

+ **网络**：HTTP 缓存、CDN、Gzip/Brotli 压缩、HTTP2 多路复用、资源预加载/预连接。
+ **加载**：代码分割、路由/组件懒加载、Tree Shaking、图片懒加载与格式优化。
+ **渲染**：减少重排重绘、虚拟列表、防抖节流、骨架屏；首屏可用 SSR/预渲染。
+ **量化**：以 FCP、LCP、TTI、CLS 为指标，用 Lighthouse / Performance API 度量。

#### （12）PWA技术的了解与应用

PWA（Progressive Web App）让 Web 应用具备接近原生的体验，核心是三件套：

+ **Service Worker**：拦截网络请求、离线缓存资源，实现离线可用与秒开。
+ **Manifest**：配置图标/名称，支持「添加到桌面」安装。
+ **Push API**：消息推送。优势是免安装、可离线、渐进增强，不支持时降级为普通网页。

#### （13）微前端的概念与实践

微前端把大型应用拆成**可独立开发、部署**的子应用，由主应用负责路由分发和加载，解决巨石应用维护难、技术栈绑定的问题。

+ **方案**：qiankun（基于 single-spa，提供 JS 沙箱和样式隔离）、Module Federation（webpack5 模块共享）、iframe（隔离最彻底但体验受限）。
+ **价值**：技术栈无关、独立发布、增量升级老系统。

#### （14）WebAssembly的技术特点

WebAssembly（Wasm）是一种可移植的**二进制指令格式**，运行在浏览器沙箱中，执行速度接近原生。

+ **来源**：C/C++/Rust 等语言编译产物，可与 JS 互相调用。
+ **场景**：音视频编解码、图像处理、游戏、加解密等 CPU 密集任务，弥补 JS 在计算性能上的短板。

#### （15）数据库连接操作的区别

+ **LEFT JOIN**：保留左表全部行，右表无匹配的字段补 NULL。
+ **RIGHT JOIN**：保留右表全部行，左表无匹配补 NULL。
+ **INNER JOIN**：只保留两表都匹配的行；FULL JOIN 保留两边全部。

#### （16）数据库索引的作用与实现

索引用于加速查询，避免全表扫描，本质是为列建立**有序的数据结构**。

+ **B+树索引**（最常用）：叶子节点有序且链表相连，支持等值和**范围查询**，MySQL InnoDB 默认。
+ **哈希索引**：等值查询 O(1)，但不支持范围和排序。

#### （17）索引的副作用及其影响

+ **写性能下降**：增删改时需同步维护索引结构，写入变慢。
+ **占用存储**：索引本身需要额外磁盘空间。
+ **过多反效**：索引过多会增加优化器选择成本；低区分度列（如性别）建索引几乎无效。

#### （18）非关系型数据库的种类与特点

+ **键值型**：Redis，内存读写极快，用于缓存、会话、排行榜。
+ **文档型**：MongoDB，存 JSON 文档，schema 灵活。
+ **列族型**：HBase/Cassandra，适合海量宽表。
+ **图数据库**：Neo4j，适合社交关系等图结构。共同特点：灵活 schema、易水平扩展、高并发。

#### （19）索引类型的区别

+ **单列索引**：仅覆盖单个字段。
+ **联合索引**：`(a,b,c)` 遵循**最左前缀原则**，能命中 `a`、`a,b`、`a,b,c` 的查询，且可做**覆盖索引**（查询列都在索引里，免回表）。
+ 联合索引通常比多个单列索引更高效，能减少回表并覆盖更多组合查询。

#### （20）Node.js的了解与应用

除运行时特性外（单线程事件循环、非阻塞 I/O），在前端工程中 Node 的典型应用：

+ **中间层/BFF**：聚合裁剪后端接口、做 SSR 服务端渲染。
+ **工程化**：webpack/vite 等构建工具、脚手架 CLI、自动化脚本均跑在 Node 上。
+ **服务端缓存/代理**：作为轻量网关或接口缓存层。
