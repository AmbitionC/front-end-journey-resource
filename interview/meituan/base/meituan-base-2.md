### 面试中遇到的技术问题：

#### （1）实习经历分享

**答题思路**：用 STAR 结构讲一段实习，重点不在「做了什么功能」，而在**你的角色、独立负责的模块、遇到的难点和量化产出**。突出与应聘岗位相关的技术栈和协作经历，结尾可提炼一句成长收获。

#### （2）算法题的准备

**答题思路**：说明你的准备方式即可——按专题（数组/字符串、链表、二叉树、DP、双指针等）刷题，注重**掌握模板和边界处理**而非题量。可提一句常用平台（LeetCode/牛客）和刷题量级，体现持续投入。

#### （3）实验室项目经验

**答题思路**：介绍项目背景与你承担的职责，突出**独立解决问题的能力、团队协作和项目管理**中的具体贡献。避免泛泛而谈，给出一两个能体现深度的技术点或结果指标。

#### （4）技术栈选择的理由

选择 **Vue** 的常见理由：

+ **上手成本低**：模板语法贴近 HTML，单文件组件结构清晰，学习曲线平缓。
+ **渐进式**：可从简单页面增强逐步扩展到全家桶（Vue Router、Pinia/Vuex）。
+ **生态与社区**：中文文档完善、组件库丰富（Element、Ant Design Vue 等），团队协作和招聘成本低。

答题时应结合**项目实际约束**（团队熟悉度、迭代速度、生态需求）来说明，而非只夸框架。

#### （5）组件库

除 Element UI 外常见的还有：**Ant Design Vue**（企业中后台）、**Vant**（移动端）、**Naive UI / Arco Design**（较新、TS 友好）。选型考量：

+ **场景匹配**：PC 中后台 vs 移动端。
+ **定制能力**：主题、按需引入、样式覆盖是否方便。
+ **维护与生态**：更新活跃度、文档、TypeScript 支持。

#### （6）项目合作经验

**答题思路**：讲清团队规模、你的分工与协作方式（Git 分支管理、Code Review、接口联调）。突出**沟通如何减少返工**、如何对齐接口约定（如与后端约定字段、用 mock 并行开发）等具体做法。

#### （7）项目规模和页面数量

**答题思路**：客观说明项目体量（页面数、模块数、团队人数、上线周期）。可延伸讲随规模增长带来的**工程化诉求**（组件复用、路由拆分、状态管理、构建优化），体现你对项目全局的理解。

#### （8）状态管理的实践

Vue 中常用 **Vuex / Pinia** 做跨组件状态共享：

+ **适用场景**：多个非父子组件依赖同一份数据（用户信息、购物车、权限）。
+ **Pinia 优势**：API 更简洁、原生 TypeScript 支持、去掉了 mutation、天然支持模块化，是 Vue3 推荐方案。
+ **注意**：并非所有状态都放 store，局部状态仍应留在组件内，避免全局状态膨胀。

#### （9）mixin的使用和作用

**mixin** 用于把**多个组件复用的逻辑**（data、methods、生命周期钩子）抽离共享。

+ **合并规则**：数据对象递归合并，同名冲突以组件自身优先；同名钩子会合并成数组依次调用。
+ **缺点**：来源不清晰（属性从哪来难追踪）、命名易冲突。Vue3 更推荐用 **组合式 API（Composition API / hooks）** 替代 mixin 复用逻辑。

#### （10）axios的二次封装

对 axios 二次封装是为了**统一管理请求**：

+ **实例化**：设置 `baseURL`、`timeout`、公共请求头。
+ **请求拦截器**：统一注入 token、开启 loading。
+ **响应拦截器**：统一处理业务状态码、错误提示、401 跳登录、剥离外层数据结构。
+ **收益**：调用方只关注业务参数，错误处理和鉴权逻辑集中维护，避免重复代码。

#### （11）Nginx配置要点

前端常用的 Nginx 配置：

+ **静态资源托管**：`root`/`index` 指向打包产物，配合 `gzip` 压缩。
+ **history 路由**：`try_files $uri $uri/ /index.html` 解决单页应用刷新 404。
+ **反向代理**：`location /api { proxy_pass ... }` 解决跨域、转发后端。
+ **缓存策略**：对带 hash 的静态资源设置长期 `expires`，对 `index.html` 不缓存。

#### （12）页面路由的实现

**答题思路**：讲清单页应用（SPA）前端路由原理——**hash 模式**监听 `hashchange`、**history 模式**用 `pushState`/`popstate`，二者都在不刷新页面的前提下切换视图。再结合项目讲路由懒加载（`() => import()`）和路由守卫做鉴权。

#### （13）小程序开发技巧

**答题思路**：结合具体项目讲小程序开发经验，可覆盖：**双线程架构**（逻辑层与渲染层分离、`setData` 通信有性能成本）、分包加载、原生组件层级问题、以及跨端框架（Taro/uni-app）的取舍。突出你踩过的坑和优化手段。

#### （14）DNS解析原理

**DNS** 把域名解析为 IP，采用**分级、迭代/递归查询**：

浏览器缓存 → 系统 hosts/缓存 → 本地 DNS 服务器 → 根域名服务器 → 顶级域（.com）服务器 → 权威域名服务器，逐级查到最终 IP 并逐级缓存（受 TTL 控制）。这样既保证可读性，又通过分布式与缓存降低单点压力和查询延迟。

#### （15）HTTP与HTTPS的区别

+ **安全性**：HTTP 明文传输，易被窃听/篡改；HTTPS 在 TCP 与 HTTP 间加 **TLS/SSL** 层，提供加密、完整性校验和身份认证。
+ **端口**：HTTP 默认 80，HTTPS 默认 443。
+ **成本**：HTTPS 需要 CA 证书，握手多一次 TLS 协商，但现代性能开销已很小，是当前标配。

#### （16）HTTPS的加/解密过程

HTTPS 采用**非对称加密协商 + 对称加密传输**的混合方案：

**（1）** 客户端发起握手，服务端返回**证书（含公钥）**。
**（2）** 客户端校验证书合法性，生成随机的**对称密钥**，用服务端公钥加密后发送。
**（3）** 服务端用私钥解密得到对称密钥。
**（4）** 之后双方用该对称密钥加解密业务数据——既保证了密钥交换安全，又兼顾了对称加密的高效率。

#### （17）HTTPS可能遭受的攻击方式

+ **中间人攻击（MITM）**：伪造证书截获通信；防范靠**证书链校验 + HSTS + 证书钉扎**。
+ **SSL 剥离**：把 HTTPS 降级为 HTTP；用 HSTS 强制 HTTPS 防范。
+ **证书伪造 / CA 被攻破**：依赖可信 CA 体系和 CT（证书透明度）监控。
+ **降级攻击 / 弱加密套件**：禁用过时协议（SSLv3、TLS1.0）和弱套件。

#### （18）HTTP缓存机制

**答题思路**：分**强缓存**与**协商缓存**两层作答。强缓存由 `Cache-Control`（`max-age`）、`Expires` 控制，命中不发请求；强缓存失效后走协商缓存，用 `ETag`/`If-None-Match`、`Last-Modified`/`If-Modified-Since` 询问服务端，未变更返回 304 复用本地。静态资源常用「hash 文件名 + 长 max-age」策略。

#### （19）HTTP/2.0的新特性

相比 HTTP/1.x：

+ **二进制分帧**：把报文拆成帧传输，取代文本解析。
+ **多路复用**：一个 TCP 连接上并行收发多个请求，解决 HTTP/1.x 的队头阻塞和连接数限制。
+ **头部压缩（HPACK）**：压缩冗余的请求头。
+ **服务端推送（Server Push）**：主动推送客户端将需要的资源。

对前端的影响：过去的雪碧图、域名分片等优化手段在 HTTP/2 下收益变小甚至反效果。

#### （20）进程间通信方式

常见 **IPC** 方式：**管道（匿名/命名）、消息队列、共享内存、信号量、信号（Signal）、套接字（Socket）**。其中**共享内存**速度最快但需配合信号量做同步；**Socket** 可跨主机通信。

#### （21）进程与线程的区别

+ **资源**：进程是资源分配的基本单位，拥有独立地址空间；线程是 CPU 调度的基本单位，同一进程内的线程共享地址空间和资源。
+ **开销**：线程创建/切换开销远小于进程。
+ **隔离性**：进程间相互隔离更稳定（一个崩溃不影响其他）；线程共享内存，通信方便但需处理同步与竞态。

#### （22）死锁的概念和产生条件

**死锁**指多个进程互相持有对方所需资源并等待，导致都无法推进。产生的**四个必要条件**：

+ **互斥**：资源同一时刻只能被一个进程占用。
+ **占有并等待**：持有资源的同时等待其他资源。
+ **不可剥夺**：资源不能被强行抢占。
+ **循环等待**：存在进程互相等待的环路。

破坏任一条件即可预防（如按序申请资源破坏循环等待）。

#### （23）前端框架的学习经历

**答题思路**：讲清学习路径与方法——从原生 JS/DOM 基础入手，到掌握某个框架（Vue/React）的核心机制（响应式、组件化、虚拟 DOM），再到读源码或做项目加深理解。突出**主动学习和知其所以然**的态度，避免只列框架名。

#### （24）Vue计算属性与watch的区别

+ **computed（计算属性）**：**基于依赖缓存**，依赖不变则不重新计算；适合由已有数据派生出新值（如全选状态、格式化文本）。
+ **watch（侦听器）**：**监听某数据变化后执行副作用**，无缓存；适合异步操作、开销大的逻辑（如输入防抖请求、路由变化拉数据）。
+ **一句话**：能用 computed 就用 computed，需要「变化时做事」才用 watch。

#### （25）Vue3与Vue2的改进点

+ **响应式**：由 `Object.defineProperty` 改为 **Proxy**，可监听属性新增/删除和数组索引变化。
+ **组合式 API**：`setup`/`ref`/`reactive` 更利于逻辑复用和 TypeScript 支持。
+ **性能**：编译期优化（静态提升、Patch Flag）、更小的包体、Tree-shaking 友好。
+ **其他**：支持多根节点（Fragment）、Teleport、Suspense。

#### （26）Proxy的优势和应用

需先澄清：**Proxy 是 Vue3 响应式的核心，Vue2 用的是 `Object.defineProperty`**。

Proxy 相对 `defineProperty` 的优势：

+ **拦截更全面**：可拦截 13 种操作，能监听**属性新增、删除**及**数组索引/长度**变化，无需 Vue2 里 `Vue.set`/数组方法重写的 workaround。
+ **惰性代理**：对整个对象代理，访问到嵌套属性时再递归，无需初始化时深度遍历所有属性，性能更好。

#### （29）数据结构与算法

**答题思路**：二叉树**层序遍历（BFS）**用队列实现——根节点入队，每次出队一个节点并把其左右子节点入队，直到队列为空。

```javascript
function levelOrder(root) {
  const res = [], queue = root ? [root] : [];
  while (queue.length) {
    const len = queue.length, level = [];
    for (let i = 0; i < len; i++) {
      const node = queue.shift();
      level.push(node.val);
      if (node.left) queue.push(node.left);
      if (node.right) queue.push(node.right);
    }
    res.push(level);
  }
  return res;
}
```

#### （30）链表操作技巧

**答题思路**：删除**有序链表**中的重复元素，利用有序特性——只需比较当前节点与下一节点值，相等则跳过（`cur.next = cur.next.next`），否则后移。

```javascript
function deleteDuplicates(head) {
  let cur = head;
  while (cur && cur.next) {
    if (cur.val === cur.next.val) cur.next = cur.next.next;
    else cur = cur.next;
  }
  return head;
}
```
