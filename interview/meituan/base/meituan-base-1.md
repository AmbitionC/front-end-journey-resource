### 面试中遇到的技术问题：

#### （1）WebSocket与HTTP的区别及使用场景

**HTTP** 是无状态的请求-响应协议，每次通信都由客户端发起，服务端无法主动推送。**WebSocket** 在一次 HTTP 握手（`Upgrade: websocket`）后升级为**全双工、长连接**的 TCP 通道，双方可随时互发消息，帧头开销远小于 HTTP。

+ **使用场景**：即时通讯、在线协作文档、实时行情、弹幕、消息推送等需要服务端主动下发的场景。
+ **对比轮询**：相比短轮询/长轮询，WebSocket 避免了重复建连和冗余请求头，延迟更低、开销更小。

#### （2）let和const的区别

+ **可变性**：`let` 声明的变量可重新赋值；`const` 声明的是常量引用，不可重新赋值（但对象内部属性仍可改）。
+ **共同点**：二者都是**块级作用域**，都存在**暂时性死区（TDZ）**，且不允许在同一作用域内重复声明——这与 `var` 的函数作用域和变量提升不同。

#### （3）代码题：Promise相关代码输出

**答题思路**：考察事件循环中**同步代码 → 微任务（Promise.then/catch）→ 宏任务（setTimeout）**的执行顺序。答题时按顺序标注每一步进入哪个队列。

```javascript
console.log('start');
setTimeout(() => console.log('timeout'), 0);
Promise.resolve().then(() => console.log('promise'));
console.log('end');
// 输出：start → end → promise → timeout
```

同步代码先执行，一轮同步结束后清空所有微任务，再取一个宏任务，如此循环。

#### （4）缓存相关知识

浏览器缓存分为**强缓存**和**协商缓存**：

+ **强缓存**：命中则直接用本地副本，不发请求。由 `Cache-Control`（如 `max-age`、`no-cache`）和过期时间 `Expires` 控制，`Cache-Control` 优先级更高。
+ **协商缓存**：强缓存失效后带 `If-None-Match`/`If-Modified-Since` 询问服务端，未变更返回 **304** 复用缓存，变更则返回 200 和新资源。对应响应头是 `ETag` 和 `Last-Modified`。

合理设置缓存可减少请求、提升加载性能，静态资源常用「文件名 hash + 长 max-age」实现更新即失效。

#### （5）代码题：this指向问题

**答题思路**：`this` 由**调用方式**决定，而非定义位置。判断规则：

+ **普通函数调用**：`this` 指向 `window`（严格模式为 `undefined`）。
+ **对象方法调用**：`this` 指向调用它的对象。
+ **`call`/`apply`/`bind`**：`this` 为传入的第一个参数。
+ **`new`**：`this` 指向新创建的实例。
+ **箭头函数**：没有自己的 `this`，继承定义时外层作用域的 `this`。

#### （6）HTTP和HTTPS的三次握手过程

**HTTPS = HTTP + TLS/SSL**，在 TCP 之上增加加密层，解决明文传输被窃听、篡改的问题。

+ **TCP 三次握手**：客户端发 `SYN` → 服务端回 `SYN+ACK` → 客户端回 `ACK`，用于确认双方收发能力、同步初始序列号后建立连接。
+ **HTTPS 额外的 TLS 握手**：TCP 建连后再协商加密套件、通过证书验证服务端身份、用非对称加密交换对称密钥，之后用对称密钥加密业务数据。

### 开放性问题：

#### （1）开发过程中遇到的问题

**答题思路**：用 **STAR** 结构（背景 → 任务 → 行动 → 结果）讲一个真实技术难题。突出：问题定位过程（如何排查、用了什么工具）、你的解决方案与权衡、最终量化收益（性能提升、bug 收敛等）。选一个能体现深度的问题，避免罗列琐碎小事。

#### （2）为什么选择做前端

**答题思路**：从**兴趣 + 匹配 + 规划**三方面组织。可讲：前端「所见即所得」的正反馈让你有热情，结合一两次实际做出可交互产品的经历；再讲对工程化、性能、体验的持续深入意愿。避免空泛的「我觉得前端简单」这类回答。
