### 面试中的问题：

#### （1）http1.1, 2, 3 分别有什么问题，解决什么问题

+ **HTTP/1.1**：文本协议、每个 TCP 连接同一时刻只能处理一个请求，虽有 keep-alive 复用连接但存在**队头阻塞**；靠多条 TCP 连接和域名分片缓解。
+ **HTTP/2**：引入**二进制分帧**和**多路复用**，一个连接上并发多请求，解决了 HTTP 层队头阻塞；还有头部压缩（HPACK）、服务端推送。但仍跑在 TCP 上，存在 **TCP 层队头阻塞**。
+ **HTTP/3**：改用基于 **UDP 的 QUIC**，各流独立，一个流丢包不影响其他流，彻底解决 TCP 队头阻塞；内置 TLS 1.3（0-RTT/1-RTT 更快建连），且支持**连接迁移**（切换网络不断连）。

#### （2）详细说一下队头阻塞，tcp阻塞

+ **HTTP 层队头阻塞**：HTTP/1.1 一个连接内请求串行，前一个响应没回来后面的排队。HTTP/2 用多路复用解决了这一层。
+ **TCP 层队头阻塞**：TCP 保证**按序交付**，只要有一个数据包丢失，后面已到达的包也必须在缓冲区等待重传补齐才能上交应用层。HTTP/2 多路复用虽在 HTTP 层并发，但共用一条 TCP 连接，一旦丢包所有流一起卡住。
+ QUIC 把多路复用下沉到传输层，各 stream 独立确认与重传，丢包只影响单个流，从根本上消除 TCP 队头阻塞。

#### （3）tcp udp 传输

+ **TCP**：面向连接（三次握手）、可靠、有序、有流量控制与拥塞控制，开销大。适合文件传输、网页、邮件等要求可靠的场景。
+ **UDP**：无连接、不保证可靠和有序、无拥塞控制，头部小、延迟低。适合实时音视频、直播、游戏、DNS 等对时延敏感、可容忍少量丢包的场景。

#### （4）udp 怎么保证可靠传输

UDP 本身不可靠，需在**应用层自行实现类 TCP 机制**：

+ **序列号 + 确认应答（ACK）**：给每个包编号，接收方回 ACK，保证有序和不丢。
+ **超时重传**：未收到 ACK 则重发。
+ **滑动窗口 / 流量控制**、**拥塞控制**。
+ **前向纠错（FEC）**：冗余编码，少量丢包可直接恢复。

QUIC、KCP、RUDP 就是在 UDP 上实现可靠传输的典型方案。

#### （5）tls 加密

TLS 用**非对称加密协商 + 对称加密传输**结合：

+ **（1）握手**：客户端 Hello（携带支持的加密套件、随机数）→ 服务端返回证书和随机数 → 客户端**验证证书**（CA 链、有效期、域名）。
+ **（2）密钥协商**：通过 RSA 或 ECDHE 交换/协商出**会话密钥**（ECDHE 提供前向安全）。
+ **（3）通信**：之后用会话密钥做**对称加密**（性能高），并用 MAC/AEAD 保证完整性。

即：非对称解决密钥安全分发，对称解决数据高效加密。

#### （6）强缓存，协商缓存

+ **强缓存**：不发请求直接用本地缓存。由 `Cache-Control`（如 `max-age=3600`，优先级高）和 `Expires`（绝对时间，已过时）控制，命中返回 `200 (from disk/memory cache)`。
+ **协商缓存**：强缓存失效后带条件请求问服务器是否更新。用 `Last-Modified` / `If-Modified-Since`（秒级、按时间）或 `ETag` / `If-None-Match`（内容指纹、更精确，优先级高）。未变返回 **304**，复用本地缓存；变了返回 200 和新内容。

#### （7）cache-control 中 no-cache 和其他参数

+ **no-cache**：**可以缓存，但每次使用前必须向服务器发协商请求**验证是否过期（并非不缓存）。
+ **no-store**：完全不缓存，最严格。
+ **max-age=秒**：强缓存有效期。
+ **public / private**：是否允许中间代理（CDN）缓存 / 仅浏览器缓存。
+ **must-revalidate**：过期后必须重新验证，不能用陈旧副本。
+ **s-maxage**：针对共享缓存（CDN）的有效期。

#### （8）cdn 加速

CDN 是分布在各地的**边缘节点缓存网络**：

+ 用户请求经 **DNS 调度**解析到**地理/网络最近的边缘节点**，就近返回静态资源，降低时延。
+ 边缘节点缓存命中直接返回；未命中则回**源站**拉取并缓存（回源）。
+ 好处：减轻源站压力、加速静态资源（JS/CSS/图片/视频）、抗高并发与部分 DDoS。适合静态资源和大流量分发。

#### （9）option

`OPTIONS` 用于查询目标资源支持的通信选项，最常见于 **CORS 预检请求（preflight）**。当发起**复杂跨域请求**时，浏览器先自动发一个 OPTIONS 请求，携带 `Access-Control-Request-Method` / `Access-Control-Request-Headers`，服务端用 `Access-Control-Allow-*` 响应表明是否允许，通过后才发真正请求。

#### （10）复杂请求，简单请求

CORS 中：

+ **简单请求**：方法为 `GET/POST/HEAD`，且 `Content-Type` 仅限 `text/plain`、`application/x-www-form-urlencoded`、`multipart/form-data`，无自定义头。浏览器**直接发送**，只在响应校验 CORS 头。
+ **复杂请求**：使用 `PUT/DELETE/PATCH`、自定义头或 `application/json` 等。浏览器**先发 OPTIONS 预检**，服务端允许后才发真实请求。预检结果可用 `Access-Control-Max-Age` 缓存以减少开销。

#### （11）浏览器网络进程，怎么工作的

网络进程是浏览器多进程架构中**专门负责网络 I/O**的进程：处理 DNS 解析、建立 TCP/TLS 连接、发送 HTTP 请求与接收响应、维护连接池与缓存、执行安全策略（HSTS、证书校验）。渲染进程需要资源时通过 IPC 委托网络进程去拉取，拿到数据再交回渲染进程解析。独立成进程可提升安全性和稳定性。

#### （12）浏览器架构

现代浏览器采用**多进程架构**：

+ **浏览器主进程（Browser）**：管理界面、地址栏、进程协调、文件访问。
+ **渲染进程（Renderer）**：每个标签页/站点一个，负责 HTML/CSS 解析、布局、绘制、执行 JS（含 JS 引擎与合成线程），运行在**沙箱**中。
+ **网络进程**：处理网络请求。
+ **GPU 进程**：负责合成与图形加速。
+ **插件进程**：运行扩展/插件。

站点隔离（Site Isolation）让不同源在不同渲染进程，提升安全。

#### （13）进程

进程是操作系统**资源分配的基本单位**，拥有独立的内存地址空间，进程间相互隔离、通过 IPC 通信；线程是**CPU 调度的基本单位**，共享所在进程的内存。浏览器用多进程隔离各标签页，一个页面崩溃不影响其他页面，安全性也更高，代价是内存占用更大。

#### （14）浏览器进程之间怎么进行数据传输

浏览器进程/线程间通过 **IPC（进程间通信）** 交换数据，Chromium 中用 **Mojo** 框架实现（底层基于共享内存、管道等）。例如渲染进程通过 IPC 请求网络进程加载资源；大块数据（如图片位图）常用**共享内存**避免拷贝，控制消息走消息管道。渲染进程内部的多线程（主线程、合成线程、Worker）则通过消息队列/`postMessage` 通信。
