**答题思路**：这类问题考察项目深度与你的实际贡献。建议按 **STAR**（背景 → 任务 → 行动 → 结果）讲：项目背景与要解决的问题 → 你负责的模块 → 用到的关键技术和你做的技术决策 → 量化产出与遇到的难点如何攻克。重点讲「你做了什么、为什么这么做」，而非罗列功能。

#### （3）Buffer的概念和应用

**Buffer** 是 Node.js 用于处理**二进制数据**的类，本质是一段固定长度、分配在 V8 堆外的内存，用来存放字节序列（因为 JS 原生字符串不适合处理二进制流）。

+ **特点**：创建后大小固定，元素是 0~255 的字节；可与字符串按指定编码（utf-8、base64、hex）互转。
+ **应用**：文件读写、网络 TCP/HTTP 流、加密哈希、图片/音视频等二进制处理。流（Stream）传输时数据就是以 Buffer 分块流动的，配合背压机制高效处理大数据。

#### （4）分片API的使用

前端分片主要靠 **`Blob.prototype.slice`**（`File` 继承自 `Blob`），把大文件按字节区间切成多个小分片。

```javascript
const CHUNK = 5 * 1024 * 1024;
const chunks = [];
for (let cur = 0; cur < file.size; cur += CHUNK) {
  chunks.push(file.slice(cur, cur + CHUNK)); // slice(start, end) 返回 Blob
}
```

+ **配合场景**：大文件分片上传、断点续传，每片单独请求上传，失败只重传该片。
+ **读取内容**：需要读分片内容时用 `FileReader` 或 `blob.arrayBuffer()`，例如计算哈希做秒传校验。

#### （5）哈希的生成机制

**哈希函数** 把任意长度输入通过确定性算法映射为**固定长度**的输出（摘要）。

+ **核心特性**：确定性（同输入同输出）、雪崩效应（输入微小变化输出大不同）、单向不可逆、抗碰撞（不同输入难产生相同输出）。
+ **常见算法**：MD5、SHA-1、SHA-256 等。
+ **应用**：文件唯一标识（大文件上传秒传用 `spark-md5` 算 hash）、密码存储（加盐哈希）、数据完整性校验、哈希表寻址、一致性哈希。

#### （6）JavaScript基本数据类型

JS 有 **7 种原始类型** + 1 种引用类型：

+ **原始类型**：`Number`、`String`、`Boolean`、`Undefined`、`Null`、`Symbol`（ES6，唯一值）、`BigInt`（大整数）。原始值**按值存储在栈**、不可变。
+ **引用类型**：`Object`（含数组、函数等），存储在堆，变量保存的是引用地址。
+ **类型判断**：`typeof` 可辨别原始类型（但 `typeof null === 'object'` 是历史遗留 bug），引用类型用 `instanceof` 或 `Object.prototype.toString.call()` 精确判断。

#### （7）JavaScript继承的实现

+ **原型链继承**：子类原型指向父类实例，缺点是引用类型属性被所有实例共享。
+ **构造函数继承**：子类构造函数里 `Parent.call(this)`，能传参且属性独立，但无法继承父类原型方法。
+ **组合继承**：前两者结合（`call` 继承属性 + 原型继承方法），经典方案，但父构造函数被调用两次。
+ **寄生组合继承**：用 `Object.create(Parent.prototype)` 替代实例作为原型，避免二次调用，是 ES5 最优解。
+ **`class extends`**（ES6）：语法糖，`extends` + `super()` 实现，底层即寄生组合，是现在的推荐写法。

#### （8）TCP握手过程

TCP 建连需**三次握手**（SYN 标志位、seq 序列号）：

+ **第一次**：客户端发 `SYN`（seq=x），进入 SYN_SENT，表示请求建连。
+ **第二次**：服务端回 `SYN + ACK`（seq=y, ack=x+1），进入 SYN_RCVD，表示同意并确认。
+ **第三次**：客户端回 `ACK`（ack=y+1），双方进入 ESTABLISHED，连接建立。

目的是让双方都确认自己的**收发能力正常**，并同步初始序列号。

#### （9）三次握手的原因

核心是**确认双方收发能力都正常，防止历史失效连接建立造成资源浪费**。

+ **两次为什么不够**：只有两次，服务端无法确认客户端**能收到**自己的报文；且若一个**早已失效的旧 SYN** 延迟到达服务端，服务端建连并等待，客户端却不理会，会造成服务端资源空耗。第三次 ACK 让客户端确认这次连接有效。
+ **本质**：三次是「客户端能发能收、服务端能发能收」都得到确认的**最小次数**。

#### （10）UDP的作用和特点

**UDP** 是无连接的传输层协议，**不保证可靠**但高效。

+ **特点**：无需握手、无连接、不保证顺序和到达、无拥塞控制、头部小（仅 8 字节）、传输快、支持一对多（广播/组播）。
+ **对比 TCP**：TCP 面向连接、可靠、有序、有流量/拥塞控制，开销大；UDP 快但不可靠。
+ **适用场景**：实时性优先于可靠性的场合——音视频通话、直播、在线游戏、DNS 查询、DHCP。

#### （11）HTTP2.0多路复用的优势

**多路复用** 指在**一条 TCP 连接**上并行传输多个请求/响应，彻底解决 HTTP/1.1 的队头阻塞。

+ **原理**：HTTP/2 把消息拆成二进制**帧（frame）**，每个请求对应一个**流（stream）** 有唯一 ID，帧可交错发送、按 ID 在接收端重组，因此互不阻塞。
+ **优势**：无需像 HTTP/1.1 那样开多条连接或做域名分片，减少连接开销和延迟；配合头部压缩（HPACK）、服务端推送进一步提速。
+ **注意**：它解决的是 HTTP 层队头阻塞，TCP 层的队头阻塞仍存在（这正是 HTTP/3 改用 QUIC/UDP 的原因）。

#### （12）浏览器事件机制

包含两部分：**事件流** 与 **事件循环**。

+ **事件流**：DOM 事件分三阶段——捕获（从外向内）→ 目标 → 冒泡（从内向外）。`addEventListener` 第三参 `true` 在捕获阶段触发，默认冒泡阶段；可用 `stopPropagation` 阻止传播、`preventDefault` 阻止默认行为。
+ **事件委托**：利用冒泡把子元素事件绑定到父元素统一处理，减少监听器、支持动态元素。
+ **事件循环**：JS 单线程，通过事件循环处理异步——同步任务走主栈，异步回调进任务队列，主栈清空后先清微任务（Promise）再取一个宏任务（setTimeout、事件回调），如此循环。

#### （1）水平垂直居中的实现

给出几种常用方案：

```css
/* 1. Flex（最常用） */
.parent { display: flex; justify-content: center; align-items: center; }

/* 2. Grid */
.parent { display: grid; place-items: center; }

/* 3. 绝对定位 + transform（不需知道子元素宽高） */
.child {
  position: absolute; top: 50%; left: 50%;
  transform: translate(-50%, -50%);
}
```

+ **Flex/Grid**：现代布局首选，简洁且不依赖子元素尺寸。
+ **绝对定位 + transform**：兼容性好，`translate(-50%,-50%)` 按自身宽高回移实现精确居中。

#### （2）移动零的问题

**答题思路**：以经典「把所有 0 移到数组末尾、保持非零元素相对顺序」为例（若要求移到前面同理反向处理）。用**双指针**，`slow` 指向下一个非零元素应放的位置，`fast` 遍历数组，遇非零就交换到 `slow` 并前移，一趟 O(n)。

```javascript
function moveZeroes(nums) {
  let slow = 0;
  for (let fast = 0; fast < nums.length; fast++) {
    if (nums[fast] !== 0) {
      [nums[slow], nums[fast]] = [nums[fast], nums[slow]];
      slow++;
    }
  }
  return nums;
}
```
