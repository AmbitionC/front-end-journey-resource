# 去重与热度加权（知识点 + 原文）

目标：知识库里**同一个考点只有一条知识点**（哪怕不同面经里表述不同），并且**越高频出现的考点在目录树里越靠前**（热点→冷门）。每次归档新面经，都要顺带维护去重与热度。

网站侧已有基建：叶子的 `currRank`（0~5，火苗 `HeatBadge`）即"热度"，索引页默认按热度排序，目录树按 `_tree.json` 数组顺序渲染。因此**无需改网站**——只要加工时正确维护 `heat`/`currRank` 并把兄弟叶子按热度排好序即可。

## 一、原文去重（面经）

先于建面经贴判断这条原文是否已入库：

1. **同 URL**：采集器已按规范化 URL 的稳定内容 ID 幂等覆盖，天然不重复。
2. **跨 URL 转载（同文搬运）**：比对 `_inbox/**/meta.json` 里的 `contentHash`（正文归一化后的指纹）。若已有面经贴来源含相同 `contentHash` → **不重复建贴**，把这条来源并入已有面经贴（或直接出队丢弃），仅在其"来源"里补记该 URL。
3. **近似改写转载**：`contentHash` 不同但正文高度相似（你判断）→ 同样并入，不新建。

## 二、知识点语义去重 + 加权

对某面经提炼出的**每个**知识点候选：

1. **检索已有**：读 `knowledge/_tree.json`，并用站内检索 / embedding 找**语义相近**（不只看标题，看内容）的既有知识点。
2. **判定并处理**：
   - **命中近似**（表述不同但内容相近，如「事件循环」vs「Event Loop 宏任务微任务」）→ **绝不新建**：
     - 把当前面经登记进该知识点正文末尾的「## 出现于（热度来源）」列表（按面经 `key` / `contentHash` 去重，**同一帖只计一次**）。
     - `heat = 去重后来源数`；据分档表重算 `currRank`。
     - 若新面经带来**新角度/新例子/新考法**，补进知识点正文（**内容加权，而非丢弃**）；无新意则只加权、不改正文。
   - **全新考点** → 用 [`generate-knowledge-docs`](../../generate-knowledge-docs/SKILL.md) 新建知识点，`heat: 1`，来源=该面经。
3. **重排目录树**：把受影响父节点下 `knowledge/_tree.json` 的兄弟叶子按 `heat` **降序**（稳定排序，热度相同保持原序）重排，使目录树从热点→冷门。
4. **面经↔知识点互链**：面经贴里链接到对应知识点；知识点「## 出现于」反链回面经。

## 三、`heat` → `currRank` 分档

`heat` 是原始复现计数（整数，来源数）；`currRank` 是 0~5 的展示火苗，由 `heat` 映射：

| heat（去重来源数） | 1 | 2 | 3–4 | 5–7 | 8–12 | 13+ |
|---|---|---|---|---|---|---|
| currRank | 1 | 2 | 3 | 4 | 5 | 5 |

叶子上同时写 `heat`（真相源，用于排序与再计算）与 `currRank`（网站火苗展示）。faas 同步会原样透传叶子上的额外字段，无需改后端。

## 四、叶子示例（带热度）

```json
{
  "label": "事件循环 Event Loop",
  "key": "event-loop",
  "isLeaf": true,
  "filePath": "js-es6-ts/runtime",
  "contentStatus": "published",
  "heat": 6,
  "currRank": 4
}
```

## 五、存量迁移（首次启用）

现有约 300 条知识点无 `heat`。首次可按当前 `currRank` 反推一个基线 `heat`（如 currRank×2），或统一置 `heat: 1` 后随新面经自然升温。不必一次性回填，随归档增量维护即可。
