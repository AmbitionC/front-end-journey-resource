大模型与前端的结合，不只是把聊天框放进网页。真正有价值的方向，是把语言、图像、语音和工具调用能力嵌入原有业务流程，让用户更快完成一个可以验证的任务。

## 一、先用三个问题判断是否值得做

一个场景适合使用大模型，通常同时具备以下特征：

1. 输入是自然语言、图片、文件等非结构化内容，传统规则成本很高。
2. 输出允许一定概率性，并且能够由用户、规则或数据源验证。
3. 错误可以被发现、纠正或回退，不会无声地造成不可逆损失。

如果任务要求每次结果完全一致，或者一个错误就会造成付款、删除、法律承诺等严重后果，应让确定性代码负责最终决策，大模型只做建议、解释或草稿。

## 二、场景一：对话式问答与检索

这是最常见的入口，但价值并不来自“能聊天”，而是降低信息查找成本。

典型产品包括：

- 产品帮助中心：根据官方文档回答使用问题，并显示引用来源。
- 企业知识助手：在用户有权限的内部资料中检索答案。
- 数据问答：把自然语言转换为受控查询，再将结果解释成人话。
- 页面内助手：理解当前页面上下文，帮助用户完成操作。

前端需要展示流式回答、引用片段、停止生成、重新生成和“不知道”的状态。后端通常采用 RAG：先检索可信资料，再让模型基于证据回答。

风险在于用户容易把流畅表达误认为事实。不要用一个模糊的“置信度百分比”制造确定性；更可靠的做法是展示来源、时间、检索范围和缺少证据的提示。

## 三、场景二：内容生成与编辑

大模型适合把“从空白开始”变成“从草稿开始”：

- 根据商品信息生成标题、卖点和不同渠道文案。
- 将长文压缩为摘要、提纲或社交媒体版本。
- 改写语气、纠错、翻译和术语统一。
- 生成或编辑配图、图标说明和视觉方案。

好的交互不是只有一个“生成”按钮，而是让用户控制目标、受众、语气、长度、必须包含和禁止出现的内容。生成结果要支持局部修改、版本比较和恢复，不应覆盖用户原稿。

内容类场景要特别处理版权、品牌规范、事实核验和敏感信息。模型输出是候选内容，不是天然可发布内容；重要页面应保留人工审批。

## 四、场景三：结构化工作流

前端经常需要把自然语言变成确定的数据结构，例如：

- 从会议记录提取负责人、截止日期和待办事项。
- 从简历或合同中提取字段，预填业务表单。
- 把用户描述转换为搜索过滤条件、图表配置或工单字段。
- 根据检查结果生成结构化报告。

这类场景应使用 Structured Outputs 或等价的 Schema 约束，而不是用正则从一段自然语言中截取 JSON。

```ts
const TaskSchema = z.object({
  title: z.string().min(1),
  owner: z.string().nullable(),
  dueDate: z.string().date().nullable(),
  priority: z.enum(["low", "medium", "high"]),
});
```

Schema 只保证结构，不能证明字段事实正确。前端应把提取结果放进可编辑表单，突出低确定性或缺失字段，让用户确认后再提交到业务系统。

## 五、场景四：多模态理解

支持视觉的模型可以理解截图、图片和文档页面，适合：

- 截图问答和界面问题定位。
- 票据、表格、图表的辅助理解。
- 商品图片描述、无障碍替代文本生成。
- 图片内容审核和分类的候选结果。

前端必须让用户知道模型实际收到了什么：原图、压缩图、裁剪区域还是 OCR 文本。图片缩放可能丢失小字，复杂表格和空间关系也可能识别错误。

隐私同样重要。上传前说明用途与保存期限；对身份证、医疗影像、内部截图等敏感材料使用明确授权、最小化传输和访问控制。

## 六、场景五：实时语音与翻译

实时模型把交互从“提交—等待”变成连续会话，可用于：

- 语音客服和口语练习。
- 会议字幕、实时翻译和摘要。
- 无障碍语音控制。
- 游戏或虚拟角色的低延迟对话。

浏览器到实时服务通常使用 WebRTC，因为它适合媒体传输并能处理网络抖动。前端需要管理麦克风权限、连接状态、打断、静音、字幕同步和设备切换。

语音场景的体验取决于端到端延迟，而不仅是模型速度。采集、网络、语音检测、推理、合成和播放都会增加延迟。还要明确提示用户正在录音，并提供即时停止和数据删除能力。

## 七、场景六：工具与自动化

工具调用让模型可以建议查询订单、创建草稿、安排日程或调用企业接口。关键原则是：**模型提议，应用执行**。

工具可以按风险分层：

| 类型 | 示例 | 建议交互 |
|---|---|---|
| 只读 | 搜索、查询天气、读取订单 | 校验权限后自动执行 |
| 可撤销写入 | 创建草稿、添加标签 | 展示结果并支持撤销 |
| 高风险写入 | 付款、删除、对外发送 | 展示最终参数并二次确认 |

前端应显示工具名称、当前状态和对用户有意义的结果，而不是暴露内部思维过程。服务端负责参数 Schema、权限、幂等、超时和审计；用户确认不能由一段模型文本替代。

## 八、组合场景比单点能力更有价值

成熟产品通常会组合多种能力。例如“售后助手”可以：

1. 通过对话理解问题。
2. 根据图片识别商品或故障现象。
3. 从知识库检索解决方案。
4. 查询当前用户订单。
5. 生成退换货表单草稿。
6. 用户确认后创建工单。

组合会增加失败路径。每一步都要有可观察状态、明确输入输出和人工接管点，不能把整个流程藏在一个无限旋转的“思考中”。

## 九、通用前端架构

无论选择哪种场景，都建议使用浏览器 → 自有 BFF → 模型/工具的边界：

- 浏览器不保存模型密钥。
- BFF 完成登录鉴权、限流、配额、提示组装和提供方适配。
- 流式文本通过自有 SSE 协议返回；实时媒体可使用 WebRTC。
- 高风险工具在服务端重新鉴权，并通过独立确认接口执行。
- request ID 串联前端事件、模型请求与工具日志。

详细工程实现由“如何集成大模型到前端应用”章节负责；这里需要记住的是，场景不同会改变交互和风险，但不会改变安全边界。

## 十、AI 专属交互设计

### 不确定性可见

展示“基于哪些材料”“哪些字段由模型推断”“哪些步骤尚未确认”。当证据不足时允许模型拒答，而不是强迫生成一个看似完整的答案。

### 用户始终可控制

提供停止、编辑输入、重新生成、撤销和人工接管。长任务显示当前阶段，避免只展示无法解释的进度条。

### 结果可编辑、可比较

生成内容应作为版本保存。用户应能比较新旧结果，选择局部接受，而不是被迫全量替换。

### 无障碍

流式更新不要让屏幕阅读器逐 token 播报；可以按句子或完成状态更新 `aria-live`。键盘用户也要能停止生成、切换版本和确认工具。

## 十一、从一个最小场景开始

选型可以用一条漏斗：

1. 选择高频且成本明确的用户痛点。
2. 定义可测量的成功结果，例如完成时间、人工修改率、检索命中率。
3. 让输出可验证，并安排人工接管。
4. 先上线建议或草稿，再逐步增加自动化权限。

评测集应来自真实任务，覆盖正常输入、模糊输入、恶意输入、敏感信息和依赖服务失败。模型升级、提示修改或工具调整后都要回归。

## 十二、常见误区

- **为了 AI 而 AI**：如果普通搜索、表单和规则已经足够，不要增加不确定性和成本。
- **把聊天框当完整产品**：用户真正想完成的是查询、编辑、决策或操作。
- **隐藏错误与来源**：流畅回答不代表正确，引用和状态必须可见。
- **浏览器直连模型**：密钥、权限和成本控制应在 BFF。
- **一开始就全自动**：先做人机协作，验证稳定后再扩大权限。
- **只测演示样例**：真实用户输入更长、更脏，也更容易触发边界问题。

## 选择检查清单

- [ ] 场景对应一个真实、高频、可衡量的痛点。
- [ ] 输出可以由用户、规则或可信数据验证。
- [ ] 失败能够被发现、回退或转人工。
- [ ] 前端提供停止、编辑、重试和版本控制。
- [ ] 高风险操作展示最终参数并要求确认。
- [ ] 密钥、权限、配额和日志都在 BFF。
- [ ] 隐私、版权、可访问性和数据保存策略已明确。
- [ ] 有真实评测集衡量质量、延迟、成本和用户修改率。
