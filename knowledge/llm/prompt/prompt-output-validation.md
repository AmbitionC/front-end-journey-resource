模型输出永远是外部、不可信输入。结构化输出能减少解析失败，却不能证明内容真实、符合业务规则或有权触发动作。生产链路要把语法、Schema、语义、证据和授权拆成独立关卡，并为失败设置有上限的修复与安全退出。

## 六层校验

1. **传输与解析**：响应是否完整，JSON 能否解析，是否被截断；
2. **Schema**：字段、类型、必填项、枚举和嵌套结构是否符合契约；
3. **业务规则**：金额、日期、状态转换和字段组合是否合理；
4. **来源与事实**：关键声明能否回到授权证据，引用是否存在；
5. **权限**：当前主体是否能对目标资源执行动作；
6. **副作用控制**：是否需要幂等键、确认、审批或人工复核。

前一层通过只是后一层的输入条件。一个完全符合 Schema 的退款请求仍可能金额错误、订单不存在或用户无权操作。

## JSON Schema 负责结构契约

[JSON Schema 2020-12 Core](https://json-schema.org/draft/2020-12/json-schema-core)定义了描述与验证 JSON 文档的核心机制。应用可用 `type`、`required`、`enum`、数值范围、字符串模式和组合关键字表达结构约束。

Schema 应尽量收紧：禁止未知字段；金额使用整数最小单位或明确精度；日期使用约定格式；可选与 nullable 分清；版本字段显式存在。Schema 本身也要版本化，因为调用方和执行器都依赖它。

## 结构化生成仍要检查响应状态

OpenAI 的[Structured Outputs 指南](https://developers.openai.com/api/docs/guides/structured-outputs)说明，在支持范围内可要求模型输出遵循给定 Schema。即便如此，应用仍需处理拒绝、内容截断、连接失败、模型版本差异和不支持的 Schema 特性。

如果使用本地或其他供应商运行时，也要以其当前文档确认结构约束方式。例如 `llama.cpp` 的[服务器文档](https://github.com/ggml-org/llama.cpp/blob/master/tools/server/README.md)提供 grammar/JSON Schema 相关接口，但兼容字段与支持子集不应从“OpenAI 风格端点”推断。

## 验证与修复闭环

![模型输出依次通过解析、Schema、业务、证据和授权校验，失败进入有上限的修复或人工复核](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/prompt-output-validation-repair-loop-v1.webp)
*图：有效 JSON 不等于真实或已授权；任何关键关卡失败都不能直接执行副作用。*

失败时先产生机器可读诊断，如 `path=/items/2/amount`、`code=OUT_OF_RANGE`，再决定是否可自动修复：

- 纯格式错误可用确定性代码修正时，优先不用模型；
- 缺字段或枚举错误可把最小错误信息反馈给模型重生成；
- 事实缺证据时应重新检索或要求用户补充，不能让模型“修成看起来合理”；
- 权限失败绝不通过重试 Prompt 绕过；
- 超过最大次数进入安全失败或人工复核。

每次重试保存原始输出、错误列表、修复提示和结果，防止循环吞掉真实故障。

## 业务规则不能全塞进 Schema

“结束日期不得早于开始日期”“退款不得超过已支付余额”“状态只能从 pending 变为 approved”涉及字段关系或外部状态，通常由业务代码验证。对金钱、库存、权限等规则，以数据库和服务端状态为准，不以模型复述为准。

可以让模型给出 `proposed_action`，但执行器应重新读取最新资源并做原子条件更新，避免校验后到执行前状态变化产生 TOCTOU 问题。

## 证据与引用校验

要求模型返回来源 ID，而不是自由编造 URL。例如：

```json
{
  "answer": "订单已于 7 月 14 日出库",
  "citations": [
    { "source_id": "order-event-8841", "claim": "shipped_at" }
  ]
}
```

应用检查 `source_id` 是否确实在本轮授权上下文中、该来源是否包含对应字段、时间是否最新。引用存在不代表支持声明；必要时做精确字段比对或人工审查。

## 修复重试的边界

合理的修复提示只包含：原任务契约、上次输出、具体错误、允许修改范围。不要每次把完整对话无限叠加，否则上下文变长且错误可能固化。

设置总重试次数、总 token、总时间和总工具调用上限；对相同错误指纹连续出现直接停止。高风险动作即使自动修复成功，也可能仍需用户确认。

## 安全执行模式

- 默认先返回预览，不直接执行；
- 使用服务端生成的资源 ID，拒绝越权 ID；
- 写操作携带幂等键，避免重试产生重复副作用；
- 权限与租户从认证上下文获取，不从模型输出获取；
- 删除、付款、发布等动作有二次确认或审批；
- 审计日志关联输入、模型版本、校验、授权与执行结果。

## 测试矩阵

至少覆盖：非法 JSON、缺必填字段、额外字段、极端数字、错误日期关系、伪造来源、越权资源、拒绝响应、长度截断、重复重试和并发状态变化。每个用例都要断言**没有发生未授权副作用**，不能只检查返回了错误文字。

## 常见误区

- 能解析 JSON 就直接写数据库；
- Structured Outputs 等于事实保证；
- 让模型自己判断用户权限；
- 所有失败都重新问模型，不设次数与成本上限；
- 修复后只重跑失败关卡，不重新检查受影响的后续规则；
- 校验错误被吞掉，线上只看到“模型偶尔不稳定”。

## 小结

输出校验是一条逐层收紧的信任流水线：先确认可解析和结构正确，再验证业务语义、证据、权限与副作用。自动修复只适合可恢复错误，必须有预算和次数上限；事实缺失、权限不足或高风险不确定性应安全失败或转人工。
