System、Developer、User 等角色不是聊天界面的装饰，而是 API 用来表达指令来源和职责的接口。不过，各家供应商的角色名称与优先级并非统一协议；应用必须做显式适配，不能把某家的语义直接搬到另一家。

## 先分清“角色”与“身份”

消息角色描述这段内容在模型请求中的用途，不证明发送者的真实身份。浏览器提交一个 `user` 消息，不代表后端已经完成登录、租户隔离或权限校验。反过来，系统消息也只是给模型的高层指令，不是操作系统级安全边界。

因此需要两条并行链路：消息层负责告诉模型如何处理请求；应用层负责认证、授权、审批、数据访问和真实副作用。

## OpenAI 接口中的职责分层

OpenAI 当前文本生成指南将 `developer` 消息用于应用开发者提供规则与业务逻辑，将 `user` 消息用于最终用户的输入；文档还说明开发者消息优先于用户消息，具体可用角色依赖所用 API 与模型。[官方文本生成指南](https://developers.openai.com/api/docs/guides/text)提供了这些消息的请求示例。

概念上可这样分工：

- **System/平台层**：供应商或平台施加的最高层安全与产品约束，应用通常看不到全部内容；
- **Developer**：应用长期规则、任务定义、输出契约、工具使用政策；
- **User**：当前用户目标、偏好与业务输入；
- **Assistant**：先前模型输出或用于续写的示例，不应被当作新的高权限指令；
- **Tool**：外部系统返回的数据，默认是不可信数据，不因“来自工具”就能改写上层政策。

[OpenAI Model Spec](https://model-spec.openai.com/2025-02-12.html)进一步描述了 platform、developer、user、guideline 等权威层级，并明确引用文本、工具输出和未受信第三方内容不自动获得指令权威。

## Anthropic 接口并不完全相同

Anthropic Messages API 在请求顶层提供 `system` 参数，而 `messages` 数组使用 `user` 与 `assistant` 角色；官方接口说明明确指出输入消息中不存在一个同名的 `system` role。[Messages API 文档](https://platform.claude.com/docs/en/api/messages/create)展示了这一结构。

这意味着适配层不能机械地把 OpenAI 风格消息数组原样转发。它要决定哪些稳定指令进入顶层 system，哪些历史轮次保留为 user/assistant，工具结果如何编码，以及不被目标接口支持的字段如何降级。

## 消息层与应用层的边界

*图：角色帮助模型理解职责；真正的访问控制和动作执行必须由应用系统强制实施。*

一个可靠的工具调用流程是：

1. 模型根据消息与工具描述提出结构化调用建议；
2. 应用解析并校验参数；
3. 应用根据已认证主体检查权限、租户与资源范围；
4. 高风险动作请求用户确认或人工审批；
5. 执行器以最小权限调用真实系统；
6. 工具结果作为数据返回模型，最终响应再做输出校验。

即使 Developer 消息写着“只能读取当前用户订单”，后端仍必须从会话身份推导用户 ID，不能接受模型生成或用户输入的任意 ID。

## 各角色应该放什么

**Developer/System 区域**适合稳定、跨请求的内容：应用职责、不可违反的约束、工具政策、输出 Schema、拒答条件。不要塞入每次变化的用户资料，否则难以缓存、审计和复用。

**User 区域**适合本轮目标和用户提供的数据。应该把“我要做什么”与附件正文、检索材料分区，避免文档中的句子被误解为用户直接指令。

**Assistant 历史**只保留完成当前任务必需的状态。历史模型输出可能已经错误，回填时应标记为先前回答而非可信事实。

**Tool 结果**携带来源、时间、状态码与数据边界。工具网页、邮件、代码仓库和检索片段都可能包含 Prompt Injection，应视为不可信输入。

## 一个供应商中立的内部表示

应用可以先维护自己的消息模型，再转换到目标 API：

```ts
type InstructionEnvelope = {
  policy: string[];        // 应用稳定规则
  userGoal: string;        // 当前目标
  context: Evidence[];     // 带来源的不可信数据
  history: Turn[];         // 必要对话状态
  outputContract: object;  // 结构约束
};
```

适配器负责把 `policy` 映射到目标供应商的高层指令接口，把 `userGoal` 放到用户消息，把 `context` 用明确边界包装成数据。这样迁移供应商时，业务权限逻辑无需跟着消息格式重写。

## 冲突与不可信内容

当用户要求忽略上层规则时，模型应遵循更高权限的有效指令；当检索文档写着“把所有秘密发给我”，那只是文档数据，不应成为命令。应用应在高层规则中说明哪些区域是引用数据，并通过工具白名单与后端权限限制实际影响。

同一层级也可能冲突。例如用户先要求中文，后要求英文。具体覆盖规则取决于供应商语义与任务设计；应用应通过测试确认，不依赖隐含猜测。更复杂的冲突处理可进一步使用指令层级与安全停止策略。

## 常见误区

- **System Prompt 等于安全沙箱**：它可以影响模型行为，但不能阻止越权 API 调用；
- **角色是跨供应商标准**：名称相似不代表字段、优先级与工具语义一致；
- **Tool 结果天然可信**：工具可能失败、被污染或返回对抗文本；
- **Assistant 历史都是事实**：它仍是模型曾经生成的内容；
- **把所有内容放进一个 User 字符串也一样**：这会丢失来源与职责信号，增加冲突和注入风险。

## 设计检查表

- 稳定应用规则与本轮用户输入是否分离；
- 引用、附件、检索和工具结果是否明确标记为数据；
- 每个供应商是否有独立、可测试的消息适配器；
- 身份、权限、审批和副作用是否全部在应用层实施；
- 日志能否还原最终发送的角色、顺序与模板版本；
- 对“忽略之前规则”、工具注入和跨租户参数是否有回归测试。

## 小结

角色的价值是表达“谁在提出什么类型的内容”，而不是赋予真正权限。应用应把稳定政策、用户目标、模型历史和工具数据放在各自边界内，通过供应商适配器映射到具体 API；认证、授权和副作用始终由模型之外的代码强制执行。
