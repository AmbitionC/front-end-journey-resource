Few-Shot 的价值不在“示例越多越好”，而在当前请求能看到少量、正确、相关且互补的行为示范。示例库扩大后，选择、排序、标签平衡和 token 预算都会改变结果，因此示例应像检索数据一样治理，而不是永久写死在 Prompt 中。

## 示例在教什么

一个完整示例同时传递任务边界、输入结构、推理所需证据、输出格式和错误处理方式。如果示例只展示理想成功路径，模型就学不到缺失信息、拒答或工具失败时该怎么做。

示例记录至少包含：稳定 ID、输入、期望输出、任务/标签、语言、难度、来源、审核状态、敏感级别、token 估算和适用版本。错误标签、过期规则或隐私数据必须在进入选择器前淘汰。

## 四种选择策略

**静态选择**：为一个任务固定少量代表样例。简单、可复现、便于缓存，适合分布稳定的窄任务；不足是无法适应新主题。

**相似度检索**：将当前请求与示例输入向量化，检索语义相近样例。Sentence Transformers 的[语义搜索说明](https://www.sbert.net/examples/sentence_transformer/applications/semantic-search/README.html)展示了查询与语料向量相似度检索的基本做法。相似并不等于适用，还需过滤权限、语言和标签。

**多样性选择**：在相关候选中惩罚重复，覆盖不同意图、边界或输出标签。它可避免 top-k 全是同一表述，却可能牺牲局部相似度。

**规则/任务过滤**：先按产品、租户、语言、模型版本、合规等级等硬条件筛选，再做相似度与多样性排序。生产系统通常是这几种策略的组合。

## 动态 Few-Shot 流水线

![示例池经资格过滤、向量检索、相关性多样性平衡和 token 预算形成 Few-Shot Prompt](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/prompt-examples-selection-retrieval-ranking-v1.webp)
*图：敏感或错误标签样例先被拒绝；相关性、多样性与标签平衡在预算内共同决定最终集合。*

推荐顺序：

1. **资格过滤**：审核状态、语言、权限、版本、标签完整性；
2. **候选检索**：用当前请求或规范化任务描述召回 top-k；
3. **重排**：综合相关性、示例质量、多样性和标签覆盖；
4. **预算装箱**：估算固定指令、用户输入、示例与输出空间；
5. **排序**：决定示例呈现顺序；
6. **记录**：保存所选 ID、分数、顺序和选择器版本。

[Learning To Retrieve Prompts for In-Context Learning](https://aclanthology.org/2022.naacl-main.191/) 将示例选择建模为可学习的检索问题，并展示了训练检索器选择对下游模型更有用示例的路线。这提醒我们：词面或通用向量相似度只是代理指标，最终应以目标任务表现评估选择器。

## 一个可解释的重排分数

可以从简单线性分数开始：

```text
score = 0.55 × relevance
      + 0.20 × quality
      + 0.15 × label_need
      + 0.10 × recency
      - 0.25 × redundancy
```

权重不是通用答案。硬性资格规则不应折算成分数：含敏感数据或错误标签的示例即使相关度最高，也必须拒绝。对每次选择保留各项分值，才能解释“为什么用了这个例子”。

## 顺序也会产生信号

示例顺序可能影响模型对最近模式、标签分布和输出格式的关注。常见策略包括：最相关放最后靠近用户问题；从简单到复杂；按不同标签交错；固定排序以提高可复现性。

不能假设某种顺序普遍最好。应在同一示例集合上随机或系统交换顺序，测量波动。若轻微换序就导致巨大变化，说明 Prompt 对表面模式过度敏感，可能需要更清晰的规则或更好的模型。

## Token 预算怎样分配

先为输出和安全余量预留 token，再分配示例预算。选择问题近似一个带价值和成本的装箱问题：一个长示例可能比三个短示例更相关，也可能只重复已有规则。

预算紧张时优先：删除冗余解释、保留关键输入输出对；减少重复标签；将稳定格式写成 Schema，而不是每个示例重复；避免截断单个示例，因为半个输出会教出错误模式。

## 示例污染与隔离

示例库可能来自人工标注、线上成功案例或模型合成。进入库前应：去除个人信息和密钥；验证答案；标记来源；防止用户内容跨租户复用；检测文档中的注入文本；为合成样例设置独立审核状态。

用户不能通过修改自己的请求直接指定任意示例 ID。检索服务应从已授权集合中选择，并把示例当作版本化资产审计。

## 怎样评估选择器

- 与 zero-shot、固定 Few-Shot、随机选择和纯相似度 top-k 比较；
- 按标签、语言、难度和新旧分布分桶；
- 报告质量、输入 token、延迟与选择稳定性；
- 做消融：去掉多样性、标签平衡或过滤规则后发生什么；
- 检查错误时选中了哪些示例，而不只修改最终 Prompt；
- 用保留集调权重，避免选择器过拟合开发集。

OpenAI 的[Prompt engineering 指南](https://developers.openai.com/api/docs/guides/prompt-engineering)把示例作为指导模型输出的重要结构；动态系统还必须把“哪些示例进入请求”纳入同等严格的版本与评估。

## 常见误区

- top-k 相似度最高就直接使用，不做资格过滤；
- 只放正例，缺少拒答与边界示例；
- 示例越多越好，挤掉用户证据与输出空间；
- 截断长示例后仍保留残缺标签；
- 不记录示例 ID 与顺序，失败无法复现；
- 用线上用户数据做示例，却没有授权和隔离。

## 小结

动态 Few-Shot 是一个受约束的检索与排序系统：先过滤不合格样例，再在相关性、多样性、标签覆盖和 token 成本之间平衡，最后用任务评估而非相似度分数验收。示例库、选择器和顺序都要版本化，才能让效果可解释、可回归。
