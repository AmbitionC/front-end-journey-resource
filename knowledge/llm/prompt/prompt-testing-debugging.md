Prompt 不是一段“凭感觉调到顺眼”的文案，而是模型系统中的可版本化程序。可靠迭代需要固定测试集、可解释的判定器、失败分类和回归门禁，让每次修改都回答：改善了什么、破坏了什么、代价是多少。

## 先写可观察的行为契约

测试目标应来自产品要求，而不是“回答看起来不错”。把要求拆成可观察行为：

- 必须输出哪些字段，字段类型与范围是什么；
- 哪些信息必须来自给定证据，何时应该拒答；
- 哪些语气、语言和长度是可接受的；
- 哪些内容不得泄露，哪些动作必须请求确认；
- 延迟、token 与成本预算是多少。

一个样例可以同时有多个断言。例如客服回复既要引用正确订单状态，也要避免承诺未授权退款，还要控制在 120 字内。把这些目标合成一个模糊总分，会让失败原因消失。

## 测试集要覆盖分布与边界

从真实日志中脱敏抽样，再补充人工构造的边界用例。至少分为：常见成功路径、输入缺失、歧义、相互冲突的证据、长上下文、对抗指令、工具失败和必须拒答的请求。

每条用例保存稳定 ID、输入、必要上下文、期望属性、标签和来源。训练/调试集用于快速迭代；保留集只在候选版本定型时运行，避免反复看答案造成“对测试集写 Prompt”。[OpenAI 的评估最佳实践](https://developers.openai.com/api/docs/guides/evaluation-best-practices)也强调按任务定义目标、收集代表性数据并持续评估，而不是依赖少量直觉示例。

## 判定器按确定性从高到低组合

优先使用便宜、可复现的检查：

1. 解析是否成功，JSON 是否匹配 Schema；
2. 必填字段、枚举、长度、正则和引用 ID 是否有效；
3. 代码能否编译、测试是否通过，数字能否重算；
4. 答案中的关键声明能否在授权证据中定位；
5. 规则难以表达的语义质量，再交给模型评分器或人工。

模型评分器适合比较相关性、完整性或风格，但它也会受顺序、措辞和自身偏差影响。应先用人工标注集校准，给出评分量表与正反例，检查评分器和人工的一致性；高风险样本仍由人工复核。OpenAI 的[Evals 指南](https://developers.openai.com/api/docs/guides/evals)和 Anthropic 的[评估工具文档](https://platform.claude.com/docs/en/test-and-evaluate/eval-tool)都把数据集、评分标准与多次运行作为评估闭环的核心。

## 调试闭环

![从测试数据、判定器和失败分类回到 Prompt 修订与回归门禁的闭环](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/prompt-testing-debugging-eval-loop-v1.webp)
*图：先定位失败类型，再做最小修复；新版本只有通过回归门禁才进入下一阶段。*

一次可复现的运行应保存：模型与端点版本、Prompt/模板版本、渲染后消息、采样参数、工具定义、检索结果、原始输出、解析结果、评分和延迟。敏感字段要脱敏或只存哈希与引用。

收到失败后先归类：

- **指令缺失**：从未明确要求某行为；
- **指令冲突**：多个位置给出不兼容目标；
- **上下文缺失**：模型没有完成任务所需信息；
- **检索/工具失败**：错误证据或错误工具结果进入上下文；
- **格式失败**：语义可能正确但无法解析；
- **事实/推理失败**：有证据却得出错误结论；
- **安全/权限失败**：执行了不应执行的请求；
- **非确定性波动**：同一输入在多次运行中不稳定。

分类后只改变一个主要变量，再重跑相关切片和全量回归。一次同时换模型、Prompt、示例和参数，即使分数上升也无法解释原因。

## Pairwise 与回归比较

比较旧版 A 与候选 B 时，尽量对同一批输入成对运行。客观指标直接比较；主观指标让评分器在隐藏版本标识后判断 A/B/平局，并随机交换呈现顺序以检测位置偏差。

除了平均分，还要报告每个失败标签的胜、负、平，特别关注严重回退。例如总体提升 3%，但“必须拒答”切片从 98% 降到 90%，不应发布。小数据集还应展示原始计数，不用虚假的小数精度掩盖不确定性。

## 把回归门禁放进 CI

CI 不必每次跑最昂贵的全集，可以分层：

- 提交时运行渲染快照、Schema、静态规则和少量固定样例；
- 合并前运行代表性回归集与多次采样；
- 定时任务运行完整评估、模型评分与成本/延迟对比；
- 线上按版本采样，监控分布漂移和新失败簇。

门禁要预先写清楚：关键规则必须 100% 通过，其他指标允许多大退化，成本与 P95 延迟上限是多少。随机模型输出不适合用“逐字相等”断言，应检查结构、关键事实和允许的语义范围。

## 一个最小评估记录

```json
{
  "case_id": "refund-no-authorization-017",
  "tags": ["permission", "tool-result-conflict"],
  "prompt_version": "support@2.4.0",
  "assertions": {
    "valid_schema": true,
    "no_refund_promise": true,
    "asks_for_confirmation": true
  },
  "latency_ms": 842,
  "input_tokens": 1260,
  "output_tokens": 96
}
```

记录断言结果比只保存一个总分更利于追踪。失败时可以按标签聚合，判断应改 Prompt、数据、工具还是应用代码。

## 常见误区

- 只保留“标准答案”，却没有可判定属性；
- 用同一个模型生成答案又给自己评分，不做人工校准；
- 把格式正确当成事实正确；
- 只看平均值，不看安全与拒答切片；
- 测试时固定模型，线上却使用浮动别名且不记录版本；
- 每次失败都继续加文字，最终形成相互冲突的超长 Prompt。

## 小结

Prompt 调试的核心是可复现的证据链：行为契约决定数据集，确定性规则和经校准的评分器给出信号，失败分类指导最小修改，回归门禁阻止局部优化破坏其他场景。这样 Prompt 才从“手艺”变成可维护的工程资产。
