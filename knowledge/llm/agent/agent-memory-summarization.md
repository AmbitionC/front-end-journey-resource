会话摘要是有损压缩，不是原始事实的新版本。它的目标是在有限上下文中保留任务继续所需的信息，同时让每个关键结论能回到原始消息、文件或工具证据。把摘要当真相，会让一次误解在后续几十轮持续放大。

## 先定义要保留什么

不同摘要服务不同目的。交接摘要关注目标、状态与下一步；记忆摘要关注稳定偏好和重要经历；证据摘要关注来源与结论；用户可读摘要关注解释。一个通用保留清单包括：

- 用户目标、不可变约束和显式决定；
- 已完成/未完成任务及完成证据；
- 关键实体、版本、时间和引用关系；
- 已验证事实及来源；
- 假设、冲突、不确定性和失败路径；
- 仍有效的审批、预算和风险边界。

不要把语气、重复寒暄或完整工具日志都塞进工作摘要；也不要因为内容很长就丢掉否定词、单位、例外和授权条件。

## 原始记录、提炼事实与摘要三层分离

```ts
type SummaryFact = {
  id: string;
  statement: string;
  sourceMessageIds: string[];
  confidence: number;
  temporalScope?: string;
  status: 'confirmed' | 'inferred' | 'disputed';
};
```

原始时间线不可由摘要覆盖；提炼事实是带 provenance 的结构化记录；工作摘要只是当前上下文视图。摘要更新时引用 fact ID，而不是反复对“上一版自然语言摘要”再总结，否则错误会像有损转码一样累积。

## 为什么不能只依赖更长上下文

[Lost in the Middle](https://arxiv.org/abs/2307.03172)及其 [TACL 版本](https://aclanthology.org/2024.tacl-1.9.pdf)发现，长上下文模型利用相关信息的效果会受信息位置影响，相关内容位于中间时可能表现更差。结论不是“长上下文没用”，而是输入长度和信息可用性不是同一件事。

摘要可以把当前决策所需证据移到显著位置，但必须防止压缩者决定性遗漏。对于法律条款、精确代码、表格和用户原话，优先保留引用或原文片段，不让模型自行改写关键数值。

## 增量摘要流程

[LangGraph memory 文档](https://docs.langchain.com/oss/python/langgraph/add-memory)给出扩展 running summary 并移除较旧消息的模式。生产流程可细化为：

1. 根据 token/轮次/阶段触发，不在每条消息后重写；
2. 冻结待压缩消息区间和版本；
3. 提取实体、决定、事实、任务状态和证据引用；
4. 解析“它、那个文件、上一步”等指代；
5. 与现有事实表去重、检测矛盾；
6. 生成结构化摘要和短自然语言视图；
7. 运行覆盖率与关键约束校验；
8. checkpoint 后再从 active context 移除旧消息。

若摘要失败，保留原消息并缩小新输入，而不是先删后补。

## 压缩策略

可组合三种方式：

- **抽取式**：保留关键原句/片段，准确但不够紧凑；
- **结构化**：把目标、决定、事实、任务和来源写入 schema，适合机器消费；
- **生成式**：写连贯短文，阅读友好但最容易引入推断。

可靠系统以结构化层为核心，生成式视图由结构化事实派生。每条字段标记 confirmed/inferred/disputed，避免把模型推断写成用户决定。

## 质量门

摘要校验不只看语言通顺。自动检查包括：所有未完成任务是否出现；数值、日期、否定和权限是否一致；source ID 是否存在；摘要是否包含原记录没有的新实体；是否遗漏冲突与不确定性。高风险场景还可让独立 verifier 对照原记录。

覆盖率不是越高越好。目标是保留完成当前任务所需的高价值信息。可以按 rubric 加权：目标/约束和用户决定最高，完成证据次之，闲聊最低。

## 何时建立 checkpoint

适合在阶段完成、交接、上下文接近阈值或长期任务暂停前摘要。checkpoint 包含摘要版本、所覆盖消息范围、事实记录、计划状态、工件 hash 和原始证据引用。恢复时先验证引用仍可访问、底层资源版本是否变化，再继续执行。

不要在一次高风险工具调用尚处于“结果未知”时压缩掉原始请求与响应；这会破坏幂等和恢复判断。

## 常见错误

- 对上一版摘要反复摘要，不再看原始证据；
- 把“用户考虑 A”压成“用户选择 A”；
- 省略“不允许发布”等否定约束；
- 合并两个同名实体或不同时间版本；
- 删除旧消息后才发现 summary schema 无效；
- 将包含敏感信息的摘要复制到更广的日志或长期存储。

## 测试与观测

准备长对话 fixture，包含目标改变、撤回决定、同名实体、矛盾事实、精确数值、工具失败和权限边界。比较原记录与压缩后 Agent 的任务成功、约束遵守、事实正确和来源引用。还要做多轮压缩测试，观察误差是否累积。

记录 summary version、覆盖区间、输入/输出 token、保留 fact 数、验证错误、被移除消息和恢复命中。指标包括压缩率、关键事实召回、虚构率、冲突保留率、引用有效率、恢复成功率和后续重新读取原文次数。

## 小结

安全摘要保留两条线：原始证据线和可执行状态线。系统从不可变记录提取带来源事实，解析指代与冲突，生成工作摘要，经过覆盖和不确定性校验后才替换 active context 中的旧消息。摘要可以短，但不能失去证据、版本和边界。
