Agent Run Loop 把一次自然语言请求推进为一系列可控状态：读取观察，决定下一步，执行动作，吸收结果，再判断是否继续。它的工程价值不在“循环调用模型”，而在于每一轮都有输入快照、预算、进展证据和可判定终止条件。

## 一轮到底包含什么

可以把最小循环写成六步：

1. `observe`：读取当前任务状态、允许工具、上轮结果与剩余预算；
2. `decide`：模型选择输出最终答案、调用工具、请求澄清、转交或放弃；
3. `validate`：校验结构、权限、审批和预算；
4. `act`：执行被允许的动作；
5. `ingest`：把结构化结果写入状态与 trace；
6. `evaluate`：判断目标是否达成、是否取得进展以及能否进入下一轮。

[OpenAI Agents SDK 的运行文档](https://openai.github.io/openai-agents-python/running_agents/)描述了 runner 在模型输出、工具调用、handoff 和最终输出之间循环的当前行为。框架可以替你调度，但业务目标、进展判定和安全终态仍由应用定义。

## 显式状态优于聊天记录猜测

运行状态至少包含目标、约束、已确认事实、未解决问题、动作历史、资源版本、剩余预算和当前阶段。不要只把整段聊天重新喂给模型，让它从文本中猜“做到了哪里”。结构化状态可做并发控制、恢复、去重与离线评估。

```ts
type RunBudget = {
  maxTurns: number;
  maxToolCalls: number;
  maxTokens: number;
  deadlineAt: string;
  maxCost: number;
};

type Terminal = 'complete' | 'abstain' | 'escalate' | 'failed' | 'cancelled';
```

轮次是调度单位，不是进展单位。一个回合可能调用多个并行只读工具，也可能只等待审批。状态版本应在每次写入时递增，旧 worker 的迟到结果不能覆盖新状态。

## 多个预算共同约束

仅有 `maxTurns` 不够。某轮可能产生昂贵工具调用或超长上下文；也可能用很少轮次把系统拖到超时。生产系统通常同时限制轮数、模型 token、工具次数、并发度、墙钟时间、费用、下载字节和重试次数。

[Agents SDK Run 参考](https://openai.github.io/openai-agents-python/ref/run/)提供了最大轮次等运行配置和异常语义的当前接口。应用应在每轮开始与每次外部动作前重新计算剩余预算，确保子调用的 timeout 不超过全局 deadline。预算耗尽应进入明确终态并保留未完成原因，而不是伪装成成功答案。

## 终止不只分成功和失败

- `complete`：可验证的成功条件已满足，并有结果证据；
- `abstain`：信息不足或置信度不够，系统主动不猜；
- `escalate`：需要权限、人工判断或超出能力范围；
- `failed`：不可恢复错误或重试预算耗尽；
- `cancelled`：调用方请求停止且本地执行已确认结束。

模型声称“完成”只能是候选信号。系统应检查所需产物存在、工具回执有效、业务不变量满足。例如“创建工单”必须有受信任服务返回的工单 ID；自然语言里写出编号不能作为证据。

## 进展检测防止空转

为每轮计算 progress fingerprint：未完成子目标集合、关键状态版本、已获得证据、最近动作及其归一化参数。若连续多轮 fingerprint 不变，或重复相同工具和参数得到相同结果，就进入修复策略：换方法、请求缺失信息、缩小目标或停止。

重复不总是错误。分页、轮询和重试需要把 cursor、attempt、nextEligibleAt 等纳入指纹。相反，仅改变措辞却没有新证据，不应被视为进展。

一个实用规则是同时设置：相同动作重复上限、连续无新事实上限、连续验证失败上限，以及同一错误码的重试上限。达到阈值后把 trace 摘要交给 fallback 或人工，而不是继续消耗上下文。

## ReAct 与可审计轨迹

[ReAct 论文](https://arxiv.org/abs/2210.03629)展示了推理与行动交错、再用环境观察推进任务的范式。工程系统可以采用“行动—观察—更新”的外部结构，但不应把私有推理文本当作审计依据。应记录可验证的决定摘要、工具名、参数来源、策略结果、观察和终态理由。

这样既能复盘“为什么调用了这个工具”，又不会依赖不可稳定复现的隐式思维过程。需要解释时，生成面向审计的结构化 rationale，例如选项、证据引用、约束与置信度。

## 错误如何回到循环

错误先分类，再决定是否继续：结构错误可让模型修参；授权拒绝通常不能重试；限流按服务器建议等待；网络瞬断可在幂等前提下退避；副作用状态未知则先查询状态。工具的自由文本错误不能直接控制循环，必须映射为稳定 code 和 retryability。

每次重试都消耗原任务预算，不能创建一个新的无限子预算。若结果晚到，按 call ID 与 state version 对账，禁止把旧结果盲目写进当前轮。

## 测试与可观测性

用确定性 fixture 测试：首轮完成、多工具链、参数修复、重复动作、无进展、预算耗尽、审批暂停、handoff、取消、迟到结果和恢复。属性测试可以断言任何路径最终进入合法状态、预算永不为负、终态后不再调用工具。

trace 为 run/turn/model/tool 分层，记录状态版本、剩余预算、动作指纹、结果码、progress delta 和终止原因。核心指标包括任务成功率、平均轮数、工具放大倍数、无进展率、重复动作率、预算终止率、p95 时长和每个成功任务成本。

## 常见误区

- 把“模型没有继续调用工具”直接当成功；
- 只设最大轮次，不设时间、费用与工具预算；
- 每轮重放所有原始输出，导致上下文膨胀；
- 工具失败后无分类地再次调用；
- 用措辞变化判断进展；
- 终态后仍接收旧 worker 写入；
- 把隐藏推理文本当作唯一审计记录。

## 小结

Run Loop 是状态机、预算器与证据循环的组合。观察必须来自版本化状态，动作先过确定性门禁，结果再写回，进展由可验证变化衡量。完整、放弃、升级、失败和取消都是正式终态；只有这样循环才会在该停时停，而不是靠轮数碰运气。
