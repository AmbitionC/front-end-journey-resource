Agent 能从用户输入、记忆、工具结果和密钥系统读取数据，也能通过模型请求、工具参数、日志、文件和网络把数据带出。数据外泄防御的第一步不是寻找一个“敏感词过滤器”，而是画出来源、转换、接收方和边界，在每个确定性出口执行最小化、脱敏、授权、目的地与 DLP 策略。

[OWASP Sensitive Information Disclosure](https://genai.owasp.org/llmrisk/llm022025-sensitive-information-disclosure/)把个人、金融、健康、凭证和专有数据列为风险，并明确指出仅依赖系统 Prompt 限制返回并不可靠，可能被 Prompt Injection 绕过。关键控制必须位于模型之外的数据路径上。

## 先做数据流盘点

列出资产与处理：

| 阶段 | 例子 | 关键问题 |
| --- | --- | --- |
| 来源 | 用户、CRM、记忆、文件、密钥、工具 | 谁拥有、什么分类、允许用途？ |
| 转换 | 检索、摘要、OCR、模型、代码执行 | 是否产生派生敏感数据？ |
| 存储 | 状态、缓存、日志、Trace、向量库 | 加密、租户、保留与删除？ |
| 出口 | 模型 API、工具、邮件、文件、网络 | 接收方、地区、协议和授权？ |

模型输入本身就是一次数据传输。即使输出没有泄露，向第三方模型发送机密文档也可能违反用途与地区策略。工具参数和遥测同理。

## 数据分类与标签传播

使用少量可执行分类，例如 Public、Internal、Confidential、Restricted，并附租户、主体、用途、地区和保留期。标签随派生数据传播：从 Restricted 文档生成的摘要默认仍是 Restricted，除非经过验证的去标识化流程降低级别。

分类不只靠模型猜。来源系统提供元数据，Schema 标记敏感字段，DLP 发现未知内容；冲突时采用更严格级别。模型可以建议，但策略引擎决定。

## 最小化

在每个出口前只选择完成任务必需字段。查询客户订单不需要发送整个客户档案；工具需要对象 ID 时，不附带备注和邮箱；模型摘要使用相关片段而非整库。

最小化比事后过滤可靠，因为未发送的数据无法从该路径泄露。它还降低 token、日志和审计成本。Schema 采用允许列表，未知字段不透传。

## 脱敏与令牌化

根据用途选择掩码、哈希、令牌化或聚合。不可逆哈希仍可能被字典攻击；token mapping 是敏感系统，需要独立保护。脱敏前先识别数据类型和上下文，避免把订单金额误当手机号，也避免遗漏自由文本中的凭证。

密钥、访问令牌和密码采用“永不进入模型/日志”的硬规则，命中时阻断而不是掩码后继续。测试使用合成 canary secret，验证每个出口能检测且不真的暴露生产秘密。

## 授权：谁为谁发送什么

出口请求携带用户 subject、Agent actor、目的、数据分类、接收方和动作。策略同时检查用户是否有权读取、Agent 是否被委托外发、目标是否获准、用途是否匹配。拥有读取权不自动拥有通过邮件或外部模型发送权。

高风险外发需要人工批准，界面显示接收方、字段、分类和不可逆影响。批准绑定内容摘要和短期；参数变化后重新审批。

## 目的地与网络出口

域名允许列表、TLS 身份、DNS 解析、IP/端口和工作负载身份共同判断目标。仅允许 443 不足以区分合法服务与攻击者站点；仅固定 IP 又会被 CDN 与 DNS 变化困扰。通过 egress gateway 统一策略，工作负载不直接访问互联网。

[Kubernetes NetworkPolicy](https://kubernetes.io/docs/concepts/services-networking/network-policies/)可以在网络插件支持时控制 Pod 的 L3/L4 流量，并说明 Pod 默认对 egress 是非隔离的。它是基础隔离，不提供完整域名、应用身份、payload 与数据分类策略，因此还需网关或代理层。

## 模型请求出口

为模型供应商建立数据处理配置：允许数据等级、地区、保留/训练设置、模型类型和账号。请求构造器按分类选择供应商或本地模型；Restricted 数据若无获准路径则拒绝或降级，而不是偷偷选择默认 API。

记录供应商、实际模型、区域、输入分类、token 和策略结果，但不记录原文。工具调用和模型请求使用不同出口策略，避免工具返回的 URL 被模型诱导直接访问。

## 日志与 Trace 出口

日志常是被忽略的外泄通道：Prompt、工具参数、异常、URL 查询、堆栈环境变量都可能含秘密。应用内先脱敏，采集器二次防御，存储按字段访问和短保留；调试证据放受控仓库，日志只保存引用。

指标标签禁止用户 ID、完整 URL 和自由文本，防止高基数与泄露。Trace 属性采用允许列表，内容记录默认关闭。

## 文件与展示出口

生成报告、下载、剪贴板、浏览器渲染和邮件附件都属于出口。文件生成器标注分类，下载服务再次授权，临时 URL 短期且绑定主体，缓存响应使用正确头。表格防止隐藏列和公式注入，PDF/图片 OCR 也可能包含敏感内容。

输出 DLP 检查结构化字段与自由文本，发现时可删除字段、替换、要求审批或阻断。策略决定动作，模型不能自行选择“看起来不敏感”。

## 检测、阻断与响应

审计记录来源、分类、目的、接收方、策略、批准、数据量和 Run/Trace。检测异常体积、未知域名、首次接收方、密钥模式、跨租户、编码/压缩绕过和工具链组合。

阻断事件包含可调查证据但不复制秘密。高风险命中触发凭证轮换、网络隔离、历史查询和影响评估。若数据已外发，不能只修 Prompt；要进入事故响应、通知与补偿流程。

[NIST AI RMF Generative AI Profile](https://doi.org/10.6028/NIST.AI.600-1)强调数据隐私、信息安全和持续风险管理。新增数据源、模型供应商、工具或地区都应重新审查数据流与控制。

## 常见误区

- 只在 Prompt 写“不要泄露”，没有确定性出口策略。
- 拥有读取权限就允许向任意工具或邮件发送。
- 只保护最终回答，忽略模型输入、日志和工具参数。
- 用 NetworkPolicy 允许 443，误以为已有域名和 payload 控制。
- 脱敏摘要后就降低分类，却没有验证去标识化效果。
- 阻断日志复制完整密钥作为证据，造成第二次泄露。

## 小结

数据外泄防御覆盖来源、转换、存储与所有出口。分类标签随派生数据传播，每个 sink 前执行最小化、脱敏、授权、目的地和 DLP；Secrets 永不进入模型或日志，批准绑定具体数据与接收方。通过统一 egress、审计和事故响应，即使模型被诱导，系统也不会自动把可读数据变成可外发数据。
