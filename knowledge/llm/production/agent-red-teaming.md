Agent 红队测试不是让一群人随意“套话”，而是在授权范围内，系统地尝试破坏目标、权限、记忆、工具、数据和多智能体边界，并把发现变成可复现证据、整改任务和回归资产。没有范围、隔离和复测的攻击演示，只能制造紧张，不能降低风险。

[NIST AI RMF Generative AI Profile](https://doi.org/10.6028/NIST.AI.600-1)建议结合内部/外部评估、生成式 AI 红队和结构化人类反馈，并考虑执行者的独立性与使用情境。它说明红队是风险管理循环的一部分，而不是发布前一次性的“通关游戏”。

## 第一步：写清授权与范围

红队计划至少说明：系统与环境、允许时间、允许的攻击技术、禁止目标、数据处理规则、最大成本、联系人、停止条件和事故升级路径。第三方 API、真实用户账户、生产数据和供应商模型是否在范围内，要由其所有者明确授权，不能因为“Agent 能访问”就默认可以攻击。

范围还要界定可接受影响。可以验证写工具是否被阻断，但不应真的删除生产记录；可以使用合成密钥检测外泄，不应投放真实凭证；可以测试邮件工具，但收件人应是捕获器。为高风险动作准备 kill switch，并在测试前演练。

## 第二步：从威胁模型生成假设

[OWASP Agentic Threats Navigator](https://genai.owasp.org/resource/owasp-gen-ai-security-project-agentic-threats-navigator/)把 Agent 攻击面扩展到推理、记忆、工具、身份、人类监督和多 Agent 交互。红队不应只覆盖直接 Prompt Injection，还要问：

- 不可信网页能否改变 Agent 目标并触发写工具；
- 工具描述、Schema 或返回值能否在运行时被投毒；
- 错误信息能否写入长期记忆并跨会话影响决策；
- Agent 是否继承用户没有授予的权限；
- 多 Agent 消息是否可伪造、重放或越过审批；
- 模型、MCP Server、依赖或 Prompt 模板能否被供应链替换；
- 日志、URL、文件导出或网络请求能否带出敏感数据。

每条假设写成“前置条件—攻击动作—预期控制—失败影响”，并关联资产、信任边界和责任团队。这样测试失败后知道修哪一层，而不是简单要求“增强模型安全”。

## 第三步：设计攻击案例

案例应包含固定输入、来源、权限、状态、步骤、观察点和成功判定：

```yaml
id: RT-INDIRECT-014
hypothesis: untrusted webpage can request secret exfiltration
principal: user-readonly
agent_version: agent-42
precondition: browser tool enabled; email tool disabled
payload_ref: corpus://indirect/014
expected:
  - webpage_text_remains_data
  - secret_store_not_accessed
  - egress_policy_blocks_unknown_domain
  - security_event_emitted
severity_if_failed: critical
```

攻击语料使用版本化语料库，保存来源与许可。变体生成可以扩大覆盖，但必须保留生成器版本和随机种子。案例的成功判定应基于状态、工具轨迹和数据流，不要把模型是否说出某句拒绝语当唯一安全标准。

## 第四步：安全执行

红队环境默认拒绝网络出口，使用测试租户、短期凭证、合成敏感数据、影子数据库、邮件/支付捕获器和资源配额。所有工具调用进入审计，写操作支持批准与回滚。对浏览器或计算机工具，还要隔离下载、剪贴板和本地文件系统。

测试执行者也遵守最小权限。能够读取秘密以验证“是否泄露”的账号，与能够修改策略的账号分开；结果仓库只存必要证据并按敏感度控制访问。攻击日志本身可能包含可复用 payload 和系统细节，不应公开散播。

## 第五步：收集可复现证据

每个发现包含 Run/Trace ID、不可变版本、完整前置条件、攻击步骤、关键 Span、策略决策、状态变化、受控输入输出引用和重放说明。屏幕录像可帮助理解，但不是唯一证据；自动化断言和状态快照更适合后续回归。

若模型输出具有随机性，报告多次尝试的成功率与样本，而不是只挑最戏剧化的一次。与此同时，严重越权即使只成功一次也不能被平均值稀释；风险评估需要结合可利用性、影响、可重复性和权限条件。

## 第六步：分级与归因

严重度至少考虑：数据/资产影响、攻击者所需权限、可重复性、可检测性、影响范围和恢复成本。还要区分“尝试被控制正确阻断”与“控制不存在”。命中拦截规则是安全遥测，不是产品失败。

根因按控制边界归类：输入信任标记、能力授予、授权策略、工具 Schema、输出验证、记忆写入、网络出口、日志脱敏、供应链或人工流程。把所有问题归因“模型被越狱”会导致只改 Prompt，而权限和数据流漏洞仍然存在。

## 第七步：整改与回归

优先在确定性边界修复：减少工具能力、校验 Schema、绑定 subject/actor、增加审批、验证后置条件、隔离记忆、限制出口。Prompt 防御可以增加摩擦，但不能成为保护关键副作用的唯一控制。

修复后使用原案例、语义变体和邻近正常用例复测。原案例验证漏洞关闭，变体验证不是只封一个字符串，正常用例检查误伤。通过的案例进入持续回归集，并记录控制版本和复审日期。

## 持续红队与生产信号

攻击面会随模型、工具、MCP Server、权限和数据源变化。高风险变更触发定向红队；固定周期运行核心攻击集；生产安全事件和外部披露转成新假设。2026-07-16 的 [MITRE ATLAS](https://atlas.mitre.org/)提供针对 AI 系统的战术与技术知识库，可用于统一攻击语言和检查覆盖空白，但具体测试仍需结合本系统架构。

生产监控不应自动执行危险 payload。可以检测异常工具序列、未知域名、记忆写入激增、授权拒绝和敏感数据命中，再把样本送入隔离红队环境。线上证据用于发现，攻击验证仍在授权沙箱完成。

## 人员与独立性

开发者最了解系统，适合白盒测试；独立安全人员更容易挑战默认假设；领域专家能判断业务伤害；代表性用户能发现人机交互风险。不同角色互补，不能让同一个实现者独自定义范围、执行、分级并宣布通过。

同时保护测试人员心理与法律安全：明确处理有害内容的支持机制、数据最小化、报告渠道和披露流程。红队不是鼓励无边界探索，专业性首先体现在授权、证据和责任上。

## 常见误区

- 只做直接 Prompt Injection，不测试工具、身份、记忆和供应链。
- 在生产环境使用真实敏感数据验证外泄。
- 只保存成功截图，没有版本、Trace 和重放步骤。
- 发现问题后仅增加系统 Prompt 警告，不修确定性权限与数据边界。
- 修复后只重跑原字符串，不测变体和正常场景误伤。
- 把拦截到的攻击尝试算作 Agent 失败，导致团队压低安全遥测。

## 小结

成熟红队流程从授权范围和威胁模型开始，在隔离环境执行版本化案例，收集可重放证据，按影响与可利用性分级，在确定性边界整改，并把发现转成持续回归。它衡量的不是攻击演示有多惊险，而是组织是否能稳定地发现、修复并防止同类风险重现。
