把 Agent 放进容器，不等于它已经安全。容器里的代码仍与宿主机共享内核；一旦运行时、内核、挂载、凭据或编排配置存在缺口，不可信任务就可能越过预期边界，读取其他租户数据、控制宿主机，甚至借云凭据扩散到更大的环境。所谓“沙箱逃逸防御”，不是寻找一个绝对不会被突破的盒子，而是把预防、限制、检测和恢复做成彼此独立的多层控制。

[NIST SP 800-190](https://csrc.nist.gov/pubs/sp/800/190/final)把容器风险放在镜像、注册表、编排器、容器运行时和宿主操作系统等多个层面分析。这个视角很重要：只检查镜像漏洞，或者只写一份容器权限配置，都无法覆盖整条执行链。

## 先定义真正的安全边界

沙箱的输入不只是用户写的一段代码，还包括模型生成的命令、下载的依赖、解压后的文件、浏览器内容和工具返回值。把这些内容统一视为不可信，安全目标才会清晰：

- 任务只能访问分配给它的文件、网络目标、CPU、内存和执行时长；
- 即使获得容器内 root，也不能获得宿主机、其他任务或控制平面的权限；
- 任务无法读取长期云凭据、宿主 socket、设备或敏感挂载；
- 越界尝试能够被阻止、关联到 Run，并触发隔离与调查；
- 沙箱实例可以被销毁和重建，不能把本地可变状态当作唯一事实。

安全评审应写出“保护什么、攻击者控制什么、突破一层后还能做什么”。如果团队只说“使用了 Docker”，就还没有完成威胁建模。

## 预防层：减少可攻击面

镜像从最小基础开始，固定摘要而非浮动标签，构建阶段生成 SBOM 并扫描已知漏洞。运行时不安装编译器、包管理器和调试工具；若任务确实需要这些能力，使用专用镜像而不是把通用生产镜像做成工具箱。

身份方面使用非 root 用户，禁止特权模式、host PID、host network、hostPath 和容器运行时 socket。根文件系统只读，可写目录通过小容量临时卷显式开放。Linux capabilities 从全量删除，再按实际系统调用补最小集合。不要把“容器内 root 不是宿主 root”当成允许高权限的理由。

[Kubernetes Pod Security Standards](https://kubernetes.io/docs/concepts/security/pod-security-standards/)中的 Restricted 配置提供了一组可落地的基线：限制特权提升、要求非 root、使用受限 seccomp 配置并控制 capabilities。它是基线而非完整方案，具体字段仍需随集群版本验证。

## 内核与宿主隔离

seccomp 通过允许列表减少进程可调用的系统调用，但它不是完整沙箱。[Kubernetes 安全检查清单](https://kubernetes.io/docs/concepts/security/security-checklist/)同时强调工作负载隔离、最小权限、网络策略、密钥保护和持续更新。工程上应把 seccomp、AppArmor 或 SELinux、用户命名空间、只读文件系统和 cgroup 限额组合起来。

高风险代码执行可以进一步使用独立内核或轻量虚拟机边界，并把这类任务调度到专用节点。宿主节点不承载控制面组件，不挂载生产数据库凭据，使用最小操作系统并快速轮换。边界越强，启动开销和资源成本通常越高，因此应按任务风险分级，而不是所有任务共用同一运行池。

## 网络出口是关键控制面

很多真实攻击不需要完成内核逃逸：只要容器能访问云元数据、内部管理接口或任意公网，就可能直接窃取凭据或外传数据。默认拒绝出站，按工具能力开放域名、端口、协议和请求方法；DNS 解析结果与实际连接目标都要校验，避免 DNS rebinding、重定向和代理绕过。

对 HTTP 工具使用受控出口代理，阻止环回、链路本地、私有网段和云元数据地址。下载限制大小、MIME、重定向次数和解压比例。网络策略应绑定任务身份与策略版本，而不是靠应用在 Prompt 中提醒模型“不要访问内网”。

## 凭据与数据最小化

沙箱不接触长期密钥。需要调用外部服务时，由可信网关根据任务身份签发短时、窄权限的能力令牌，或者由代理代发请求。令牌限制受众、动作、资源和有效期，任务结束立即失效。环境变量、命令行参数和共享目录都可能被进程读取，不适合存放高价值长期凭据。

输入文件复制到任务专属目录，输出先进入隔离区，经类型、大小、恶意内容和策略检查后再发布。不同租户、不同信任级别的任务不能复用可写工作目录。镜像层、构建缓存和崩溃转储同样需要纳入数据清理。

## 检测、终止与恢复

运行时检测关注异常系统调用、权限提升、敏感路径访问、意外子进程、网络扫描、元数据访问和大量数据外发。事件必须带 tenant、run、sandbox、image digest、node 和 policy version，才能把告警还原到具体任务。检测到高置信越界时先冻结或终止实例，撤销令牌，隔离节点，再保存只读证据。

不要在可疑实例里“修好后继续跑”。把任务状态保存在沙箱外的可信存储，从最后一个已验证检查点在新实例恢复。若宿主边界可能失守，应轮换节点和相关凭据，检查同节点其他任务，并验证镜像、策略和控制面的完整性。

## 验证清单

自动化测试至少覆盖：容器内 root 尝试、挂载和 socket 探测、危险系统调用、资源炸弹、fork bomb、私网与元数据访问、DNS 重绑定、大文件与压缩炸弹、跨租户目录读取、过期令牌和实例销毁后的数据残留。故障注入还要验证检测链路、令牌撤销、节点隔离和从可信状态恢复。

上线门禁可以检查：

- 镜像是否固定摘要并通过漏洞与来源验证；
- 是否非 root、禁止提权、删除多余 capability；
- 根文件系统是否只读，临时卷和资源是否有上限；
- 默认网络拒绝是否实际生效，代理是否阻断私网和重定向绕过；
- 是否不存在宿主 socket、设备和长期凭据挂载；
- 审计是否能从告警追到 Run，并完成销毁、撤销和恢复演练。

## 小结

容器只是风险降低层，不是安全证明。可靠的 Agent 沙箱把不可信执行包在应用能力策略、容器限制、内核控制、隔离宿主和网络出口策略之中，并在外部建立审计、撤销与恢复能力。设计时始终问“这一层失守后下一层能否阻止扩散”，比追求一个神奇的 sandbox 开关更接近真实安全。
