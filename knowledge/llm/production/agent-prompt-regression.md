Agent 升级很少只改 Prompt：模型、工具描述与 Schema、权限策略、检索数据和执行代码都会变化。回归测试的第一原则是把变化轴分开，用同一套 Harness 比较基线和候选；否则结果变好或变坏时，团队只能猜是谁造成的。

[NIST AI TEVV](https://www.nist.gov/ai-test-evaluation-validation-and-verification-tevv)强调明确任务、测试床、数据集和测量方法，并理解不同情境下指标的局限。回归矩阵把这些条件版本化，使每次升级都能回答“在相同证据上发生了什么变化”。

图右侧的 Baseline 与 Candidate 进入 Same Harness，得到 Delta Report。若两侧的执行器、评分器、缓存或工具权限不同，差异报告就不是候选效应。

## 五个主要变化轴

### Prompt

包括系统指令、角色模板、示例、输出格式和上下文组装。Prompt 需要内容摘要、模板版本和渲染后摘要；只记录模板文件不足，因为变量、功能开关和语言分支会改变最终消息。

### 模型

记录供应商、不可变模型版本、参数、路由、区域和降级链。`latest`、`auto`或别名会在团队不提交代码时改变行为；回归中应解析为实际版本，并记录每个 Run 的路由结果。

### 工具

工具名、描述、Schema、认证、超时、错误语义和后置条件都可能变化。描述改变会影响模型选择，Schema 改变会影响参数，后端语义改变可能在接口不变时改变副作用。注册表使用摘要与兼容规则。

### 策略

权限、审批、内容过滤、数据分类、出口和预算策略是 Agent 行为的一部分。策略收紧可能提高拒绝率但降低风险；策略放宽可能提高任务完成却造成越权。两者必须在报告中独立观察。

### 数据

检索索引、文档版本、特征、记忆和用户状态会改变答案。回归使用数据快照或可复建版本；若必须读取实时数据，标注不可复现并设计对照。

## 建立版本清单

每个候选生成 manifest：

```json
{
  "agent": "agent-42",
  "code": "sha256:...",
  "prompt": "sha256:...",
  "model": "model-2026-07-01",
  "toolRegistry": "sha256:...",
  "policy": "policy-41",
  "dataSnapshot": "kb-2026-07-15-02",
  "harness": "eval-18",
  "scorers": ["task-state@5", "citation@7"]
}
```

基线也生成相同结构。比较器先检查除目标轴外是否一致；不一致则标记 confounded，不应自动给出“Prompt 提升 3%”的结论。

## 测试集分层

回归集至少包含：稳定基线任务、边界案例、安全案例和工具案例。稳定集检测普遍退化；边界集覆盖长上下文、模糊输入、取消与预算；安全集覆盖注入、越权和外泄；工具集覆盖 Schema、错误和副作用。

历史事故固定在保留回归集中，每条案例记录修复版本和风险。新功能有针对性新增样本，但不要只添加能展示候选优势的案例。开发集用于迭代，保留集限制访问，防止反复调参过拟合。

## 单轴与组合测试

排查时先单轴：只替换 Prompt、只替换模型或只替换工具注册表，其他条件固定。发布候选通常是组合包，还要运行完整组合测试，因为轴之间存在交互，例如新 Prompt 只在新模型上有效，新工具描述需要新 Schema。

[NIST 的实验设计选择指南](https://www.itl.nist.gov/div898/handbook/pri/section3/pri3.htm)列出随机、区组、因子与分部因子等设计。轴很多时，不必穷举所有组合；可以用风险优先的因子设计识别主效应和关键交互，再对最终组合完整验证。选择设计和分析方法应在看结果前确定。

## Same Harness 的具体含义

两侧使用同一任务顺序、夹具、工具替身、超时、预算、缓存策略、并发、重试、评分器和运行次数。若模型有速率限制，交错或随机运行 A/B，避免时间段和容量成为混杂因素。

基础设施错误单独统计。某侧因限流少跑 10% 样本时，不能只比较剩余成功结果；先修执行条件或按预定规则重跑。Harness 自身变更先做验证桥接，再用于候选比较。

## 断言不只看最终文本

报告至少包含：任务成功、状态后置条件、工具选择与参数、授权、引用支持、质量评分、延迟、成本、安全规则和失败切片。逐字匹配只适合固定格式；语义回答使用结构化和校准评分。

工具轨迹可声明允许集合与偏序，而非固定每一步。例如检索和用户资料可并行，但写工具必须在审批后；这样既容忍合理计划差异，又保护关键不变量。

## Delta Report

差异报告同时展示绝对值、变化、样本量和不确定性：

```text
metric                 baseline    candidate    delta
qualified_success      91.4%       92.0%        +0.6pp
p95_latency             8.1s        7.5s         -0.6s
cost_per_success        $0.082      $0.079       -$0.003
critical_violations     0/240       1/240        BLOCK
```

最后一行不能被前三行改善抵消。报告按任务、语言、风险、工具和长度切片，列出新增失败、修复失败与行为改变样本。

## Prompt 特有检查

渲染测试检查角色顺序、指令优先级、变量转义、长度预算和敏感信息；行为测试检查指令遵循、澄清、工具边界与输出格式；对抗测试检查不可信内容是否跨越数据/指令边界。

不要用“Prompt 中写了禁止”作为安全通过。工具授权、Schema、审批和输出验证应提供确定性保护。Prompt 回归负责模型行为，策略回归负责实际权力。

## 模型升级与语义约定

模型可能改变 token 计数、工具调用形状、流式事件、拒绝或默认参数。适配层先做协议契约，再做行为评估。2026-07-16 查看 [OpenTelemetry semantic conventions](https://opentelemetry.io/docs/specs/semconv/)时，GenAI 相关语义仍存在演进与版本选择；遥测字段升级也要纳入回归，否则大盘变化可能只是采集口径变化。

模型别名更新属于真实发布，即使仓库没有 diff。建立模型路由变更事件，自动触发核心回归和 Canary，避免供应商更新绕开质量门禁。

## 处理随机性

对概率性输出运行多次并报告分布。固定种子可帮助重放但不保证跨平台确定；更可靠的是断言结构、状态和安全不变量。候选若均值提高但方差显著增大，长尾体验可能变差。

重试不能只保留最后一次通过。记录每次尝试，区分基础设施重试与模型变异；安全关键用例一次违反即失败。

## 常见误区

- Prompt、模型、工具、策略同时升级，却把变化归功于其中一个。
- Baseline 和 Candidate 使用不同评分器或缓存。
- 只看平均质量，不检查新增失败和关键切片。
- 逐字断言所有回答，造成无意义 flaky；同时不检查状态与工具。
- 模型别名自动更新，却不视为需要回归的发布。
- 遥测 Schema 改变后误把指标跳变当行为退化。

## 小结

Prompt、模型、工具、策略和数据是独立又会交互的变化轴。回归测试用不可变 manifest、分层测试集和 Same Harness 先做单轴归因，再验证最终组合，报告质量、延迟、成本、安全和失败切片。凡是无法固定的条件都要显式标记；凡是关键安全失败都不能被平均改善抵消。
