离线集能稳定比较版本，却看不到真实用户分布、最新网页、长尾工具和外部依赖。在线评估从生产 Run 中按规则抽样，经隐私处理后进入自动与人工评审，生成质量信号、漂移告警、事故和反馈集。它的任务是“观察并形成证据”，不是让评审结果直接修改线上 Agent。

[OpenTelemetry 对 Trace、Metric 和 Log 等信号的说明](https://opentelemetry.io/docs/concepts/signals/)提供了连接聚合与单次 Run 的基础：指标发现趋势，Trace 还原路径，日志保存结构化事实。在线评估必须保留这些关联，否则一个质量低分无法定位到模型、工具、策略和数据版本。

![生产 Run 经隐私过滤、分层采样、自动与人工评审进入反馈集](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/agent-online-eval-sampling-monitoring-loop-v1.webp)

图中 Feedback Set 只进入 Offline Test Set，不直接回流生产。评审样本需要去重、标注、复核和版本控制，再通过正常发布门禁成为改进证据；自动把用户反馈写进 Prompt 或记忆，会形成投毒通道。

## 在线评估回答什么

它适合回答：当前版本在哪些任务、语言、工具和用户群体中退化；新型失败是否出现；自动评分器与人工判断是否偏离；安全拦截和人工接管是否增加；离线集遗漏了哪些真实场景。

它不适合单独证明版本变更造成业务提升。生产流量、季节、渠道和其他功能会同时变化；需要因果判断时使用随机实验。在线评估提供监测和诊断，A/B 测试提供对照下的效应估计。

## 第一道边界：隐私与用途

在抽样前定义允许收集的字段、用途、访问者、保留期、删除与导出流程。密钥和认证令牌永不进入评估；个人信息按 Schema 脱敏；Prompt 与输出默认用受控引用而不是复制到通用队列。

抽样器看到的是经过最小化的事件摘要：任务类型、风险等级、结果、错误类型、版本、成本、Trace 引用和内容分类。需要人工查看原文时，评审者临时获得证据仓库权限，并留下审计记录。这样评审系统不成为第二个不受控的数据湖。

## 三种采样组合

### 随机采样

随机样本用于估计总体趋势，采样概率必须记录。若低流量任务永远抽不到，可按任务层分层随机，再用权重还原总体。不能把过采样的失败样本直接当生产失败率。

### 风险采样

高权限写工具、敏感数据、未成年人、医疗/金融建议、首次出现的工具组合等获得更高概率。风险采样擅长发现严重问题，但不代表总体分布，应单独报告。

### 触发采样

用户点踩、人工接管、异常高成本、策略拒绝、超时、未知域名、记忆冲突和评估器分歧可触发必审或高优先级队列。触发条件与版本写入记录，便于解释某周样本量激增。

三种队列去重后合并，同时保留每个入选原因。一个 Run 可以同时被随机和风险规则选中，但评审一次即可，统计时根据目标选择正确权重。

## 自动评估层

确定性评估优先：Schema、后置状态、工具权限、引用链接、数据分类、成本和超时。模型评分器处理相关性、完整性和表达质量，但必须固定 rubric、评分 Prompt 和模型版本，并用人工集校准。

自动评分输出不只是一分：

```json
{
  "evaluator": "citation-support@7",
  "score": 0.42,
  "label": "unsupported",
  "confidence": 0.78,
  "reasons": ["claim_3_has_no_support"],
  "evidenceRefs": ["span://.../retrieval", "output://..."],
  "evaluatedAt": "2026-07-16T12:08:00Z"
}
```

低置信、高风险和评分器分歧进入人工队列。自动评估器变化时，旧样本可重放以建立可比桥梁，不能无提示地把新旧分数拼接。

## 人工评审层

人工 rubric 使用可观察标准和示例，避免“整体感觉 1–5 分”。评审者看到必要上下文、工具轨迹和引用证据，不看到候选版本名称等可能引入偏差的信息。高风险案例采用双人复核与仲裁。

[NIST AI TEVV](https://www.nist.gov/ai-test-evaluation-validation-and-verification-tevv)强调 AI 测量的情境性，以及准确性之外的鲁棒性、偏差、透明度等特征。评审抽样应覆盖不同任务与群体，并定期测评审者间一致性；一致性下降可能是 rubric 模糊、任务变化或培训不足。

## 从样本形成质量信号

报告时分开三类数：带采样权重的总体估计、风险队列发现率、触发队列处置量。每类按 Agent 版本、任务、语言、工具、风险和数据等级切片，并提供不确定性。样本很少时显示范围或“证据不足”，不要展示看似精确的小数。

质量信号与可靠性、延迟、成本并排观察，但避免一个综合分数抵消严重安全事件。安全越权、数据泄露和不可逆副作用是独立门禁与事故条件。

## 漂移检测

至少监控三种漂移：输入分布变化、行为/工具轨迹变化、结果质量变化。输入变化可能来自新渠道或热点事件；工具变化可能来自模型路由和 Schema；质量变化可能滞后出现。

漂移告警包含基线窗口、当前窗口、样本量、主要切片和代表 Trace。检测到差异先扩大抽样并验证采集/评分器版本，避免把数据管道改动误报为模型退化。

## 事故与升级

命中关键规则时创建事故，而不是等待周报：跨租户访问、敏感数据外泄、未授权写操作、重复支付、沙箱逃逸迹象等。事故记录保留证据引用、影响范围、控制决策和响应人；评审者不在普通队列中自行尝试复现危险动作。

[NIST AI RMF Generative AI Profile](https://doi.org/10.6028/NIST.AI.600-1)把测量、管理与治理放在持续风险管理中。在线评估因此需要明确责任闭环：谁确认信号、谁决定暂停/回滚、谁把事故转成回归、谁批准重新发布。

## 反馈集治理

入选反馈集前执行：去重、脱敏、许可确认、失败分类、期望结果、责任人、复审日期和难度/风险标签。用户点踩只是候选信号，不自动等于答案错误；恶意反馈也不能直接成为记忆或系统指令。

反馈集按快照发布，开发集与保留集分离。已用于调 Prompt 的样本不再冒充独立测试。案例因法规、产品或工具变化过期时标记失效，保留历史但不继续影响当前门禁。

## 常见误区

- 只评审用户点踩，样本严重偏向不满用户，却报告为总体质量。
- 抽样前保存完整原文，隐私过滤成为事后补丁。
- 模型评分器升级后直接比较新旧曲线，没有重放桥接。
- 把评审内容自动写入长期记忆或生产 Prompt，引入投毒。
- 只展示平均分，不显示样本量、不确定性和失败切片。
- 发现关键安全问题仍等待周报，而不是进入事故流程。

## 小结

在线评估由隐私过滤、随机/风险/触发采样、自动评分、人工复核、质量信号、漂移与事故组成。所有样本通过 Run/Trace 关联到版本与行为，反馈只进入受治理的离线集，不直接改变生产。它让真实世界问题变成可审计证据，同时保持采样偏差、隐私和因果边界清晰。
