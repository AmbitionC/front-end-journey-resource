间接 Prompt Injection 不是用户在聊天框里说“忽略规则”，而是恶意指令藏在 Agent 主动读取的网页、文件、邮件、工单或检索文档中。模型很擅长理解文字，却未必天然知道哪些文字只是数据、哪些拥有指令权。只要不可信内容能够影响计划并调用工具，攻击就可能从阅读升级为副作用。

[OWASP 2025 LLM 风险列表](https://genai.owasp.org/llm-top-10/?cat=253)把 Prompt Injection 列为核心风险，并指出外部内容可以改变模型行为。对 Agent 而言，风险因为工具、记忆和持续状态被放大；“模型没有输出危险文本”不代表它没有在后台执行危险动作。

![网页、文件与邮件中的不可信内容被约束在数据通道并经过策略校验](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/agent-indirect-injection-trust-boundaries-v1.webp)

图中红色 `IGNORE RULES` 永远留在 Data Channel。它可以被摘要、引用和显示，却不能因为语气像命令就进入 Instruction Channel。真正的系统指令来自受信策略，并在工具前经过来源、清洗、策略、审批和输出验证。

## 攻击是怎样形成的

典型链路包含四步：

1. 攻击者把指令放进 Agent 将读取的内容；
2. 内容被检索、浏览或解析并拼入模型上下文；
3. 模型把内容误当高优先级任务，改变计划或生成工具参数；
4. 过大的工具权限、缺少审批或结果验证，使计划成为真实副作用。

攻击成功不是模型“相信一句话”这么简单，而是数据跨越了信任边界，又遇到可利用能力。因此防御也不能只靠另一个 Prompt 说“不要相信网页”。

## 标记来源与信任

每段外部内容携带 provenance：来源类型、规范化 URL/对象、抓取时间、租户、签名或哈希、解析器版本和信任等级。上下文组装器用结构化信封包裹：

```json
{
  "content": "...untrusted document text...",
  "provenance": {
    "source": "web",
    "origin": "https://example.invalid/page",
    "trust": "untrusted",
    "fetchedAt": "2026-07-16T12:00:00Z"
  },
  "allowedUse": ["summarize", "quote"],
  "instructionAuthority": false
}
```

标签必须被运行时策略使用。只在 Prompt 中加“UNTRUSTED”字样，但工具层仍无条件执行模型参数，只是视觉提醒，不是边界。

## 分离数据通道与指令通道

系统、开发者策略与用户明确请求各有授权来源；检索文档、网页和工具返回值是数据。解析时不要把二者拼成一段无边界文本。使用结构化消息、明确 delimiter、引用对象和只读渲染，让模型更容易区分，同时让策略引擎能检查来源。

即使模型仍被诱导，工具调用也要携带决策依据：请求来自哪个用户目标，使用了哪些来源，是否从不可信文本提取参数。高风险动作若主要依据不可信内容，应要求重新确认或人工审批。

## 最小能力与上下文能力

阅读网页的 Agent 不必同时拥有发邮件、读取密钥和写数据库。按任务授予最小工具集合、最小 scope、短期凭证和目标资源。浏览器阶段与写操作阶段可拆成不同 worker，通过结构化结果交接，而不是一个全能 Agent 持有所有能力。

[OWASP Agentic Threats Navigator](https://genai.owasp.org/resource/owasp-gen-ai-security-project-agentic-threats-navigator/)把推理、记忆、工具、身份与人类监督都视为攻击面。最小能力降低的是“模型被影响后能造成什么”，是 Prompt 防御之外最关键的纵深控制。

## 工具调用前的确定性校验

模型提出动作，策略决定能否执行。校验至少包含：主体与 actor、scope、目标资源、数据分类、参数 Schema、业务不变量、目的地、预算和审批。写工具使用允许列表，未知工具和未知字段默认拒绝。

例如网页建议“把报告发到 auditor@example.invalid”，运行时不能因为地址格式合法就发送。策略要确认用户是否明确请求外发、域名是否允许、报告分类是否可出境、收件人是否被授权，并显示清晰审批摘要。

## 人工审批要展示真实动作

审批界面显示工具名、目标、参数、数据分类、来源和不可逆影响，不显示模型生成的模糊解释。攻击者可能在文档里写“这是例行安全检查”，所以审批理由也应标记为不可信来源。

批准绑定动作摘要与短时有效期。参数改变后旧批准失效；拒绝不会被 Agent 通过改写同一动作无限重试。批量动作显示数量和抽样明细，避免把一千个收件人藏在“发送通知”后面。

## 输出与后置条件

工具返回值同样不可信，可能继续包含注入。先验证 Schema、来源、长度和内容类型，再以数据通道交给模型。执行后检查后置条件：发送对象、状态变化、行数、金额是否符合批准；不符合则停止、补偿并告警。

最终输出做数据泄露和引用验证。即使工具调用被阻断，模型也可能把上下文中的秘密直接写进回答；输出 DLP 与数据分类是独立边界。

## 记忆与跨会话风险

不可信内容不能自动写入长期记忆。候选记忆携带来源、租户、置信、到期和允许用途，经过冲突检查与审核。否则一次网页注入可能变成永久偏好，后续在完全不同任务中继续影响 Agent。

会话摘要也要保留“这是网页声称的内容”，不能压缩成“系统规则要求……”。摘要是有损转换，若丢失 provenance，就会把不可信数据洗成看似可信事实。

## 检测与测试

记录不可信内容参与计划、策略拒绝、审批、工具序列、未知域名和敏感数据命中。检测关注行为而非关键词：攻击文本可以变化，但“读取网页后立刻请求密钥并外发未知域名”是稳定信号。

测试语料覆盖 HTML 隐藏文本、PDF 注释、邮件签名、代码块、图片 OCR、检索文档、多语言和编码变体。断言工具和状态不变量，而非模型必须说出固定拒绝句。

[NIST AI RMF Generative AI Profile](https://doi.org/10.6028/NIST.AI.600-1)强调在生命周期内进行评估、红队和风险管理。间接注入测试应进入持续回归；新增浏览器、文件解析器、数据源或写工具都应触发对应威胁模型复审。

## 常见误区

- 只在系统 Prompt 写“忽略网页指令”，工具层没有授权校验。
- 浏览器 Agent 同时持有密钥、邮件和数据库写权限。
- 审批界面展示攻击者提供的理由，却隐藏真实目标与数据。
- 工具返回值被当成可信指令，再次触发调用。
- 网页摘要写入长期记忆时丢失来源与不可信标签。
- 只用关键词过滤 `ignore previous`，忽略多语言和行为链。

## 小结

间接 Prompt Injection 的根因是“不可信数据获得了指令权，并能调用过大能力”。防御从 provenance 和数据/指令分离开始，以最小工具、确定性授权、清晰审批、结果与后置验证、记忆隔离和出口控制收口。模型可以被影响，但系统不应因此自动把文字变成权力。
