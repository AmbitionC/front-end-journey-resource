Agent 事故既可能是传统可用性故障，也可能是“服务正常但行为危险”：模型向错误对象发信、工具权限过大、检索内容被污染、敏感数据进入日志，或某个发布只在特定语言下突破安全门禁。响应流程必须同时保护用户、保存证据和控制外部副作用，而不是先花几个小时争论根因。

[NIST SP 800-61 Rev.3](https://csrc.nist.gov/pubs/sp/800/61/r3/final)把事故响应放进更广的风险管理周期，连接准备、检测、响应、恢复和持续改进。这个框架适合 Agent：事故处置不是孤立值班动作，还依赖平时的资产、身份、日志、备份、演练和供应链治理。

## 预先定义严重级别

严重级别由用户影响、数据与权限、不可逆副作用、扩散速度和恢复难度决定，不只看受影响请求数。例如单次高权限越权可能比大面积可重试 5xx 更严重。

建议至少区分：

- SEV-1：广泛或不可逆伤害、敏感数据泄露、控制面或高权限失守；
- SEV-2：显著用户影响、持续质量/安全退化、关键功能不可用；
- SEV-3：局部、可控且有替代路径的退化；
- SEV-4：无用户影响的风险发现或近失事件。

每级对应响应时间、事故指挥、沟通频率和管理升级。严重安全事件可以根据“潜在最大影响”先高定级，再随证据调整。

## 角色与单一指挥

[Google SRE 的 Managing Incidents](https://sre.google/sre-book/managing-incidents/)强调清晰角色和协调。常见角色包括 Incident Commander、Operations Lead、Communications Lead 和 Planning/Recorder。一个人可以兼任，但决策权必须明确。

指挥官维护目标和优先级，不亲自深挖每条日志；运维负责人执行缓解；沟通负责人给用户和内部利益相关者一致更新；记录者维护时间线、假设、动作、负责人和证据。未经指挥确认的并行实验容易互相覆盖现场。

## 检测与快速分诊

告警或投诉进入后先回答：

1. 哪些用户、租户、地区、任务和时间段受影响；
2. 是否仍在发生，是否存在不可逆或扩散中的副作用；
3. 最近有哪些 release、Prompt、模型、工具、策略和索引变化；
4. 哪些身份和数据可能暴露；
5. 是否能用低风险开关快速缩小影响面；
6. 当前证据是否足以避免破坏现场。

以 runId、releaseId、traceId、toolOperationId 和 policyVersion 抽取代表样本。不要把模型内部自然语言解释当作根因证据；使用工具请求、状态转移、授权决定和外部系统确认等事实。

## 先止损

缓解按可逆、影响面小和速度快排序：暂停特定工具或高风险动作、把 Agent 切只读、关闭故障 bundle、限制某地区或租户、撤销令牌、隔离 Worker、冻结队列消费。若任务可能继续发起副作用，先阻止新执行，再处理排队与在途任务。

回滚不总是安全。新版本写入了旧版本无法读取的状态时，直接切回会制造二次事故；需要兼容适配或 roll-forward。模型供应商故障可启用已验证 Fallback，但不能绕过地区、安全和质量门槛。

对于数据泄露或越权，撤销相关凭据、阻断出口、保存访问记录并启动法务与隐私流程。不要为了“看起来恢复”删除日志或重启全部实例，导致证据丢失。

## 证据与时间线

建立只读事件包：告警、用户报告、配置与发布摘要、关键 Trace、审计事件、模型/Prompt/工具版本、输入输出内容的受控引用、外部副作用确认和操作员动作。证据原文按最小权限访问，导出行为也被审计。

时间线同时记录事实和假设，二者明确标记。每次缓解动作写执行者、时间、预期、实际结果和回退方式。若系统时钟不完全一致，结合 observedTime、消息序号和因果 ID，而不是强行按日志到达顺序排列。

## 恢复与验证

恢复不是大盘变绿。先定义验证条件：新危险动作停止、在途任务已隔离、队列没有继续扩大、外部系统对账完成、受影响用户得到处理、数据或凭据风险已控制。小流量恢复后逐级开放，并持续比较控制组。

对 unknown 副作用逐项查询 operationId，已提交则记录结果，未执行才安全重试，无法确认则人工处理。恢复的任务从可信检查点和新 executionEpoch 开始，旧 Worker 不得继续写入。

## 沟通

首次更新尽早说明已知事实、用户影响、正在做什么和下次更新时间，避免给未经验证的根因。后续保持固定节奏，即使只有“仍在调查”。对外语言描述影响和措施，不泄露会增加攻击风险的细节。

事故结束时明确恢复时间、剩余风险、用户需要采取的动作和后续报告计划。内部渠道避免多人发布互相矛盾的信息，所有关键决定回写时间线。

## 复盘与纠正

[Google SRE 的 Postmortem Culture](https://sre.google/sre-book/postmortem-culture/)强调无责复盘：重点是系统为何允许合理的人类行动造成影响，而不是找到一个“犯错的人”。复盘包含影响、检测、时间线、根因与促成因素、哪些做得好/不好，以及有负责人和期限的纠正项。

纠正项优先改变系统：增加权限门禁、Schema 验证、Canary、回滚兼容、审计关联或演练。只写“以后更小心”无法验证。每项说明降低哪类风险、如何验收、是否可能引入新风险。

## 演练

桌面推演与故障注入覆盖：错误 Prompt 发布、模型安全回退、工具越权、检索污染、审计不可用、队列重复副作用、密钥泄露和跨区域故障。演练验证值班表、权限、开关、证据导出、用户沟通和恢复对账，而不仅是技术脚本能运行。

## 小结

Agent 事故响应以用户伤害为中心：快速分诊、单一指挥、可逆止损、保全证据、谨慎恢复和无责改进。模型行为增加了质量与安全维度，但操作原则没有改变——先控制影响，用事实而非猜测推进，并把复盘结果变成可验证的系统改动。
