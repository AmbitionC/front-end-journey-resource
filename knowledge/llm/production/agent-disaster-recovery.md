Agent 灾难恢复不只是给数据库做备份。一次 Run 还依赖事件历史、检查点、Prompt 与模型版本、工具契约、对象、密钥和外部副作用。只恢复主表却丢失加密密钥、索引版本或幂等记录，任务可能无法继续；更糟的是，重放时再次发送邮件、扣款或发布内容。

[NIST SP 800-34 Rev.1](https://csrc.nist.gov/pubs/sp/800/34/r1/final)把应急规划与业务影响分析、恢复策略、计划、测试和维护连接起来。RTO 与 RPO 应从业务影响反推，而不是由备份产品默认值决定。

## RTO、RPO 与恢复层级

RTO 是业务允许的恢复时间，RPO 是允许丢失的数据时间窗口。控制面授权、运行中高价值任务和审计日志可能要求更短 RPO；可重新生成的嵌入、临时缓存和渲染产物可以有更宽窗口。所有数据“一律零丢失”通常昂贵且不现实。

按服务层级定义：

- Tier 0：身份、密钥、策略和任务状态，优先恢复；
- Tier 1：检查点、事件、幂等账本和关键对象；
- Tier 2：检索索引、模型缓存与可重建派生数据；
- Tier 3：调试日志、临时文件和无业务价值缓存。

依赖图决定恢复顺序。状态库恢复了但密钥系统不可用，数据仍不可读；工具网关先开但授权策略未恢复，会制造更大风险。

## 数据分类：事实与派生物

持久事实包括用户输入引用、审批、状态转移、外部 operationId、工具确认结果、策略版本和最终承诺。派生物包括可重新计算的摘要、嵌入、排序分数、预览和缓存。事实优先强一致与高频备份，派生物保存生成配方与源版本即可。

检查点是执行边界的快照，不是任意内存 dump。它包含 thread、run、step、stateVersion、计划、已完成动作、待执行动作和引用的对象摘要。事件日志提供变化历史；二者配合可以恢复当前状态并调查来源。

[LangGraph Persistence](https://docs.langchain.com/oss/python/langgraph/persistence)以 thread 和 checkpoint 保存图执行状态，展示了 Agent 运行时如何在步骤边界恢复。这里的具体 API 属于框架实现，但“版本化检查点用于继续、检查与容错”的思想可迁移。

## 备份范围

备份清单至少覆盖：

- 任务、线程、检查点、事件、幂等账本和 outbox；
- Prompt、工具、策略、工作流和模型 bundle 的不可变版本；
- 对象存储及其元数据、校验和与生命周期规则；
- 数据库 Schema、迁移和恢复顺序；
- 加密密钥或可恢复密钥材料，以及密钥版本映射；
- 身份和授权配置、网络与基础设施定义；
- 审计完整性根、监控规则和运行手册。

跨区域复制不是备份：误删除、勒索或坏迁移可能立即复制过去。保留不可变、隔离权限的历史版本，并对备份账号使用独立身份和多方审批。

## 一致性与恢复点

多个存储无法简单在同一时刻冻结时，使用一致性标记：数据库快照带 log position，对象引用使用不可变版本，事件带序号，索引记录 sourceVersion。恢复时选择一个可证明闭合的恢复点，晚于该点的孤立对象或事件进入对账。

备份作业成功只表示复制过程没有报错。每批记录对象数、摘要、加密密钥版本和依赖清单，独立验证可读性。Schema 与应用版本必须兼容；灾难时临时下载“最新版应用”可能无法读取旧快照。

## 安全重放

从检查点继续前，先建立恢复 fence，阻止旧区域或旧 Worker 继续写入。为恢复产生新的 executionEpoch，所有状态更新带 epoch 与 fencing token。否则网络分区恢复后两边会同时认为自己是主系统。

重放只重复可重构决策，不重复已确认副作用。[RabbitMQ Reliability Guide](https://www.rabbitmq.com/docs/reliability)说明不确定故障会带来重投，因此 Worker 和恢复器都要按 operationId 查询幂等账本。已 committed 则读取结果，not_started 才执行，unknown 进入对账或人工。

模型输出可能非确定，重放时应优先使用已保存的已承诺结果；确需重新调用时记录新 attempt 和模型版本，不能假装与原输出完全相同。安全审批、策略撤销和用户取消以恢复时当前政策再校验。

## Restore Drill

演练在隔离账户或区域中从备份启动完整最小系统：

1. 恢复密钥和身份根；
2. 恢复数据库与对象并校验摘要；
3. 应用匹配的 Schema 和应用版本；
4. 验证审计链、任务计数和引用完整性；
5. 选取代表 Run 做只读 replay；
6. 对模拟外部服务验证幂等、取消和补偿；
7. 测量实际 RTO/RPO，记录人工步骤与阻塞；
8. 销毁演练环境并审计访问。

演练必须包含坏备份、缺密钥、部分区域和第三方不可用，而不只验证理想路径。每项纠正动作有负责人和期限，计划随架构变更更新。

## 灾难切换与回切

宣布灾难前定义权限和证据门槛。切换时冻结旧写入、推进 DNS 或路由、逐步开放只读与写流量，并持续对账。用户可见状态说明可能的数据窗口和正在恢复的功能。

回切同样是迁移，不是把开关拨回去。先同步恢复期间的新事实，验证幂等账本和事件序号，再以 Canary 切回；保留恢复区域直到对账完成。双写期间明确唯一权威，避免脑裂。

## 小结

Agent 灾备把业务影响、RTO/RPO、数据分类、完整备份、检查点、幂等重放和恢复演练串成一条链。真正的目标不是“有备份文件”，而是在旧系统可能失效、外部副作用可能已提交的情况下，仍能在目标时间内恢复到可验证且不重复行动的状态。
