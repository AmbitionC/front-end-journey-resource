Agent 测试的难点在于：同一个“答案错误”，可能来自纯函数、Prompt、模型波动、工具契约、权限、数据状态或真实网络。若所有问题都靠端到端测试发现，反馈慢、成本高、定位困难；若只做单元测试，又无法覆盖模型与外部世界。测试金字塔的作用，是把断言放到最便宜、最稳定、又足以证伪目标契约的层级。

[NIST 的 AI TEVV 说明](https://www.nist.gov/ai-test-evaluation-validation-and-verification-tevv)区分并研究测试、评估、验证和确认，强调任务、测试床、数据集、测量方法与使用情境。Agent 的金字塔不是某个标准规定的固定形状，而是把这些活动按隔离性和现实性组织起来的工程方法。

![Agent 单元契约、工具集成、场景与端到端测试金字塔](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/agent-test-pyramid-layered-strategy-v1.webp)

越往上，环境越真实、成本越高、失败归因越难；越往下，执行越快、隔离越强、覆盖越广。健康的策略不是“尽量少做上层”，而是让每层只承担它能回答的问题。

## 第一层：单元与契约测试

底层测试不需要真实模型或网络，覆盖确定性逻辑：

- Prompt 构造器是否按角色和优先级组装消息；
- JSON/Schema 解析、默认值和错误路径是否正确；
- 工具参数验证与业务不变量是否阻断非法输入；
- 权限策略是否根据 subject、actor、scope、resource 做出预期决定；
- 状态机转移、取消、超时预算、重试与幂等键是否正确；
- 日志脱敏、数据分类和保留期是否执行。

这些测试应毫秒级、可重复、无外部副作用。模型输出可以使用手写夹具，工具使用内存实现。关键是测试“我们控制的代码”，不要把固定字符串模拟成一个过于聪明的模型，再误以为覆盖了真实行为。

示例：工具调用验证器的契约测试可以覆盖未知字段、越界金额、缺失幂等键和无权限资源：

```ts
it('rejects a write without delegated scope', () => {
  const call = { tool: 'invoice.update', args: { id: 'i-42', status: 'paid' } };
  const auth = { subject: 'u-1', actor: 'agent-7', scopes: ['invoice:read'] };
  expect(authorize(call, auth)).toEqual({ allowed: false, reason: 'missing_scope' });
});
```

## 第二层：工具集成测试

这一层验证 Agent 与真实协议或高保真替身之间的契约：HTTP 状态、认证、分页、速率限制、Schema 版本、超时、重试、幂等和错误映射。可以运行本地容器、测试数据库、录制回放服务或官方沙箱，但不要碰生产数据。

集成测试应主动注入故障：连接超时、429、部分成功、响应缺字段、旧 Schema、重复消息、延迟到达与取消竞态。只测“正常返回 200”无法证明恢复路径。写工具还要断言副作用次数和最终状态，而不只看客户端收到的文本。

工具模拟器必须有明确边界。Mock 适合验证调用形状，不能证明供应商真的按同一规则工作；因此还需要少量针对真实测试环境的契约探针，并对响应 Schema 做版本监控。

## 第三层：场景测试

场景测试运行完整 Agent 循环，但在受控沙箱中组合模型、工具、数据和多轮状态。每个场景描述：用户目标、初始世界状态、允许能力、注入事件、停止条件和可观察结果。

```yaml
scenario: cancel_after_tool_started
initial_state: {order: pending}
user_goal: cancel order
injections:
  - at: tool.order.cancel.started
    event: user_disconnect
stop_when: run_terminal
assert:
  - order.status == cancelled
  - notification.sent_count <= 1
  - run.outcome in [cancelled, completed]
  - no_unauthorized_tool_call
```

场景断言应优先针对状态、工具轨迹和安全不变量，而不是逐字匹配输出。模型有随机性，可以对同一场景运行多次，统计通过率与失败切片。对高风险规则，例如“不得跨租户读取”，应要求每次都通过，不能用平均分掩盖一次越权。

用户模拟器与环境模拟器可以增加覆盖，但它们本身也是模型，需要验证其策略和停止条件。模拟用户不断迁就 Agent，会高估成功率；环境过于干净，会漏掉真实依赖的延迟和部分失败。

## 第四层：端到端测试

端到端测试覆盖从入口、身份、队列、Agent、模型、工具到持久化和通知的完整路径。它回答“系统能否在接近真实的环境中协同工作”，适合少量关键旅程、发布烟测和事故回归。运行时证据仍应按职责拆分；[OpenTelemetry 对 Trace、Metric 与 Log 等信号的说明](https://opentelemetry.io/docs/concepts/signals/)可用来组织路径、测量和事件，避免测试失败后只剩一段无法关联的控制台输出。

这类测试昂贵且脆弱，需要严格控制数量。每个失败先区分产品缺陷、环境缺陷、数据夹具失效和第三方波动；若一个端到端用例失败后只能“再跑一次”，它的诊断设计还不合格。

端到端环境也必须安全：测试租户、测试凭证、沙箱支付、邮件捕获器、默认拒绝网络出口和清理脚本。测试完成后验证没有残留状态，不要让今天的订单影响明天的断言。

## 横向能力：评估而非传统断言

Agent 输出常没有唯一正确字符串，需要确定性规则、模型评分器和人工评审组合。[NIST AI RMF 的生成式 AI Profile](https://www.nist.gov/publications/artificial-intelligence-risk-management-framework-generative-artificial-intelligence)强调在整个生命周期管理生成式 AI 风险，并把测量、评估、红队与反馈放在情境中考虑。

评分器也要测试：用人工标注集测一致性、对任务类型切片、识别位置偏差和长度偏好、固定评分 Prompt 与模型版本。模型评分器适合提供信号，不应独自批准高风险发布。可确定的安全和数据规则应放在代码策略层，而不是交给另一个模型“感觉一下”。

## 测试数据与夹具

测试集至少包含：典型任务、边界值、历史事故、长尾工具错误、对抗输入、不同语言、无障碍需求和高风险数据分类。每条案例记录来源、授权、版本、预期、责任人和复审日期。

生产样本进入测试集前必须脱敏并确认用途。直接复制用户对话会扩大隐私和保留风险；更好的方式是抽取结构、合成等价案例或在受控证据仓库中引用。测试数据与训练数据、Prompt 开发数据要尽量隔离，避免“看过答案”的污染。

## 如何分配一条测试

可以用四个问题决定层级：

1. 失败能否在不调用模型/网络时被确定性证伪？能则放单元层。
2. 问题是否只涉及一个外部工具契约？是则放集成层。
3. 是否需要完整 Agent 循环和世界状态，但不需要真实生产基础设施？放场景层。
4. 是否只有跨组件部署、身份、队列或真实协议才能验证？放端到端层。

事故回归常需要多层：底层加入最小确定性断言，场景层重现完整决策，端到端层保留一条关键烟测。只在最高层加一条慢用例，会让同类错误的反馈仍然很晚。

## 通过标准与不稳定性

每层要有不同通过标准：单元/契约测试应接近 100% 确定通过；场景评估可以报告多次运行分布，但安全不变量是零容忍；端到端测试允许基础设施重试，但必须记录首次失败，不能用无限重跑洗掉问题。

对 flaky 测试建立预算和责任人。先判断不稳定来自模型变异、共享状态、时钟、随机数据、网络还是断言过度精确，再修根因。把用例永久隔离等于删除保护，必须有到期时间和替代门禁。

## 覆盖率怎么看

代码行覆盖率只适用于确定性组件，不代表 Agent 行为覆盖。还应追踪：工具及错误码覆盖、状态转移覆盖、策略规则覆盖、任务类型覆盖、风险/数据等级覆盖、模型/Prompt/工具版本组合，以及历史事故是否有回归案例。

覆盖矩阵用于发现空白，不应追求每个维度的笛卡尔积。采用风险优先的组合测试：高权限写工具、敏感数据、异步恢复和新模型版本获得更多组合，低风险只读问答可以抽样。

## 常见误区

- 把所有测试都写成端到端聊天脚本，慢且难定位。
- Mock 永远返回理想数据，从不测试超时、部分失败和重复副作用。
- 逐字匹配模型回答，导致无意义波动；同时又不检查工具和状态不变量。
- 只看平均质量分，让单次严重安全失败被其他高分抵消。
- 生产样本未经脱敏直接进入测试仓库。
- 用无限自动重跑掩盖 flaky 测试，最终门禁失去可信度。

## 小结

Agent 测试金字塔从单元契约、工具集成、受控场景到端到端逐步增加现实性和成本。把断言放到最低但足够的层级，用场景覆盖模型与世界交互，用少量端到端验证部署协同，并用评估与人工证据处理非唯一答案。金字塔的目标不是层数漂亮，而是让失败尽早出现、容易定位、可安全重放。
