Agent 指标不是把“成功率、延迟、token”放进同一张大盘就结束了。真正有用的指标体系要把五层问题分开：业务是否得到价值、任务是否完成且正确、系统是否可靠、用户等待多久、为了结果付出多少资源。不同层之间有关联，但不能互相替代。

[OpenTelemetry 将 Metrics 定义为运行时捕获的测量值](https://opentelemetry.io/docs/concepts/signals/)，而 Trace 与 Log 分别保留路径和事件证据。指标适合发现趋势和触发告警，不能独自解释某次失败；调查仍要能从聚合桶下钻到被采样的 Run 与 Trace。

![Agent 从业务结果到成本的五层指标栈](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/agent-metrics-metric-stack-v1.webp)

图中的漏斗提醒我们：流量并不等于完成，完成不等于成功，成功也不一定形成持续业务价值。若只优化底层调用指标，可能得到一个“响应很快、token 很省、但没人愿意继续用”的系统。

## 先写指标定义，再写查询

每个指标都应该有一张定义卡，而不是只有名称：

```yaml
name: task_success_rate
question: 有资格进入统计的任务中，有多少满足成功判定？
numerator: passed_task_count
denominator: eligible_completed_task_count
window: rolling_24h
dimensions: [agent_version, task_type, channel, risk_tier]
exclusions: [load_test, user_cancel_before_execution]
owner: agent-quality
decision: block_release_if_below_threshold
```

分子、分母、时间窗、维度、排除规则和责任人缺一不可。“成功率 92%”若没有资格条件，就可能把取消、超时、人工接管或无标准答案的请求混在一起。指标定义要版本化；定义改变时，应避免把新旧序列无提示地拼成一条趋势线。

[OpenTelemetry 语义约定](https://opentelemetry.io/docs/specs/semconv/)强调统一的名称、属性、单位和含义。对 Agent 来说，这意味着延迟统一使用秒或毫秒、成本明确币种和计价口径、token 区分输入/输出/缓存，状态字段避免同一个值在不同工具里代表不同结果。

## 第一层：业务结果

业务指标回答“为什么要运行这个 Agent”。客服场景可以关注一次解决率、转人工率、复访率和满意度；研发 Agent 可以关注被合并的变更、回滚率、从需求到交付的周期；运营 Agent 可以关注正确完成的流程数与节省的人时。

业务结果通常有滞后性，也受渠道、季节、用户结构和其他产品改动影响。不要把一次相关性上涨直接解释成 Agent 的因果效果。需要判断版本是否带来业务变化时，应进入随机实验或至少使用明确的对照设计，而不是只看上线前后两条曲线。

## 第二层：任务质量

质量指标要围绕可观察的任务契约，而非笼统的“回答不错”。常见维度包括：意图是否解决、事实是否有证据、引用是否支持附近论断、结构化输出是否符合 Schema、工具选择与参数是否正确、最终状态是否满足后置条件。

质量评估可以由确定性规则、模型评分器和人工评审组合。确定性规则适合 Schema、引用存在性、禁止动作和状态断言；模型评分器适合语义相关性，但要用人工集校准；高风险结论必须有人类规则或领域专家复核。2026-07-16 的 NIST TEVV 页面强调，AI 的测量与评估需要任务、挑战问题、测试床、数据集及对方法局限的说明，而不是一个万能分数。[NIST 的 AI TEVV 说明](https://www.nist.gov/ai-test-evaluation-validation-and-verification-tevv)也明确指出，准确性之外还要考虑鲁棒性、偏差、可解释性和透明度等特征。

## 第三层：可靠性

可靠性描述系统按约定完成工作的稳定程度。至少区分：Run 完成率、技术错误率、工具失败率、超时率、重试率、重复副作用率、人工接管率和降级率。一个最终成功但经历五次重试的 Run，不应和一次成功的 Run 完全等价；它暴露了容量、依赖或策略问题。

分母要匹配事件。例如工具错误率的分母应是有资格的工具尝试数，而任务完成率的分母是被接受的任务数。对异步任务，还要定义观察窗口：窗口尚未结束的 Run 是“进行中”，不是失败。

## 第四层：延迟

只看平均延迟会隐藏长尾。至少报告 p50、p90、p95 或 p99，并拆出首个可见输出时间、最终完成时间、模型时间、工具时间、队列等待和人工审批等待。流式交互里，TTFT 很好但最终完成很慢，用户体验仍可能很差；后台任务则可能更关心截止时间内完成率。

延迟还要按任务类型、模型版本、地区、工具和结果状态切片。高基数用户 ID 不适合作为指标标签，可在 Trace 中保留受控关联信息。时间预算应从 Run 向下分配，避免每个子调用都拥有完整的 30 秒超时，最终串行累加成数分钟。

## 第五层：成本

成本不只是模型 token。还包括检索、重排、向量数据库、外部 API、浏览器/计算机执行、沙箱、日志与 Trace 存储、人工评审和失败重试。建议同时记录：每 Run 成本、每成功任务成本、每业务结果成本、预算超限率和无效工作比例。

“每 Run 成本下降”可能来自提前失败，而不是效率提升；因此成本必须和质量、可靠性一起看。一个有用的目标是单位合格结果成本：

```text
cost_per_qualified_success = total_eligible_cost / qualified_success_count
```

当分母过小或为零时，应返回无定义并告警，而不是显示 0。计价表、币种和折扣会变化，必须记录价格版本和估算/实付口径。

## 指标之间如何联动

五层指标可以形成“症状到证据”的排查路径：业务结果下降，先检查任务构成是否变化；质量下降，查看失败切片；可靠性异常，定位工具或模型依赖；延迟上升，进入 Trace 瀑布；成本上涨，拆分 token、重试和外部工具。不要用一个综合分数遮盖这种结构。

发布门禁也应采用多条件：质量不得低于基线，安全关键用例必须全过，p95 延迟不得超过预算，单位合格结果成本在范围内。若把这些指标加权成单一分数，一个严重安全失败可能被低延迟“抵消”，这是不可接受的。

## 维度与基数治理

维度用于切片，但每多一个维度，时间序列数量可能相乘。安全的低基数维度包括环境、区域、任务类型、模型家族、Agent 版本、工具名、结果分类和风险等级。原始 Prompt、错误消息、完整 URL、文档 ID 和用户 ID 通常不应成为指标标签。

维度集合也需要契约和预算。新增标签前要估算基数、存储和查询成本；未知值使用明确的 `unknown`，不要静默丢样本。对敏感维度，应使用最小化、分桶、访问控制和保留期。

## 告警与 SLO 思维

告警应对应可采取的行动，而不是每次波动都通知。可以为关键用户路径定义成功率与延迟目标，再用多个时间窗口监测错误预算消耗。质量和安全指标往往需要采样与人工确认，可设置“疑似异常 → 扩大抽样 → 触发事件”的分级流程。

每条告警都应包含：受影响指标定义版本、观察窗口、基线、主要切片、相关 Run/Trace 示例、值班手册和责任团队。没有下钻证据的红色数字只会制造告警疲劳。

## 常见误区

- 用调用 HTTP 200 作为任务成功率，忽略答案与副作用是否正确。
- 只报告平均值，隐藏长尾延迟与少数高成本 Run。
- 在指标标签中放 Prompt、用户或错误全文，造成基数爆炸和数据泄露。
- 每次版本升级顺便修改分母，导致趋势不可比较。
- 只优化 token 成本，让质量、重试或人工接管成本转移到别处。
- 把自动评分器当真值，却从不与人工集比较一致性和偏差。

## 小结

好的 Agent 指标栈由业务结果、任务质量、可靠性、延迟和成本五层组成。每个指标先定义问题、分子、分母、时间窗和维度，再决定采集方式。指标用于发现和决策，Trace、日志与评审证据用于解释；只有把它们连接起来，团队才不会被漂亮但无行动价值的大盘误导。
