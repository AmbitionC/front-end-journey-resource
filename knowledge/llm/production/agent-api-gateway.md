Agent Gateway 是所有外部请求进入 Agent 平台的统一边界。它验证调用者、建立租户上下文、实施配额和策略、选择兼容的执行后端、传播可观测上下文，并在响应离开前做统一过滤。它不应变成保存工作流状态、执行工具和拼装业务答案的“超级服务”，否则每次 Agent 变化都会把接入层一起拖入发布和故障域。

作为 HTTP 中间层，Gateway 必须尊重方法、状态码、表示、缓存和条件请求等语义。[RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html)是这些语义的基础；“统一接入”不意味着把所有错误都改成 200，也不意味着任意重试非幂等请求。

## 请求进入时的处理顺序

建议固定处理流水线，避免各路由各自实现一套安全逻辑：

1. 限制请求行、头、正文和上传大小，拒绝解析歧义；
2. 验证 TLS、令牌签名、受众、签发者、有效期和防重放信息；
3. 解析 tenant、principal、delegation 和 workload identity；
4. 根据资源、动作、主体、环境和策略版本授权；
5. 执行租户与操作级配额、并发和成本准入；
6. 规范化请求并选择版本兼容的后端；
7. 创建或接续 Trace，转发受控上下文；
8. 对响应做 Schema、内容、敏感数据和流式边界检查。

认证只回答“是谁”，授权才回答“能做什么”。[NIST SP 800-207](https://csrc.nist.gov/pubs/sp/800/207/final)强调零信任决策不应仅依据网络位置，而应结合主体、设备或工作负载、资源和策略。即使请求来自内网或同一集群，也不能跳过身份和资源级授权。

## 对外契约与内部执行解耦

外部 API 应使用稳定、版本化的任务契约，例如同步响应、异步 Job、流式事件和取消接口。内部可以更换模型、工作流引擎和工具实现，但不能让供应商错误码或临时字段直接泄漏到客户端。

对于耗时任务，Gateway 接受请求后返回 202 和 jobId，状态查询、事件流和取消由独立任务服务承载。Gateway 可以缓存路由与策略，但不把“任务已经执行到第几步”放进进程内存。重启、扩容或切换路由时，执行状态仍应由持久层恢复。

统一错误信封至少包含稳定 applicationCode、HTTP status、requestId、可重试提示和安全的 detail。认证失败、授权拒绝、配额耗尽、参数错误、上游不可用和策略拦截必须可区分；内部堆栈、Prompt 与工具密钥不能出现在响应里。

## 路由是受约束的策略决策

模型路由不能只按“当前最便宜”选择。请求携带能力需求、数据分类、地区、上下文长度、工具集合、延迟等级和输出 Schema；路由器从已经批准的后端集合中选择满足约束者，再按健康、成本和负载排序。

一次路由决策记录 routeId、policyVersion、modelBundle、reason 和 fallbackEligibility。对话或长任务需要考虑版本粘性：中途换模型可能改变工具参数、上下文格式和安全行为。若发生降级，响应与审计要标明，不把能力缩水伪装成等价成功。

## 配额、成本与过载

配额至少分成每租户请求速率、并发 Run、模型 token、工具调用和预算。限流应发生在昂贵解析和上游调用之前。单一全局令牌桶会让大客户吞噬容量，通常需要租户桶、操作桶和平台保护阈值组合。

过载时先拒绝低优先级新工作，保护已经开始的任务和控制面。响应返回明确的 429 或 503 语义以及合理 Retry-After；不要在 Gateway 内无限排队。重试会放大压力，因此网关只对明确幂等、尚未提交且预算允许的操作做有限重试。

## Trace 与安全上下文

[W3C Trace Context](https://www.w3.org/TR/trace-context/)定义了 traceparent 和 tracestate 的传播格式。Gateway 校验入站头的长度与格式，决定接续还是创建新 Trace，再把 Trace 与 runId、tenantId 关联。不要把用户邮箱、token、Prompt 或数据分类原文塞进 tracestate，因为它会跨多个服务和供应商传播。

跨信任边界时只转发允许列表头，内部身份通过重新签发的短时服务凭据表达，不能原样转发用户 bearer token 给所有下游。日志记录主体 ID 和授权决定的引用，秘密与完整正文进入受控证据系统或完全不采集。

## 流式响应的特殊处理

SSE 或流式 RPC 一旦发送响应头，错误语义就与普通 HTTP 不同。Gateway 应在首字节前完成认证、配额和基本策略检查；流中事件使用稳定类型，并能表达正常结束、取消、策略终止和上游失败。内容过滤若需要看完整输出，就不能假装可对已经发出的 token 回滚。

断线后是否续传由事件序号和保留窗口决定。客户端重连不能自动创建第二次副作用任务；使用 jobId、Last-Event-ID 或显式 resume token 接续。Gateway 记录首字节延迟、总时长、客户端取消和已发送字节，便于区分模型慢与网络断开。

## 高可用与变更

Gateway 是高扇入组件，配置错误影响面极大。实例保持无状态，策略与路由配置版本化并签名，发布采用小流量 Canary。依赖的身份、策略和配额系统要有缓存与失败策略：授权信息过期时高风险写操作默认拒绝，公开只读路径可按明确规则短时降级。

测试包括方法和状态语义、Header 走私、超大请求、令牌边界、跨租户授权、配额竞争、路由不兼容、Trace 污染、流式中断和下游慢响应。故障演练验证某模型、区域或策略服务失效时能否快速隔离，而不是把整个入口拖死。

## 小结

Agent Gateway 的价值是形成一个可审计的统一边界：正确保留 HTTP 契约，基于身份与资源实施零信任策略，控制配额和成本，选择兼容执行后端，并安全传播 Trace。它应该薄而强：统一接入规则，但把长任务状态、模型推理和工具副作用留给专门的执行平面。
