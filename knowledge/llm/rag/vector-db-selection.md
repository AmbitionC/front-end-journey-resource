向量数据库的任务不只是“存一个 `number[]`”。它还要在大量向量中执行相似度搜索，管理元数据过滤、增量更新、多租户隔离、备份监控和成本。

Pinecone、Chroma 与 pgvector 没有脱离场景的绝对胜负。正确顺序是先明确约束和评测数据，再选择最匹配的产品。

## 一、先判断是否真的需要专用向量库

数据规模不大时，可以在应用内或数据库中做精确搜索；当向量数、并发和延迟要求提高，通常需要 ANN 索引。

ANN（近似最近邻）用部分召回换取速度和资源。选择数据库前先定义：

- 向量数量、维度、增长与删除频率。
- 峰值查询/写入吞吐和 P95 延迟目标。
- 元数据过滤、租户与 ACL 模型。
- 是否要与关系数据事务性联结。
- 数据驻留、备份、审计和合规要求。
- 团队能承担多少运维，以及预算如何计量。

## 二、共同核心：距离、索引和过滤

三种方案都围绕相同概念：记录 ID、向量、原文/引用和 metadata。查询会提供查询向量、距离函数、top-k 与过滤条件。

索引参数决定速度—召回权衡。例如 pgvector 支持精确近邻以及 HNSW、IVFFlat 两类近似索引：HNSW 通常有更好的速度—召回权衡，但构建更慢、占用更多内存；IVFFlat 构建更快、内存更省，但依赖训练数据和 lists/probes 参数。

一个常被忽略的问题是“过滤后召回”。某些 ANN 实现先从全局索引取候选，再应用 metadata 条件；当过滤很严格时，最终可能不足 top-k。评测必须使用真实租户与 ACL 过滤，而不是只测无过滤向量查询。

## 三、pgvector：向量能力留在 PostgreSQL

pgvector 是 PostgreSQL 扩展，适合已经把核心业务数据放在 Postgres，并希望用 SQL、事务、备份和权限体系统一管理的团队。

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE chunks (
  id bigserial PRIMARY KEY,
  tenant_id bigint NOT NULL,
  document_id bigint NOT NULL,
  content text NOT NULL,
  embedding vector(1536) NOT NULL,
  source_version text NOT NULL
);

CREATE INDEX chunks_tenant_idx ON chunks (tenant_id);
CREATE INDEX chunks_embedding_hnsw
ON chunks USING hnsw (embedding vector_cosine_ops);

SELECT id, document_id, content,
       1 - (embedding <=> $1::vector) AS similarity
FROM chunks
WHERE tenant_id = $2
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

维度 `1536` 只是示例，必须与实际 Embedding 模型一致。距离运算符和索引 operator class 也要匹配。

优势：

- 向量与业务表可联结，事务和删除一致性更直接。
- 复用现有 Postgres 监控、备份、权限与团队能力。
- 可以同时使用 B-tree、GIN、全文搜索和向量索引。
- 早期系统少引入一个分布式组件。

代价：

- 索引会占用数据库 CPU、内存和 IO，可能与核心 OLTP 竞争。
- 大规模扩展需要分区、读副本或外部分布式方案，容量规划由团队负责。
- HNSW 构建、VACUUM、过滤和召回参数需要数据库调优经验。

适合：已有 PostgreSQL、需要复杂关系过滤或强一致删除、规模可由现有数据库承载的应用。

## 四、Chroma：快速本地与自托管实验

Chroma 提供开源 SDK，也可本地、自托管或使用云服务。Collection 可以存文档、metadata 和向量，支持稠密、稀疏、混合与 metadata 过滤。

它的优势是上手快，适合原型、个人工具、测试和希望控制部署环境的小团队：

- 本地开发不必先搭建复杂基础设施。
- 可以使用外部 Embedding，也可以配置 Embedding Function。
- 文档与 metadata 接口贴近 AI 应用。
- 从单机实验到 client-server/云形态有清晰路径。

需要评估：

- 生产部署、持久卷、备份、升级和高可用由谁负责。
- 预期数据量与并发下的真实性能和扩展方式。
- 多租户隔离是否只靠 collection/metadata 就能满足合规要求。
- 与业务关系数据的一致性如何维护。

适合：本地原型、自托管优先、规模和运维边界可控，希望快速验证检索效果的场景。

不要因为“能在本地跑”就自动认定适合生产；上线标准还包括恢复演练、监控、权限、滚动升级与容量测试。

## 五、Pinecone：托管式向量检索

Pinecone 是全托管向量数据库，适合希望把索引运维、弹性和可用性交给服务商的团队。当前官方能力覆盖稠密、稀疏、全文/混合检索、metadata 过滤和 namespace。

namespace 可以在一个索引内隔离记录，常用于多租户，也能缩小查询扫描范围。每次 upsert 与查询都必须明确目标 namespace。

优势：

- 服务托管，减少节点、索引和扩缩容的直接运维。
- 面向向量检索提供 namespace、批量导入和监控能力。
- 适合云原生服务和增长不确定的工作负载。

代价：

- 数据通过网络 API 访问，延迟、配额和区域需要实测。
- 与业务数据库是两个系统，双写、删除和恢复必须设计一致性流程。
- 成本模型和供应商能力会变化，不能把当前价格硬编码进架构结论。
- 数据驻留、网络出口和供应商锁定要经过评审。

适合：需要托管弹性、团队不希望自建向量基础设施，并能接受外部服务与相应合规边界的应用。

## 六、不要用一张静态表替代 PoC

下面是方向性差异，不是性能承诺：

| 维度 | pgvector | Chroma | Pinecone |
|---|---|---|---|
| 主要形态 | PostgreSQL 扩展 | 本地/自托管/云 | 全托管云服务 |
| 与关系数据联结 | 最直接 | 通常经应用层 | 通常经应用层 |
| 运维责任 | 团队管理 Postgres | 自托管时由团队负责 | 主要由服务商负责 |
| 原型速度 | 已有 Postgres 时快 | 很快 | 需云账号与网络配置 |
| 扩展路径 | 数据库调优/分区/扩容 | 依部署形态 | 按服务能力扩展 |
| 数据控制 | 自有数据库边界 | 自托管时高 | 受托管服务区域与条款约束 |

性能、容量和价格都会随版本与配置变化，因此不写“某产品最多多少向量”之类容易过期的静态数字。应以当前官方文档和自己的 PoC 为准。

## 七、多租户与 ACL 设计

向量相似度不能代替权限。检索必须先确定调用者，再限制候选数据：

- pgvector：`tenant_id` 条件、行级安全、分区或独立表/库。
- Chroma：按部署能力使用数据库/collection 隔离，并结合 metadata 过滤和应用鉴权。
- Pinecone：namespace 隔离租户，必要时再用 metadata 过滤角色和文档范围。

物理隔离更强，但资源和运维成本更高；共享索引成本低，但过滤错误会造成严重泄露。选择应依据租户规模、合规等级与噪声影响。

所有查询路径，包括调试接口、批量导出和后台工具，都必须执行相同 ACL。不要只在最终回答阶段要求模型“不要泄露”。

## 八、更新、删除与一致性

RAG 数据会变化。建议用稳定 chunk ID 和版本化流水线：

1. 源文档产生版本或内容哈希。
2. 解析与切块生成稳定 ID。
3. 新版本 upsert 后切换可见版本。
4. 删除旧版本和已经撤权的记录。
5. 定期对账源数据与向量库。

如果向量库与业务数据库分离，使用 outbox、变更日志或可重放任务，而不是在一次请求里无保护地“双写两个系统”。对删除和权限变更设置更高优先级，并监控同步延迟。

## 九、PoC 必测指标

用一份固定真实数据集，三种候选方案执行相同：

- 向量数、维度、metadata 分布与租户规模。
- 真实 top-k、过滤条件和并发模型。
- 相同的 Embedding、距离函数与召回目标。

记录：

- Recall@k / 与精确搜索对比的 ANN recall。
- P50、P95、P99 延迟和吞吐。
- 严格过滤后返回不足 k 的比例。
- 写入、更新、删除可见延迟。
- 索引构建/恢复时间和故障表现。
- 存储、计算、网络与人员运维的月度总成本。

如果没有精确搜索基线，就很难知道低延迟是否来自漏掉了重要邻居。

## 十、决策路径

可以按以下顺序缩小范围：

1. **已有 Postgres 且规模适中？** 先用 pgvector 做基线，少引入一个系统。
2. **目标是本地原型或可控自托管？** 用 Chroma 快速验证数据和检索策略。
3. **需要托管弹性并减少运维？** 评估 Pinecone 的区域、成本与合规。
4. **过滤和多租户复杂？** 用真实分布测试过滤后召回与隔离方案。
5. **增长路径不确定？** 保持存储适配层和可重建索引，不把业务逻辑绑定在专有查询结构中。

选型不是永久承诺。只要原始文档、切块、metadata 和 Embedding 版本可重放，向量索引就应当是可重建派生数据。

## 常见误区

- **先看品牌再看约束**：真实过滤、规模和运维能力才决定适配度。
- **只测无过滤查询**：生产 ACL 会改变召回和延迟。
- **ANN 指标等同回答质量**：还要评估重排、上下文和生成忠实度。
- **忽略删除同步**：旧内容和撤权内容仍可能被召回。
- **向量库当唯一数据源**：原文、版本与权限仍需可靠系统保存。
- **只算云账单**：自托管也有人员、备份和故障成本。

## 选型检查清单

- [ ] 已定义规模、增长、并发、延迟和召回目标。
- [ ] 明确 metadata、ACL、多租户与数据驻留要求。
- [ ] 使用同一真实数据与查询对候选方案做 PoC。
- [ ] 测试了严格过滤后的召回与不足 top-k 情况。
- [ ] 评估更新、删除、备份、恢复和版本迁移。
- [ ] 成本包含计算、存储、网络和运维人员。
- [ ] 原始数据与流水线可重放，索引可以重建。
- [ ] 应用有存储适配层，核心权限逻辑不依赖模型。
