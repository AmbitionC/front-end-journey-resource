混合检索（Hybrid Search）不是“关键词检索和向量检索都跑一遍”这么简单。它真正解决的问题是：**同一个查询里，既可能包含必须精确命中的词，也可能包含只能靠语义理解的意图**。生产级方案需要让两条检索链路各取所长，再用稳定的方法合并、重排和评估。

## 一、为什么只用一种检索经常不够

关键词检索擅长精确匹配。错误码、产品型号、函数名、人名和合同条款编号都属于它的强项。用户搜索 **ERR_CONNECTION_RESET** 时，把这个标识符拆成一个模糊语义向量，反而可能降低准确率。

向量检索擅长处理表达差异。用户问“网页一直转圈但没有报错”，文档标题可能写的是“请求挂起与超时排查”。两段文本没有多少共同词，语义却很接近。

两者也各有盲区：

- 关键词检索容易漏掉同义表达、口语描述和跨语言表达。
- 向量检索可能淡化数字、缩写、否定词和精确标识符。
- 任意一种方法都不能代替权限过滤、版本过滤和业务约束。

因此，混合检索的目标不是让结果“看起来更多”，而是提高相关证据进入候选集的概率，同时把最有用的证据排到前面。Elastic 的官方说明也将混合检索定义为把词法与语义等多个检索器合成为一个排名结果。[Elastic：Hybrid Search](https://www.elastic.co/what-is/hybrid-search)

## 二、两条召回链路分别做什么

### 关键词链路：BM25 与倒排索引

关键词检索通常建立倒排索引，并使用 BM25 一类相关性函数排序。可以把 BM25 理解为三个问题：查询词是否出现、在当前文档中是否足够重要、在整个语料中是否足够稀有。

它特别适合：

- API 名称、类名、错误码和 SKU；
- 法律条款、产品版本和专有名词；
- 用户明确输入了必须出现的词组；
- 需要高可解释性的搜索。

分词、大小写归一、同义词和字段权重都会影响结果。中文语料还要关注分词粒度：把一个专有名词切碎，可能比算法选择本身造成更大损失。

### 语义链路：Embedding 与近似最近邻

语义检索先把查询和文档块映射为向量，再根据余弦距离、点积或欧氏距离寻找近邻。大规模系统一般使用 ANN 索引，以少量召回损失换取延迟和吞吐。

它特别适合：

- 同义改写和自然语言问题；
- 用户描述现象，但不知道标准术语；
- 多语言或跨表达方式检索；
- “意思相关”比“字面相同”更重要的内容。

Embedding 模型、距离函数、索引配置必须匹配。文档向量和查询向量也应来自兼容的模型版本，否则距离数字没有可靠含义。

## 三、一条可落地的混合检索流水线

![混合检索将关键词与语义结果通过 RRF 融合、重排后送入 RAG 上下文](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/rag-hybrid-search-pipeline-v2.png)

图中最重要的不是“两条线”，而是五个顺序约束：

1. **先做身份与过滤约束**：确定租户、权限、语言、版本和时间范围。
2. **并行召回**：BM25 与向量检索各自返回带排名的候选集。
3. **融合排名**：不要直接把两个来源的原始分数相加。
4. **重排**：对较小候选集使用更精细的相关性模型或规则。
5. **组装上下文**：去重、控制 token、保留来源与结构，再交给生成模型。

权限条件必须进入检索路径。不能先从全库召回，再期待生成模型“自觉忽略”无权内容；这既可能泄漏片段，也会让无权文档挤占 top-k。

## 四、为什么 RRF 常适合作为第一版融合方法

BM25 分数与向量相似度来自不同空间。一个可能是十几分，另一个可能在 0 到 1 附近，直接相加等于把单位不同的数字混在一起。

Reciprocal Rank Fusion（RRF）不比较原始分数，只看文档在各候选列表中的名次：

$$
\operatorname{RRF}(d)=\sum_{i=1}^{m}\frac{1}{k+\operatorname{rank}_i(d)}
$$

其中 $m$ 是检索器数量，$k$ 是平滑常数。一个文档若同时在两条链路中排名靠前，会得到更高融合分。Elastic 的 RRF 文档强调，它能组合相关性指标不同的结果集，而不要求这些分数处在同一量纲。[Elasticsearch：Reciprocal rank fusion](https://www.elastic.co/docs/reference/elasticsearch/rest-apis/reciprocal-rank-fusion)

下面是一个最小 TypeScript 实现，便于理解，不代表数据库侧实现一定要搬到应用层：

~~~ts
type RankedItem = { id: string; rank: number };

function rrf(lists: RankedItem[][], k = 60) {
  const scores = new Map<string, number>();

  for (const list of lists) {
    for (const item of list) {
      const score = 1 / (k + item.rank);
      scores.set(item.id, (scores.get(item.id) ?? 0) + score);
    }
  }

  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1])
    .map(([id, score]) => ({ id, score }));
}
~~~

RRF 的优点是稳定、少调参、容易建立基线；局限是它主要利用名次，不能天然表达“某一检索器在特定查询上更可信”。当你拥有足够标注数据后，可以尝试归一化加权、按查询类型动态路由，或学习排序模型，但必须用同一评测集证明收益。

## 五、重排解决的是“细看”，不是“补召回”

召回阶段需要快，通常只做粗筛；重排器可以对几十到几百个候选逐一比较查询与文本，识别否定、条件和细粒度语义。

常见顺序是：

~~~text
大语料库
  → BM25 top 100 + Vector top 100
  → RRF 去重融合
  → Reranker top 20
  → 上下文选择 top 5～10
~~~

重排器不能找回从未进入候选集的文档。因此调优时要先看召回，再看排名：如果正确文档不在 top 100，继续换重排器意义不大。

## 六、查询路由与动态权重

并非所有查询都应使用同一比例。可以先做轻量分类：

| 查询特征 | 更应重视 |
|---|---|
| 包含错误码、版本、编号、引号短语 | 关键词检索 |
| 完整自然语言问题、现象描述、同义改写 | 语义检索 |
| 两类特征同时存在 | 并行召回 + RRF |
| 含明确时间、租户、文档类型 | 先做结构化过滤 |

路由器本身也会出错，所以第一版通常应保留双路召回，用离线数据观察不同查询切片，再决定是否省略某一路以降低成本。

## 七、如何评估与调优

建立包含真实查询、相关文档和业务切片的评测集，至少比较：

- BM25 单路；
- 向量单路；
- 双路 RRF；
- RRF + 重排。

召回层关注 Recall@K、MRR、nDCG；RAG 端还要关注答案忠实度、引用正确率、延迟和成本。不要只看平均值，要按错误码查询、长问题、中文口语、跨语言和强过滤场景切片。

一次只改一个主要变量，例如分块、Embedding、候选数、RRF 参数或重排器。否则指标变好时无法知道原因，变差时也很难回滚。

## 常见误区

- **把混合检索写成分数相加**：不同检索器的分数通常不可直接比较。
- **只扩大 top-k**：候选越多不一定越好，噪声和生成成本也会增加。
- **让重排器弥补召回失败**：未召回的文档无法被重排。
- **最后才做权限过滤**：这既影响安全，也破坏排名质量。
- **只用几个“看起来合理”的问题测试**：需要冻结评测集和回归门槛。
- **忽略更新与删除**：旧向量、撤权文档和新索引版本都可能污染结果。

## 小结

混合检索是一条分层流水线：结构化过滤守住边界，BM25 保住精确词，向量检索理解语义，RRF 稳定融合，重排器做细粒度判断，评测集负责证明改动有效。先建立可解释的基线，再用数据决定是否引入动态权重和复杂学习排序。
