用户问题往往不是适合检索器的查询：有口语、省略、错别字、多个子问题，或把时间和版本条件藏在长句里。查询改写的目标是提高相关证据进入候选集的概率，同时保留原始意图和硬约束。它不是让模型“重新回答一遍”，更不能把生成的改写当成事实来源。

## 先诊断查询缺陷，再选策略

不同问题需要不同动作：

| 查询现象 | 合适动作 | 主要风险 |
|---|---|---|
| 拼写、大小写、格式噪声 | 规范化 | 改坏专有名词 |
| 指代或范围不清 | 请求澄清 | 增加交互成本 |
| 一个问题包含多跳条件 | 分解为子查询 | 丢失子问题依赖 |
| 表达太窄、召回不足 | 多查询扩展 | 噪声和成本上升 |
| 只有抽象概念、无词面线索 | HyDE 类假设文档 | 生成内容可能虚构 |

不要每次同时运行所有策略。先用规则与轻量分类识别错误码、引号短语、实体、时间、否定词和多问题结构，再路由到最小必要变换。

*图：生成的规范化查询、多查询或 HyDE 文档只用于寻找真实语料，不直接进入答案证据链。*

## 永远保留原始查询与硬约束

改写输出使用结构化契约，而不是一个裸字符串：

```ts
type RewritePlan = {
  original: string;
  intent: string;
  entities: Array<{ text: string; type: string }>;
  hardFilters: {
    tenantId: string;
    product?: string;
    version?: string;
    language?: string;
    validAt?: string;
  };
  strategy: "normalize" | "clarify" | "decompose" | "multi-query" | "hyde";
  queries: string[];
  unresolved: string[];
};
```

`tenantId`、权限、时间和明确版本来自可信业务状态或可验证解析，不能让模型随意删除或改写。原始查询和改写查询一起进入日志与评测，便于发现语义漂移。

## 规范化与消歧

规范化处理 Unicode、空白、常见拼写、日期和已知别名，但应为错误码、SKU、类名和引号词组设置保护规则。例如 `ERR_CONNECTION_RESET` 不应被模型改成“网络连接问题”，否则关键词检索丢掉最精确的信号。

遇到“它为什么失败”“最新版支持吗”这类缺少指代或范围的问题，可靠选择通常是澄清，而不是自信补全。可以先用会话中已确认的实体填充；仍有多个高概率解释时，向用户展示简短选择。

## 分解多跳问题

“A 版本的权限策略和 B 版本相比有什么变化，迁移后如何回滚”至少包含版本变化、迁移步骤和回滚条件。分解器生成带依赖的子查询：

```text
q1: A 与 B 的权限策略差异
q2: 从 A 迁移到 B 的前置条件
q3: B 版本的回滚步骤与限制
```

每个子查询继承权限、产品和时间过滤；合并答案时保持来源与子问题对应。若 q2 依赖 q1 的真实检索结果，应先检索 q1，再从证据提取实体，不能让模型编造中间事实继续查询。

## 多查询扩展与结果融合

多查询用同义表达、专业术语或不同视角扩大召回。例如“网页一直转圈”可扩展为“请求挂起”“加载无响应”“网络超时排查”。每个查询限制数量和 token，避免十几个近义句制造重复成本。

不同查询的相似度分数通常不可直接相加。可以按排名使用 Reciprocal Rank Fusion，或先做明确校准再加权。[RAG-Fusion](https://arxiv.org/abs/2402.03367) 展示了多查询与倒数排名融合的组合思路，但实际收益必须在自己的语料、语言和查询切片上验证。

融合后按稳定文档/块 ID 去重，保留命中的查询列表。一个文档被多个独立改写召回是有用信号，但不代表它一定支持答案。

## HyDE 的正确边界

[HyDE 原论文](https://arxiv.org/abs/2212.10496)先让语言模型生成一个“假设相关文档”，再把它编码成向量去真实语料空间找邻居。论文明确指出假设文档可能包含错误细节；它的作用是提供更丰富的语义检索表示，而不是成为答案上下文。

因此数据流应是：

```text
原始问题 → 假设文档 → 向量 → 真实语料检索 → 真实文档证据
```

不能是“假设文档 → 直接回答”。对法规、财务、医疗、版本号和精确标识符查询，HyDE 漂移风险更大，通常应保留关键词检索并设置严格过滤。

## 防止改写漂移

改写前后做约束检查：实体、数字、否定、时间、版本、范围和输出意图是否保留。可将关键槽位固定输入给模型，并要求输出“新增的推断”列表。出现未授权实体、删除否定词或改变时间范围时，回退到原始查询。

还要限制提示注入：用户查询和检索内容都是数据，改写器不能因为其中写着“忽略租户限制”就改变服务端过滤。改写只影响检索文本，授权过滤由代码构造。

## 如何评估

离线数据保存原始查询、相关文档和问题切片，比较：不改写、单一规范化、分解、多查询、HyDE 以及各自融合方案。召回层看 Recall@K、MRR、nDCG 和零结果率；同时监控查询数量、重复候选、延迟和成本。

端到端再看答案正确、引用支持和任务完成。若 Recall@K 上升而答案变差，可能是噪声挤占上下文、融合排序错误或改写改变意图。按错误码、长问题、中文口语、多跳、强过滤和高风险查询分别看指标。

## 常见误区

- 所有查询都先让 LLM 改写；
- 改写后丢掉原始问题和硬过滤；
- 把多查询的原始相似度直接相加；
- 把 HyDE 假设文档引用为答案来源；
- 只增加查询数量，不做去重和预算；
- 用召回提升代替端到端答案评测；
- 允许改写器修改租户或权限约束。

## 小结

查询改写是一层受控的检索规划：先识别缺陷，再选择规范化、澄清、分解、多查询或 HyDE；原始意图和硬约束始终保留，所有生成内容只用于召回。最终证据必须来自真实、已授权语料，并用离线召回与端到端质量共同证明收益。
