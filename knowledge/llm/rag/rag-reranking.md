召回器的任务是在大语料中快速找出“可能相关”的候选，重排器的任务是在较小候选集里“仔细比较谁更相关”。把两者拆开，是生产检索同时获得吞吐和精度的常见方法。重排不能找回未被召回的文档，因此调优顺序必须先召回、后排名。

## 两阶段检索的分工

第一阶段通常使用 BM25、稠密向量或混合检索，从百万级块中取 top-k。它强调高召回与低延迟，查询和文档常被分别编码或通过倒排索引匹配。

第二阶段让更昂贵的模型同时看到查询与候选文本，逐对输出相关性分数，再取 top-n 进入上下文。[Passage Re-ranking with BERT](https://arxiv.org/abs/1901.04085) 展示了把预训练语言模型作为段落相关性分类器的经典范式；当前商业 Rerank 服务也是这一两阶段思想的工程实现之一。

![快速召回先得到 top-k，重排器重新排序并选出小规模 top-n；未召回文档无法被恢复](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/rag-reranking-two-stage-ranking-v1.webp)
*图：低位但真正相关的候选可以被重排上移；候选集之外的文档没有机会参与。*

## 候选深度是第一关键参数

`k` 太小，正确文档可能进不了候选集；`k` 太大，重排延迟、费用和噪声都增加。`n` 则受上下文预算、文档长度和证据多样性限制。不要照抄固定的 `k=100, n=10`，应画出自己的质量—延迟曲线。

```ts
const candidates = await hybridRetrieve(query, {
  topK: policy.candidateDepth,
  filter: authorizationFilter,
});

const ranked = await reranker.rank({
  query,
  documents: candidates.map(item => item.text),
  topN: policy.contextDepth,
});

const evidence = ranked.map(result => ({
  ...candidates[result.index],
  rerankScore: result.score,
}));
```

当前 [Cohere Rerank 文档](https://docs.cohere.com/docs/rerank-overview)的结果包含原始输入 `index` 和相关度分，可据此映射回文档；具体模型、字段限制和费用属于截至 2026-07-15 的供应商实现，接入时应重新核对。

## 分数只在同一契约内解释

重排分数通常不是“答案正确概率”，也不能天然跨模型、跨查询或跨版本比较。它表示当前模型对查询—文档相关性的内部排序信号。阈值必须用带标签数据校准，并按任务切片观察。

供应商或模型升级后，分数分布可能变化。保存模型版本、候选深度、文本预处理和分数，用排名指标验证，再更新阈值。不要把 `0.8` 写成永久业务规则。

## 输入单位与截断

重排对象应是能独立判断相关性的块，同时保留标题、章节、文档类型、版本等必要上下文。块过短会缺少语义，过长会增加截断和费用。若服务自动截断，查询命中的关键内容可能正好位于尾部。

可以使用结构化拼接：

```text
[TITLE] 退款政策
[SECTION] 企业年付计划
[VERSION] 2026-06
[CONTENT] ...
```

元数据只加入与相关性判断有关且允许模型看到的字段。租户、权限等约束应在召回前由检索系统强制执行，不能交给重排器决定。

## 相关性之后还要做上下文选择

最高分的前五块可能全部来自同一文档、重复同一事实。重排后可增加确定性规则：按文档去重、限制单文档块数、合并相邻块、覆盖不同子问题，并保留最高分证据。多样化不能把低质量文档强塞进上下文，要在相关性门槛内进行。

如果答案需要多跳证据，独立给每个块打分可能低估组合价值。可以先分解子问题分别重排，再按子问题配额组装；更复杂的列表式或生成式重排只有在基线不足且评测证明收益时再引入。

## 延迟、成本与降级

重排延迟大致随候选数量和文本长度增长。生产层设置：批大小、并发、超时、总字符/token 上限和缓存。缓存键包含查询规范化结果、候选内容哈希、授权范围和重排器版本，不能跨租户复用未隔离结果。

超时后的降级策略由风险决定：低风险搜索可以退回融合排名；高风险问答可能宁可返回“暂时无法验证”，也不使用未重排的噪声证据。记录降级事件，避免表面可用掩盖长期故障。

## 分层评测

先单独评召回：相关文档是否进入 top-k，使用 Recall@K。再评重排：在固定候选集上使用 MRR、nDCG@N、Precision@N，并观察正确文档排名变化。最后评 RAG：答案忠实度、引用正确、任务成功、延迟和费用。

做消融对比：第一阶段原始排名、重排、重排加去重/多样化。按精确标识符、语义问题、长文档、语言、强过滤和多跳问题切片。平均 nDCG 提升可能掩盖某个重要业务切片退化。

## 常见误区

- 候选没召回，却继续调重排模型；
- 把重排分数解释为事实正确概率；
- 使用固定 k/n，不画质量—延迟曲线；
- 把整个超长文档直接送去重排；
- 让重排器承担租户授权；
- 只看排名指标，不看答案与引用；
- 模型升级后沿用旧分数阈值。

## 小结

Rerank 是“宽召回、窄精排”的第二阶段：第一阶段确保相关证据进入候选集，第二阶段用更强的查询—文档交互重新排序，再经去重和上下文预算选出证据。候选深度、截断、分数校准和分层评测共同决定它是否真正改善答案。
