长上下文让模型一次接收更多 token，却不自动保证它能找到、理解并正确使用其中的信息。工程上的目标不是“把窗口塞满”，而是让关键证据在有限预算内可发现、可引用、可验证。

## 三个不要混为一谈的长度

1. **声明窗口**：模型接口允许输入与输出合计占用的最大 token 数；
2. **可运行长度**：受显存、KV Cache、并发和超时约束，系统实际能承载的长度；
3. **有效上下文**：在具体任务与证据位置下，模型仍能稳定利用的部分。

第三项通常小于前两项。窗口扩大解决的是“能否放进去”，不是“能否可靠取出”。[Lost in the Middle](https://arxiv.org/abs/2307.03172) 在多文档问答和键值检索实验中发现，相关信息位于上下文开头或结尾时表现往往更好，位于中间时可能明显下降。因此，只用一个最大长度数字描述能力会掩盖位置效应。

## 位置如何进入注意力

Transformer 的自注意力本身没有天然顺序，需要位置编码告诉模型 token 的相对或绝对位置。RoPE 将位置信息编码为查询与键向量的旋转，使注意力内积带有相对位置关系；论文同时展示了距离增大时相关性衰减等性质。[RoFormer](https://arxiv.org/abs/2104.09864) 是理解现代长上下文位置机制的重要起点。

把训练长度之外的位置“外推”到更远并非免费：位置分布变了，局部模式、远距离依赖和数值尺度都可能偏离训练阶段。不同模型会采用频率缩放、继续预训练或专门长上下文数据等方案，不能仅凭“使用 RoPE”推断外推质量。

## 成本不只来自输入 token

标准全注意力在序列长度增加时需要计算更多 token 对之间的关系；生成阶段还要保存历史 token 的 Key/Value。粗略看，单层 KV Cache 与 `序列长度 × KV 头数 × 每头维度 × 2 × 字节数` 成正比，再乘层数和并发请求数。

因此长上下文服务常同时受到三类约束：预填充阶段的计算与延迟、生成阶段的 KV Cache 显存、并发调度带来的峰值资源。即使模型规格允许更长输入，生产网关仍应设置比理论上限更谨慎的预算。

## 从“最大窗口”转向“工作上下文”

![从最大上下文中检索证据、验证并构造工作上下文的流程](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/llm-long-context-effective-context-loop-v1.webp)
*图：关键证据可能位于任意位置；先检索、再验证、最后回答，比无差别堆入全文更可控。*

可把上下文分成两层：

- **最大上下文**保存所有可能相关的原始材料；
- **工作上下文**只放当前步骤必须读取的指令、证据、状态和输出约束。

构造工作上下文时，先固定不可丢失的系统约束和用户目标，再按问题检索证据；保留来源、时间和段落标识；把最关键证据放在清晰、稳定的位置；最后要求答案引用证据或在证据不足时停止。长上下文与 RAG 不是对立选项：检索负责缩小搜索空间，长窗口负责容纳更完整的局部证据和对照材料。

## 一套可执行的上下文装配顺序

1. 明确任务、输出格式和不可违反的约束；
2. 对原始材料去重，并过滤与任务无关的导航、样板和旧版本；
3. 检索候选片段，同时保留标题、来源、时间与顺序信息；
4. 把直接回答问题的证据与背景资料分区；
5. 对冲突证据并列呈现，不在压缩时提前“选边”；
6. 估算输入、期望输出和安全余量，超预算时优先删除低价值内容；
7. 记录最终送入模型的上下文版本，便于复现失败。

摘要不能替代原文证据。涉及数字、合同条款、代码和时间顺序时，应该保留原始片段；摘要只承担导航或低风险背景压缩。

## 怎样测试“有效利用”

[LongBench](https://arxiv.org/abs/2308.14508) 将长文本单文档、多文档、摘要、Few-Shot、代码等任务放在统一评测中，说明长上下文能力需要跨任务观察。自己的系统至少应建立下面的矩阵：

- **长度**：短、中、接近生产上限；
- **位置**：关键证据在开头、中间、结尾；
- **干扰**：无干扰、相似干扰、相互冲突的证据；
- **答案类型**：直接抽取、多跳组合、拒答；
- **指标**：正确率、引用命中率、遗漏率、延迟、首 token 时间和显存峰值。

不要只做“上下文中藏一个随机字符串”的针测试。它能检查检索，但不能代表多跳推理、冲突消解或真实文档理解。还应验证把同一事实换位置、加入近似事实、删除必要证据之后，行为是否符合预期。

## 常见失败模式

- **中间信息被忽略**：把证据挪到边缘后才答对，说明存在位置敏感；
- **旧信息覆盖新信息**：缺少时间或版本标记，模型把多个状态混合；
- **摘要漂移**：多轮摘要逐渐丢失数字、否定词或责任主体；
- **干扰诱导**：高词面相似片段压过真正证据；
- **预算挤压输出**：输入占满窗口，回答被截断；
- **延迟雪崩**：单请求可运行，但并发下 KV Cache 和预填充拥堵。

排查时不要直接换更大窗口。先保存完整请求，标注关键证据的位置与 token 范围，再分别改变位置、删掉干扰、缩短上下文和固定采样参数，判断问题来自检索、位置、推理还是服务资源。

## 实践检查表

- 用目标模型的 tokenizer 计算预算，不按字符数猜测；
- 给输出、工具结果和重试预留空间；
- 将稳定指令、动态证据、历史状态分区；
- 对关键数字、引用和时间线保留原文；
- 同时测试长度与证据位置，而非只测窗口上限；
- 监控输入 token、预填充延迟、KV Cache 使用率和截断次数；
- 证据不足时允许拒答，不让模型用长上下文制造确定感。

## 小结

长上下文是一种容量，不是可靠性承诺。真正的有效利用来自位置机制、上下文装配、检索、证据保护和按长度/位置分层的评测。设计时应围绕“工作上下文”优化，让模型读取少而关键的材料，并让每个结论都能回到原始证据。
