大模型最擅长的事情，是根据上下文生成**统计上合理的后续内容**。它可以写出流畅解释、归纳模式、转换格式并组合已有知识，但“说得像真的”与“经过验证是真的”不是同一个目标。理解这条边界，是正确设计 AI 应用的起点。

## 为什么必须认识能力边界

如果把大模型当作确定性的知识库或计算器，系统会在三个地方失控：模型可能补齐不存在的事实，调用错误工具造成真实副作用，或者在医疗、法律、财务等高风险场景给出看似自信的错误建议。

NIST 将这类现象称为 **confabulation（虚构）**：生成系统自信地呈现错误内容、偏离输入，或与同一上下文中的其他输出矛盾。它并非偶发的“数据库坏记录”，而与生成模型根据训练分布预测输出的机制有关。[NIST AI 600-1](https://doi.org/10.6028/NIST.AI.600-1) 因而建议把虚构作为生成式 AI 的系统性风险管理，而不是只靠一句“请不要胡说”的提示词解决。

## 从概率分布到一句答案

给定已有 token $x_{<t}$，自回归语言模型估计下一个 token 的条件分布：

$$
P(x_t \mid x_{<t})
$$

解码器再根据这个分布选择一个 token，重复执行，最终形成整段文本。训练目标主要奖励“预测训练数据中的后续 token”，并不直接验证一句话是否对应现实、引用是否存在或结论是否满足业务规则。

*图：高置信度只代表模型分布集中，不等价于现实中的正确；重要输出需要检索、工具、校准和人工复核。*

这也解释了两个常见误区：

- **低温度不等于事实正确。** 降低 temperature 会让选择更稳定，但也可能稳定地产生同一个错误。
- **token 概率不等于答案真实性。** 一个词在当前句子中很自然，只能说明语言连贯，不能证明整条因果链、日期或数字正确。

## 四层能力边界

### 1. 表达与变换

改写、摘要、分类、翻译、抽取结构等任务，通常能在输入材料内完成，风险相对可控。此时模型主要做模式变换，系统仍需用 schema、类型和长度规则校验输出。

### 2. 记忆中的知识

模型参数压缩了训练数据中的统计规律，并不是可查询、可追溯、实时更新的事实表。冷门实体、训练截止日期之后的事件、精确引文和频繁变化的信息，都应通过检索或权威 API 获取。

### 3. 推理与计算

模型能生成有用的分解过程，但长链条会累积错误。算术、代码执行、约束求解和数据聚合应交给计算器、解释器、数据库或测试框架，并让模型解释工具结果，而不是凭文本模拟执行。

### 4. 行动与责任

当输出会触发付款、删除数据、发送消息或修改生产环境时，问题不再只是“答案质量”，还包括权限、幂等、审计和可逆性。模型可以提出行动，系统必须控制可调用工具、参数范围和人工审批点。

## 幻觉为什么难以彻底消除

原因往往是叠加的：输入本身含糊或缺少证据；训练数据冲突、过时或覆盖不足；模型在分布外问题上仍被要求继续生成；解码过程放大了早期错误；应用又没有把“不知道”设计成合法结果。

基于人类反馈的指令微调可以改善帮助性、真实性和安全性，但原始 [InstructGPT 研究](https://arxiv.org/abs/2203.02155) 也明确指出模型仍会犯简单错误。对齐降低某些失败概率，不会把概率模型变成事实判定器。

## 如何构建可靠性闭环

一个可落地的流程通常包含以下层次：

1. **限定任务**：说明允许使用的材料、时间范围和输出 schema。
2. **提供证据**：对变化快或私有的信息使用 RAG、数据库或业务 API。
3. **交给工具验证**：数字用计算器，代码用测试，引用检查 URL 和原文。
4. **让不确定性可见**：要求列出假设、缺失信息和证据来源；不要把自然语言自评当成精确概率。
5. **按风险设置人工门槛**：低风险内容可抽样，高风险决策必须由有责任的人确认。
6. **持续评估**：保存失败样本，按事实性、完整性、拒答质量和工具调用正确率做回归测试。

例如，回答“合同是否允许提前终止”时，正确流程不是让模型凭记忆回答，而是检索指定合同条款、引用段落、标记解释的不确定性，并把最终法律判断留给专业人员。

## 常见误区

- **“模型说自己有 95% 把握，所以可信。”** 自报置信度可能没有经过校准，应以任务数据上的可靠性曲线和外部验证为准。
- **“加上来源格式就不会伪造引用。”** 格式约束只能保证形状；URL、作者和论断仍须实际核对。
- **“RAG 能消灭幻觉。”** 检索可能召回错误、过时或无关片段，生成阶段也可能误读证据。
- **“更大的模型没有边界。”** 扩展通常改善平均能力，但不会自动解决最新性、权限、可追溯性和责任归属。

## 小结

大模型是概率生成器，不是事实数据库、执行环境或责任主体。可靠系统的关键不是幻想模型永不出错，而是让证据、工具、权限、评估和人工复核共同包住模型的边界。
