选择模型不是给厂商排座次，而是为一个明确的业务任务选择“当前可用、满足约束、可被回归验证”的具体版本。模型更新很快，稳定的方法必须把选型、版本锁定、灰度和回滚连成闭环。

## 先定义选择对象

生产配置至少要能唯一描述：

$$
\text{provider}+\text{model ID}+\text{snapshot/alias}+\text{API surface}+\text{region}
$$

同一家族的旗舰、均衡、轻量、推理和多模态版本可能有不同的参数、上下文、工具支持和价格。只写“用 GPT”或“用 Qwen”无法复现实验。

## 第一步：把业务任务写成可验收定义

先描述输入、输出、失败条件和业务损失，而不是先列模型名。例如“客服总结”仍然过于宽泛，更可测的定义是：

- 输入：最多 30 轮中文客服对话，可能包含订单号和退款状态；
- 输出：固定 JSON，包含问题、已采取动作、待跟进动作；
- 质量：关键事实不得遗漏或编造；
- 性能：P95 总时长低于业务阈值；
- 失败：JSON 不合法、订单号错误或调用未授权工具。

任务定义决定评测数据和指标。没有它，“更聪明”只是主观印象。

## 第二步：先用硬约束淘汰候选

硬约束不满足时，质量再高也不能上线：

| 约束 | 需要确认的事实 |
|---|---|
| 合规 | 数据区域、保留策略、DPA、敏感数据处理 |
| 模态 | 文本、图像、音频、文件及各自限制 |
| 上下文 | 输入与输出预算、工具和推理 Token 的计数方式 |
| API 能力 | 结构化输出、工具调用、流式事件、批处理 |
| 部署 | 公有 API、专有实例、自托管与硬件条件 |
| 可运维性 | 速率限制、版本快照、弃用窗口、状态页和支持渠道 |

不要把“开放权重”直接等同于“可低成本私有化”。还要计算 GPU、量化、推理框架、监控、安全更新和团队运维成本。

## 第三步：构建候选版本短名单

短名单应覆盖不同的成本—能力档位，而不是只放旗舰模型。常见组合是：

- 一个高质量候选，用作能力上界；
- 一个均衡候选，作为主要生产候选；
- 一个低成本/低延迟候选，处理简单高频任务；
- 一个不同供应商或部署方式的降级候选。

记录查询日期和官方文档链接。模型别名、价格与参数支持会变化，不能依靠记忆。

## 第四步：用真实数据离线评测

### 数据集

从真实业务分布抽样，并单独覆盖困难和高风险切片：长输入、歧义、缺失字段、工具失败、提示注入、非目标语言和拒答场景。开发集用于改 Prompt，盲测集用于比较，避免把答案“教”进示例。

### 指标

至少同时测三组指标：

1. **质量**：任务正确率、字段级 F1、引用正确率、代码测试通过率、工具参数正确率。
2. **性能**：TTFT、总时长、P50/P95/P99、限流和超时率。
3. **经济性**：输入、输出、缓存、推理 Token、重试与失败造成的总成本。

比较“完成一次有效任务的成本”：

$$
C_{success}=\frac{C_{all\ requests}+C_{retry}+C_{fallback}}{N_{valid\ outcomes}}
$$

只看每百万 Token 单价会忽略输出长度、失败率和重试。

### 保持条件可比

固定 Prompt、工具 schema、最大输出、超时和重试；每个候选运行足够多次以观察随机性。若模型不支持同一参数，应明确记录差异，而不是静默替换。

## 第五步：建立路由、降级和回滚

单模型方案简单，但关键业务通常需要：

- 主模型：通过完整评测的默认路径；
- 降级模型：主服务超时、限流或异常时接管；
- 任务路由：按风险、复杂度、模态或输入长度选择模型；
- 质量闸门：解析、规则或验证器失败时重试、升级模型或转人工；
- 回滚配置：保留上一稳定模型、Prompt 和工具 schema。

降级不只是换模型 ID。不同供应商的角色、工具调用、流式事件和错误语义可能不同，应通过适配层统一，再分别做集成测试。

## 版本锁定：别名与快照怎样取舍

- **浮动别名**：便于自动获得更新，适合探索环境；行为可能变化。
- **固定快照**：更容易复现和回滚，适合生产；需要主动迁移。

若厂商只提供浮动版本，应保存每次响应中的实际模型标识，并加强回归监控。订阅弃用公告，把迁移日期提前放入工程计划。

截至 2026-07-13，DeepSeek 官方已公告旧别名 `deepseek-chat` 与 `deepseek-reasoner` 将于 2026-07-24 停用。这类变更正是版本治理不能省略的原因。

## 灰度发布与持续回归

上线先从小流量、低风险用户或影子流量开始，观察质量、成本、延迟和错误分布。模型、Prompt、示例、检索、工具 schema 任一变化，都应触发同一套离线回归；高风险变更再经过灰度。

不要只比较聚合平均值。某次升级可能总体分数更高，却让一个关键语言或客户切片退化。

## 常见误区

- **默认使用旗舰**：复杂度不足的任务可能只是多花成本和延迟。
- **只看公开榜单**：榜单任务、提示和评分标准与业务不同。
- **把大上下文当作质量保证**：可接收长度不等于有效利用长度。
- **用硬编码 if/else 猜能力**：路由规则应来自测量并持续更新。
- **只锁模型，不锁 Prompt 与工具**：三者共同决定行为。
- **没有回滚**：厂商升级、限流和区域故障都会发生。

## 小结

可靠选型顺序是：定义业务任务 → 硬约束筛选 → 建立候选版本 → 真实数据评测 → 灰度发布 → 版本锁定与持续回归。最终选择是一个可验证、可降级、可回滚的系统配置，而不是一个品牌名称。
