训练一个现代大模型，不是把海量文本一次性“喂给 Transformer”就结束。它是一条从数据治理、预训练、评估到后训练和上线监控的循环流水线；每个阶段优化的目标、使用的数据和能解决的问题都不同。

## 先看全生命周期

![从数据预训练到后训练安全评估和部署监控的大模型生命周期](https://font-end-journey-resources.oss-cn-hangzhou.aliyuncs.com/images/llm-training-overview-lifecycle-v1.webp)
*图：预训练获得通用预测能力，后训练塑造交互行为；评估与线上反馈会反向推动数据和训练策略迭代。*

一个典型流程可以拆成九步：数据收集、清洗去重、tokenize、预训练、基础能力评估、监督微调、偏好优化、安全评估、部署与监控。真实团队会交叉迭代这些步骤，而不是严格只走一遍。

## 阶段一：定义目标与数据契约

训练前要先决定模型服务什么任务、覆盖哪些语言与代码、上下文多长、许可和隐私边界是什么。否则“数据越多越好”会把重复、低质、敏感信息和评测题答案一起扩大。

数据流水线通常包含：

- 来源登记、许可与使用限制；
- 语言和内容类型识别；
- 质量、恶意内容与个人信息过滤；
- 精确去重和近似去重；
- 数据混合比例与采样权重；
- 训练集、验证集及基准污染检查。

[Llama 3 技术报告](https://arxiv.org/abs/2407.21783) 把数据过滤、去重、混合和质量分类作为预训练的重要组成部分，也说明模型能力不只由参数数量决定。

## 阶段二：Tokenizer 与样本构造

Tokenizer 把原始文本变成 token ID。训练系统再把文档拼接或切分成固定长度样本，并构造 attention mask、位置编码和目标标签。

这里的选择会影响：多语言 token 效率、代码与空白保留、文档边界是否泄漏、长上下文利用率，以及训练中有多少 token 只是 padding。Tokenizer 一旦改变，embedding 和输出层的词表含义也随之改变，不能把旧 checkpoint 当成完全兼容。

## 阶段三：预训练

解码器式语言模型通常做 next-token prediction：给定前缀，最小化正确后续 token 的负对数似然。通过在大量样本上反复优化，模型学习语言结构、事实关联、代码模式和部分可迁移的任务能力。

[GPT-3](https://arxiv.org/abs/2005.14165) 展示了自回归预训练模型可通过提示和少量示例完成多种任务，而无需为每个任务更新参数。但预训练目标只要求预测文本，不天然要求遵循用户意图、拒绝危险请求或提供可验证来源。

大规模训练还需要分布式工程：数据并行切分样本，张量并行切分矩阵，流水线并行切分层，checkpoint 定期保存优化器与模型状态。吞吐、通信、数值稳定性和故障恢复都会影响最终可用训练量。

## 阶段四：预训练评估

训练 loss 下降只说明更会预测训练分布。团队还要检查：

- 保留验证集上的 loss 与困惑度；
- 语言、代码、数学、知识和长上下文等能力基准；
- 数据污染和记忆复现；
- 不同群体与语言上的性能差异；
- 毒性、隐私、网络安全和其他风险探针。

评估结果必须带版本、提示模板、解码参数和数据集版本。单个聚合分数无法替代真实业务任务的错误分析。

## 阶段五：监督微调（SFT）

预训练模型会续写文本，却不一定按“用户—助手”协议回答。SFT 使用高质量的指令—回答示范，继续做有监督的 token 预测，让模型学习任务格式、语气、工具调用结构和拒答范式。

SFT 的关键不是简单堆数量，而是覆盖行为边界并保持示范一致。矛盾示范会让模型学到不稳定策略；过窄数据则可能牺牲基础能力。

## 阶段六：偏好优化

同一提示通常存在多个“语法正确但质量不同”的回答。偏好数据以 chosen/rejected、排序或评分表达帮助性、真实性、风格和安全要求。

[InstructGPT](https://arxiv.org/abs/2203.02155) 的经典流程先用示范做 SFT，再训练奖励模型，最后用 RLHF 优化策略。现代流水线也可能使用 DPO 等直接偏好方法。它们塑造的是数据和评价标准定义的行为分布，不会自动产生客观、普适的价值函数。

## 阶段七：安全评估与发布门禁

上线前应使用独立于训练数据的测试集，覆盖越权、提示注入、危险建议、隐私泄露、偏见、过度拒答和工具误调用。红队发现的样本可以进入下一轮数据，但不能用“训练过这些题”代替保留集评估。

发布通常还需要比较新旧模型：核心任务是否提升、已知回归是否复现、延迟与成本是否可接受、量化或服务优化是否改变质量。

## 阶段八：部署与持续监控

生产中的输入分布会漂移，模型和工具也会升级。系统应记录模型版本、提示版本、工具轨迹、延迟、成本、失败类型和用户反馈，并对敏感数据做最小化与脱敏。

监控发现的问题可能回到不同环节：知识过时可优先修检索；格式失败可修 schema/后训练；基础能力缺口才可能需要继续预训练或新一代基座。不是每个线上问题都值得重新训练模型。

## 常见误区

- **训练 loss 更低就一定更好用**：交互行为、事实性和安全性需要单独评估。
- **后训练能注入无限新知识**：少量微调更适合塑造行为；频繁变化知识应外接检索。
- **benchmark 高就能直接上线**：业务分布、权限和副作用与公开题库不同。
- **安全只在最后过滤**：数据、目标、后训练、工具设计和运行时策略都参与安全。
- **训练完成就是项目完成**：真正生命周期还包括服务优化、监控、回滚和持续评估。

## 小结

大模型训练是一套闭环系统：预训练学习通用分布，SFT 学习示范行为，偏好优化调整选择倾向，安全评估与线上监控约束真实使用。把阶段和目标分开，才能针对问题选择正确的改进手段。
