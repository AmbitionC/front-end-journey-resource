点赞和点踩不是质量闭环，只是一个可能有偏的事件。很多满意用户不会点击，遇到严重问题的人可能直接离开；“复制答案”既可能表示有用，也可能是为了举报。要让反馈推动产品改进，必须把事件上下文、问题分类、人工标注、离线评测和灰度发布连成可验证链路。

## 先定义反馈事件

一个反馈事件不需要保存所有敏感内容，但要能回答“谁在什么任务、看到哪个版本、为什么反馈”：

```ts
type FeedbackEvent = {
  feedbackId: string;
  responseId: string;
  traceId: string;
  tenantId: string;
  taskType: string;
  signal: "up" | "down" | "correction" | "report";
  reason?: "incorrect" | "unsupported" | "unsafe" | "style" | "slow" | "other";
  correction?: string;
  modelVersion: string;
  promptVersion: string;
  retrievalVersion?: string;
  consent: { contentReview: boolean; trainingUse: boolean };
  createdAt: string;
};
```

`responseId` 连接当时的可复现快照；`reason` 比单一正负反馈更可行动；内容复核和训练使用分别取得授权。不要把邮箱、完整提示和模型回复默认复制进分析表。

## 组合六类信号

- **显式评价**：点赞、点踩、星级和原因码，解释清楚但覆盖人群有限；
- **用户修正**：用户给出更正文本或正确答案，信息密度高，仍需核验；
- **行为信号**：重试、改写、复制、继续追问、放弃，覆盖广但语义含糊；
- **业务结果**：问题是否解决、表单是否完成、人工是否接管，更接近目标但受其他因素影响；
- **支持与安全事件**：投诉、退款、越权、伤害和隐私报告，低频却高风险；
- **专家抽样**：按任务和风险分层的人工评审，用于发现“没有人点击”的系统性错误。

这些信号不能直接相加成一个“满意度”。先按任务、用户群、语言、风险和版本分层，再解释差异。

## 从反馈到问题分类

每日或每周聚合并去重相似事件，把问题归因到可修复层：检索没召回、证据错误、提示遗漏、工具失败、模型能力不足、界面误导、延迟或安全策略。一个低分可能同时涉及多层，允许多标签。

抽样策略同时包含：

- 随机样本，估计总体分布；
- 负反馈和高风险事件，快速处置；
- 版本/语言/租户等低覆盖切片；
- 新功能与分布漂移样本；
- 正反馈对照，避免只研究失败。

只看点踩会产生选择偏差，得到的修复可能优化“愿意点踩的人”，却伤害沉默的大多数。

## 人工标注需要量表与一致性

为不同任务定义可观察标准，例如事实正确、证据支持、任务完成、安全、格式和语气。不要只问“好不好”。高风险或主观样本由多人标注，定期检查分歧并更新说明；分歧本身可能揭示需求不清。

用户更正不能自动成为金标准：它可能带有误解、恶意内容或只适用于个人偏好。经过权限、隐私和事实核验后，才进入评测候选。

## 冻结评测集，再做候选修复

从确认问题中构建三层数据：

1. **冻结回归集**：长期不可随意改动，用于比较版本；
2. **新鲜挑战集**：反映近期变化和新失败；
3. **探索集**：用于诊断，不作为发布门槛。

[OpenAI Evals 指南](https://developers.openai.com/api/docs/guides/evals)强调围绕任务定义数据和评分。无论使用哪种工具，都应保存样本来源、授权、期望行为、评分器版本和人工说明。自动评分器先与人工校准，不能让同一模型在无监督情况下同时生成并裁决高风险答案。

候选修复可能是改检索、提示、工具、模型或 UI。一次尽量改变一个主要变量，在冻结集与关键切片上比较质量、延迟、费用、拒答和安全指标。

## 灰度发布与回滚

离线通过不等于线上有效。先向小比例流量灰度，使用稳定的分流单位，避免同一用户在新旧版本间跳动。提前定义主指标、护栏指标、最短观察期和停止条件；严重安全事件不等待统计显著性，直接回滚。

上线后把 `modelVersion`、`promptVersion`、`retrievalVersion` 和实验组写入 trace。否则新的反馈无法归因。若指标改善，只把通过审核的失败样本加入回归集，不把全部线上内容自动进入训练。

## 隐私、同意与治理

[NIST AI RMF](https://www.nist.gov/itl/ai-risk-management-framework) 的[核心框架](https://airc.nist.gov/airmf-resources/airmf/5-sec-core/)用 Govern、Map、Measure、Manage 组织持续风险管理，提醒反馈闭环不仅是模型指标，也包括责任、文档和生命周期治理。收集前说明用途，区分服务改进、人工复核与训练授权；对敏感内容做最小化、脱敏、访问控制、保留和删除。

租户合同或地区政策可能禁止内容离开业务系统。此时可以只保存聚合指标、错误类别和不可逆标识。用户撤回授权后，删除路径应覆盖反馈库、标注平台和派生数据登记。

## 常见误区

- 把点踩率直接当准确率；
- 把用户修正未经核验地当金标准；
- 只看负反馈，不做随机和分层抽样；
- 反馈表没有模型、提示、检索和实验版本；
- 用反馈样本调参后仍在同一批样本上报成绩；
- 离线提高后直接全量发布；
- 默认把所有会话内容用于人工复核或训练。

## 小结

有效反馈闭环把“一个按钮”扩展成可治理系统：结构化采集信号，控制隐私，分层抽样，人工确认问题，冻结评测，验证候选修复，再灰度、监控和回滚。反馈告诉团队去哪里调查，评测和实验才负责证明修复真的更好。
