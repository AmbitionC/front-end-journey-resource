TCP作为传输层的协议，**本身是面向连接的、可靠的流式传输协议，其设计目标是保证数据的可靠传输而非消息边界**。TCP会将数据流分割成多个数据段（segment），但并不会区分每个数据段属于哪个应用层的消息边界，这就可能导致接收端接收到的数据是在连续的几个TCP数据包中拼接而成的，这种现象就叫做TCP粘包（Packet Coalescing）。

#### TCP粘包产生的原因：
+ **Nagle算法**：为了减少网络中小包的数量，TCP栈可能合并小的数据包，形成较大的数据块一起发送。 
+ **TCP缓冲区**：发送端的数据可能积累到一定量才一次性发送，或者接收端来不及处理缓冲区的数据，导致多条消息在缓冲区中堆积，从而引起粘包。 
+ **数据包大小**：如果要发送的数据较小，可能不足以填满一个TCP数据段，TCP可以在一个数据段中包含多个消息。 



#### 解决TCP粘包的办法：
+ **禁用Nagle算法**：对于不需要合并小包的场景，可以关闭Nagle算法，通过设置`TCP_NODELAY`选项来禁用。 
+ **消息定界**： 
    - **自定义协议**：在应用层设计时加入消息边界标记，例如通过在每个消息前加上消息长度字段，或使用特定的结束符等，接收端根据这些标记来分割原始数据流。
    - **固定长度消息**：如果消息长度固定，可以直接按照固定长度来截取数据包。
    - **帧格式约定**：制定自定义协议时，可以规定特定的帧格式，包括头部信息和负载内容，通过头部信息来识别和分离每一个完整的消息。
+ **发送端控制**： 
    - **设置PUSH标志**：尽管PUSH标志主要用于提示接收端尽快将数据提交给应用层，但并不能从根本上解决粘包问题，但它可以帮助改善数据提交的时机。
    - **即时发送**：在发送重要边界的消息时，确保立即发送而不是累积，以此尽量避免粘包。
+ **接收端处理**： 
    - **缓冲区管理**：在接收端，程序需要有足够的逻辑去解析接收到的数据流，例如循环读取缓冲区直到遇到消息结束标记，并在每个消息处理完成后，继续处理缓冲区内的下一个消息。
