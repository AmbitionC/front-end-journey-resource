| 步骤 | 客户端行为 | 服务器行为 |
| :---: | --- | --- |
| 第一步 | 客户端发送一个FIN包给服务器，请求关闭连接 | 收到客户端的FIN包，进入CLOSE_WAIT状态 |
| 第二步 | - | 服务器向客户端发送一个ACK包，确认收到关闭请求 |
| 第三步 | - | 服务器发送一个FIN包给客户端，请求关闭连接 |
| 第四步 | 收到服务器的FIN包，向服务器发送一个ACK包 | 收到客户端的ACK包，连接关闭 |

<font style="color:#8A8F8D;">TCP三次握手和四次挥手流程</font>

TCP四次挥手过程是TCP连接断开时所遵循的一种机制，用来确保连接双方都能够顺利关闭连接，并且清理资源。四次挥手的具体步骤如下：

#### 第一次挥手（Fin from client）
    - 客户端（A）完成数据传输任务后，准备关闭连接，它会向服务器（B）发送一个特殊的TCP报文段，该报文段中的`FIN`（Finish）标志位被置为1，同时带有序列号seq=x。客户端进入`FIN_WAIT_1`状态，表明客户端已经没有数据要发送，但是还可以接收从服务器传来的数据。

```plain
Client (A) -> Server (B): FIN = 1, ACK = 1, SEQ = x, ACK_NUM = y
```



#### 第二次挥手（Ack from server）
    - 服务器（B）接收到客户端的`FIN`报文段后，确认收到这个消息，回复一个确认报文段，`ACK`标志位被置为1，确认号`ACK_NUM`设置为客户端的序列号加1（即x+1），同时服务器可能还有剩余数据需要发送，因此仍处于`ESTABLISHED`状态。客户端接收到这个确认后，进入`FIN_WAIT_2`状态，等待服务器关闭它的数据流向。

```plain
Server (B) -> Client (A): ACK = 1, SEQ = y, ACK_NUM = x + 1
```



#### 第三次挥手（Fin from server）： 
    - 服务器（B）在发送完所有待发送的数据后，也准备关闭连接，它会向客户端（A）发送一个`FIN`标志位为1的报文段，带有序列号seq=z。服务器进入`LAST_ACK`状态，等待客户端的最终确认。

```plain
Server (B) -> Client (A): FIN = 1, ACK = 1, SEQ = z, ACK_NUM = x + 1
```



#### （4） 第四次挥手（Final Ack from client）： 
    - 客户端（A）收到服务器的`FIN`报文段后，发送最后一个确认报文段，`ACK`标志位依然为1，确认号`ACK_NUM`设置为服务器的序列号加1（即z+1）。此时，客户端彻底关闭连接进入`TIME_WAIT`状态，等待足够长的时间以确保服务器收到了最后的确认报文，并且可以处理网络中残留的报文段。而服务器收到这个确认后，则会关闭连接，进入`CLOSED`状态。

```plain
Client (A) -> Server (B): ACK = 1, SEQ = x + 1, ACK_NUM = z + 1
```



在这个过程中，客户端在发送最后的确认报文后通常会保持`TIME_WAIT`状态一段时间（2MSL，即两倍的最大段生存时间），确保在最后的ACK报文丢失时可以重传，同时等待可能出现的迟到的报文段，直到这段时间过去后才会关闭连接。这样能够保证TCP连接在两端都已经被可靠地关闭。
