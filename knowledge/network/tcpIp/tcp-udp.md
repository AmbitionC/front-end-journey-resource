TCP 与 UDP 提供不同的传输语义，不是简单的“可靠但慢”和“不可靠但快”。正确选型要回答：应用传递的是连续字节还是独立消息、过期数据是否还有价值、谁负责重传与拥塞控制，以及能否承受乱序和重复。

## TCP 提供有序字节流

[RFC 9293](https://www.rfc-editor.org/rfc/rfc9293.html)定义 TCP 为面向连接的可靠有序字节流。应用写入的多次数据可能被合并或拆分，接收端不能依赖一次 `send` 对应一次 `recv`，必须自己定义长度前缀、分隔符或固定帧格式。

TCP 为字节编号，使用累计 ACK、重传、接收窗口和拥塞控制。若较早的字节丢失，应用读取有序流时会等待缺口恢复；这给文件、HTTP 响应和数据库协议带来简单的完整性语义，但对“旧帧不如新帧”的实时场景可能形成队头阻塞。

连接状态还带来握手、关闭、keep-alive 和资源管理成本。成本是否显著取决于连接复用、RTT、TLS 与业务模式，不能脱离场景宣称 TCP 一定更慢。

## UDP 提供独立数据报

[RFC 768](https://www.rfc-editor.org/rfc/rfc768.html)规定 UDP 首部包含源端口、目的端口、长度和校验和。一次发送对应一个 datagram，接收端能够观察消息边界；网络却不保证到达、顺序或唯一性。报文过大还可能触发 IP 分片或超过路径 MTU，因此实时协议通常限制单个 datagram 大小。

UDP 并非“没有校验”。IPv4 中校验和字段为零表示发送方未计算校验和；非零值覆盖伪首部、UDP 首部和数据。[RFC 8200](https://www.rfc-editor.org/rfc/rfc8200.html)进一步规定 IPv6 默认必须计算 UDP 校验和，接收方通常丢弃零校验和报文，只有少数隧道场景可按额外规则例外处理。

UDP 本身没有内建重传、排序、流量控制或拥塞控制。若应用需要这些能力，就必须采用成熟上层协议，或明确实现消息 ID、确认、超时、退避、去重、乱序缓冲与发送速率控制。

## 可靠性可以构建在 UDP 之上

QUIC 使用 UDP 承载，却自行提供加密握手、可靠流、丢包恢复和拥塞控制。RTP/WebRTC 也会结合序列号、抖动缓冲、反馈与编解码容错。它们说明底层 datagram 只是接口选择，最终服务语义由完整协议栈决定。

相反，某些 UDP 请求/响应即使很短也需要重试和验证。DNS 在 UDP 响应被截断、超时或场景要求时可使用 TCP；不能把“DNS 使用 UDP”概括为永不建连。

## 选择问题清单

| 约束 | 更适合可靠字节流 | 更适合 datagram / 基于 UDP 的成熟协议 |
| --- | --- | --- |
| 必须完整、有序交付所有内容 | TCP | 上层需自行提供 |
| 旧数据过期后宁可丢弃 | 可能造成等待 | 可按消息独立丢弃 |
| 必须保留消息边界 | 应用自行 framing | UDP 天然保留 |
| 需要多条相互独立的流 | 多连接或应用复用 | QUIC 等协议可提供 |
| 团队能否实现拥塞与安全细节 | 直接使用成熟 TCP 协议 | 不应手写“可靠 UDP” |

选择前还要考虑 NAT 映射、企业网络对 UDP 的限制、移动网络切换、负载均衡状态、观测工具与库生态。协议在实验室的单次延迟不能替代真实网络下的丢包和拥塞测试。

## 一个最小数据报接收契约

业务 datagram 至少应包含版本、消息类型、message ID、有效载荷长度和可选过期时间。接收端先校验长度与版本，再按 `(peer, messageId)` 去重；需要响应时限制放大倍数，避免源地址伪造造成反射攻击。重试使用指数退避和总期限，不能在丢包时无限加速发送。

测试应注入丢包、重复、乱序、延迟尖峰和 MTU 变化，验证业务结果，而不只是测“平均延迟”。如果无法写清这些条件，优先采用成熟协议或 TCP。

## 小结

TCP 的核心是连接状态下的可靠有序字节流；UDP 的核心是保留边界的独立数据报。UDP 有长度与校验和，但没有交付、排序和拥塞保证。性能来自完整协议和网络条件，选型应围绕应用容错契约，而不是“快/慢”的绝对标签。
