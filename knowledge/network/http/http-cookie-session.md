**Cookie与Session之间的区别主要体现在以下几个方面：**

| 特征 | Cookie | Session |
| :---: | --- | --- |
| 存储位置 | 存储在客户端（浏览器） | 存储在服务器端 |
| 安全性 | 相对较低，可以被用户修改 | 相对较高，用户无法直接访问 |
| 存储容量 | 有限制，通常为4KB左右 | 理论上无限制 |
| 生命周期 | 可以设置过期时间 | 通常与用户会话相关联 |
| 用途 | 用于跟踪用户行为、会话管理 | 用于保持用户状态、会话管理 |


#### 存储位置 
    - **Cookie**：存储在客户端（通常是用户的浏览器），以文本文件形式保存在用户电脑上。
    - **Session**：存储在服务器端，不在用户的设备上保留任何信息，而是服务器为每个客户端维护一份单独的数据区域。



#### 数据类型与大小 
    - **Cookie**：通常存储为简单的键值对字符串，大小受到浏览器限制，一般为4KB左右。
    - **Session**：可以存储任意类型的数据（包括对象），存储大小受服务器配置限制。



#### 生命周期 
    - **Cookie**：具有明确的过期时间，可设置为会话级（浏览器关闭时清除）或持久化（到达特定日期后清除）。
    - **Session**：依赖于服务器的会话管理机制，用户关闭浏览器并不直接导致Session结束，但服务器可以根据会话超时或显式销毁策略来结束Session。



#### 安全性 
    - **Cookie**：数据容易暴露，因为存储在客户端，可能遭到窃取、篡改或伪造。
    - **Session**：相对更安全，因为数据存储在服务器端，不直接暴露给客户端，但Session ID可能通过Cookie传递，因此也需要采取措施防止Session ID被盗用。



#### 性能影响 
    - **Cookie**：由于数据存储在客户端，每次请求都会带上Cookie，对带宽有一定消耗，但不会直接影响服务器资源。
    - **Session**：服务器需要维持每个Session的数据结构，随着在线用户数量的增长，可能会显著增加服务器内存消耗和数据库负载。



#### 依赖性 
    - **Session**：尽管Session机制通常基于Cookie来实现，即通过Cookie传递唯一的Session ID，但也可以通过URL重写等技术绕开Cookie，不过这种方式对用户体验和安全性都有一定影响。



#### 用户体验 
    - **Cookie**：可以在客户端缓存用户信息，无需每次都与服务器交互，提升页面加载速度。
    - **Session**：每次请求都需要验证Session ID的有效性，如果服务器不再维护该Session，用户可能需要重新登录或重新初始化会话状态。
