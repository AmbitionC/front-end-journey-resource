# MySQL 索引为什么用 B+ 树?

## 从一次"全表扫描"说起

假设你有一张用户表 `users`,里面存了 500 万行数据。现在要执行一条很普通的查询:

```sql
SELECT * FROM users WHERE id = 3829100;
```

如果这张表上没有任何索引,MySQL 只能做一件事:**从磁盘里把数据一页一页读出来,逐行比对 `id` 是不是 3829100**。这就是"全表扫描"(full table scan)。500 万行可能要读几万个磁盘页,即使命中的那一行在最后,也得把前面全部扫一遍。对单条查询而言,这意味着几百毫秒甚至更久的耗时。

而如果 `id` 上有索引,同样的查询往往只需要读 3~4 个磁盘页就能定位到那一行——耗时从几百毫秒降到亚毫秒级。

**索引解决的核心问题就是:把"逐行扫描"变成"按结构快速定位"。** 它本质上是一种为了加速查找而额外维护的有序数据结构。代价是写入时要同步维护索引、占用额外存储,但换来的是查询性能上的数量级提升。

> 对 Agent 工程的意义:Agent 在做检索增强(RAG)或调用工具查库时,底层往往就是一条 `WHERE` 查询。理解索引能让你写出走索引的 SQL,避免 Agent 在高频调用下被慢查询拖垮——这和向量库里"建 HNSW 索引而不是暴力遍历"是同一种思维。

下面的问题是:既然要建有序结构,数据结构那么多,为什么 MySQL(InnoDB)偏偏选了 **B+ 树**?我们用排除法一个个看。

## 为什么不用哈希索引?

哈希表的查找是 O(1),听起来比树还快。事实上 MySQL 的 Memory 引擎、以及 InnoDB 的"自适应哈希索引"确实用到了哈希。但它**不能作为通用索引结构**,原因有两个致命缺陷:

| 能力 | 哈希索引 | B+ 树索引 |
| --- | --- | --- |
| 等值查询 `= ` | 快(O(1)) | 快(O(log n)) |
| 范围查询 `>` `<` `BETWEEN` | **不支持** | 支持 |
| 排序 `ORDER BY` | **不支持** | 天然有序 |
| 最左前缀匹配 | 不支持 | 支持 |

哈希把 key 打散成一个无规律的哈希值存放,所以它**丢失了顺序信息**。下面这类极其常见的查询,哈希索引完全无能为力:

```sql
SELECT * FROM orders WHERE created_at > '2026-01-01' ORDER BY created_at;
```

而实际业务里,范围查询和排序占了相当大的比例。一个连范围和排序都支持不了的结构,显然当不了主力。

## 为什么不用二叉搜索树 / 红黑树?

二叉搜索树(BST)、红黑树、AVL 树这些**平衡二叉树**是有序的,既支持等值也支持范围查询,看起来满足需求。问题出在它们和**磁盘**的配合上。

要点是:索引数据量很大,放不进内存,必须存在磁盘上;**磁盘 IO 是整个查询里最慢的环节**,比内存计算慢几个数量级。所以索引结构优化的真正目标,不是减少 CPU 比较次数,而是 **减少磁盘 IO 次数**。

二叉树每个节点最多两个子节点,树的高度大致是 `log₂(N)`。对 500 万行数据:

```
log₂(5,000,000) ≈ 22
```

也就是说树高约 22 层。最坏情况下,一次查找要从根走到叶,经过约 22 个节点。而由于树节点在磁盘上是分散存放的,**每访问一个节点就可能触发一次随机磁盘 IO**——一次查询要 20 多次随机 IO,这太慢了。

红黑树虽然保证了大致平衡,但它"瘦高"的形态没变:每个节点只存一个 key、只有两个分支。问题的根源在于**叉太少,导致树太高**。要降低 IO 次数,就得让树变"矮胖"——这正是 B 树和 B+ 树的思路。

## B 树:让每个节点"装更多"

B 树(B-Tree)的核心改进是:**一个节点不再只存一个 key,而是存很多个 key 和多个分支**。一个节点的大小正好对应一个磁盘页(InnoDB 默认 16KB)。

为什么这样能变矮?因为分叉数从 2 变成了几百甚至上千。树高从 `log₂(N)` 变成 `log_m(N)`(m 是每个节点的分叉数)。m 越大,同样的数据量树就越矮,IO 次数就越少。

但 B 树有个问题:**它的每个节点(包括非叶子节点)都存放完整的数据行**。一个磁盘页只有 16KB,如果非叶子节点也塞满了数据行,那它能容纳的 key 数量就少了,分叉数随之下降,树又会被迫变高。同时,B 树的范围查询也不方便——你得在树里来回上下跳跃才能遍历一段区间。

## B+ 树:MySQL 的最终选择

B+ 树是对 B 树的进一步优化,有两个关键设计:

### 1. 非叶子节点只存索引,不存数据 → 树更矮胖

B+ 树规定:**所有真实的数据(或数据地址)只存在叶子节点,非叶子节点只存 key 和指向下层的指针。**

这样一来,非叶子节点不用为数据行浪费空间,一个 16KB 的页能塞下非常多的 key,分叉数大大提高。我们粗略算一下:

- 假设一个非叶子节点(一页 16KB)存索引项,每个索引项 = 主键(bigint,8 字节)+ 页指针(6 字节)≈ 14 字节;
- 一页能存约 `16 * 1024 / 14 ≈ 1170` 个索引项,即每个非叶子节点有约 **1170 个分叉**;
- 叶子节点存完整数据行,假设一行约 1KB,那一个叶子页能存约 **16 行**。

于是:

| 树高 | 能容纳的行数(粗略) |
| --- | --- |
| 2 层(1 个根 + 叶子) | 1170 × 16 ≈ 1.8 万行 |
| 3 层(根 + 中间 + 叶子) | 1170 × 1170 × 16 ≈ **2190 万行** |

**结论:一棵三层的 B+ 树,就能存下两千多万行数据。** 而查找任意一行,最多只需要读 3 个页——其中根节点和中间节点通常常驻内存,真正的磁盘 IO 往往只有 1~2 次。这就是 B+ 树相对二叉树(22 次 IO)的碾压性优势。

### 2. 叶子节点用双向链表串联 → 范围扫描友好

B+ 树的另一个设计是:**所有叶子节点按 key 顺序用双向链表连起来**。

```
[非叶子] ──> 只存索引,负责导航
   │
[叶子] 10,20,30 <──> 40,50,60 <──> 70,80,90   (双向链表)
```

这带来一个巨大的好处:做范围查询 `WHERE id BETWEEN 30 AND 70` 时,只要先定位到 30 所在的叶子,然后**沿着链表顺序向右扫描**即可,不用再回到上层节点跳来跳去。`ORDER BY` 也因为叶子天然有序而无需额外排序。这正好补上了 B 树范围查询不便、哈希索引完全不支持范围的短板。

把整张图拼起来,B+ 树同时满足了:**矮(IO 少)+ 有序(支持范围和排序)+ 叶子链表(扫描高效)**,这是哈希、二叉树、B 树都做不到的组合。

## 聚簇索引、二级索引与回表

理解了结构,再看 InnoDB 怎么用它来组织数据。

**聚簇索引(Clustered Index)**:InnoDB 的表数据**本身就是按主键组织成一棵 B+ 树**的。也就是说,主键索引的叶子节点里存的不是"指向数据的地址",而是**整行完整数据**。所以 InnoDB 的表又叫"索引组织表"——数据和主键索引是一体的。每张表有且只有一个聚簇索引。

**二级索引(Secondary Index)**:你在其他列上建的索引(比如 `name`、`email`)都是二级索引,它们也是 B+ 树,但叶子节点存的不是整行数据,而是**该行的主键值**。

**回表(Back to Table)**:于是当你用二级索引查询时,会发生两步:

```sql
SELECT * FROM users WHERE name = '张三';
```

1. 先在 `name` 的二级索引树里找到 `'张三'`,拿到它对应的主键 `id`;
2. 再拿这个 `id` 回到**聚簇索引**树里,查出完整的行数据。

第二步"拿着主键再查一次主键索引"就叫**回表**,意味着走了两棵 B+ 树。如果查询只需要的列恰好都在二级索引里(比如 `SELECT id, name FROM users WHERE name='张三'`),就能"覆盖索引",省掉回表,这也是优化慢查询的常用手段。

## 为什么主键建议用自增整型?

理解了聚簇索引,这个建议就顺理成章了:

**1. 整型比较快、占用小。** 主键会被所有二级索引的叶子节点引用,主键越小,二级索引就越省空间。整型(尤其 bigint)比较运算也比字符串(如 UUID 字符串)快得多。

**2. 自增能避免"页分裂",保证顺序写入。** 这是最关键的原因。聚簇索引按主键有序排列,新插入的行必须放到主键对应的位置:

- 如果主键**自增**,新行总是追加到当前最后一个叶子页的末尾,写入是顺序的,页填满了就开新页,干净利落;
- 如果主键**随机**(比如用 UUID 或随机数),新行可能要插到某个已经填满的中间页里,这会迫使该页**分裂**(page split)——把一半数据挪到新页,既产生额外 IO,又造成页内空间碎片,降低空间利用率和性能。

```sql
-- 推荐:自增整型主键
CREATE TABLE users (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(64),
  email VARCHAR(128)
);
```

所以"自增整型主键"不是教条,而是对聚簇索引 B+ 树结构特性的顺应:**让写入尽量顺序、让索引尽量紧凑。**

## 小结

我们用排除法走了一遍 MySQL 的选择逻辑:

- **哈希索引**——不支持范围查询和排序,出局;
- **二叉/红黑树**——叉太少导致树太高,随机磁盘 IO 太多,出局;
- **B 树**——非叶子节点存数据,导致分叉少、树偏高,且范围查询不便;
- **B+ 树**——非叶子只存索引让树矮胖(三层即可存两千多万行,查找仅 1~3 次 IO),叶子双向链表让范围扫描和排序高效,完美契合"磁盘 IO 是瓶颈"这一前提。

再叠加 InnoDB 的聚簇索引(主键即数据)、二级索引与回表机制,以及"自增整型主键利于顺序写入、减少页分裂"的实践,整个索引体系的设计动机就连成了一条逻辑链。

下次写 SQL 时,不妨想想:这条查询会走哪棵 B+ 树?会不会回表?有没有走索引?——理解了"为什么",这些判断就会变成本能。
