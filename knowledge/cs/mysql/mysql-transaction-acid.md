# 事务的 ACID 特性是如何保证的?

假设你在银行 App 上给朋友转账 100 元。系统要做两件事:**从你的账户扣 100**,**给朋友的账户加 100**。如果扣款成功后服务器突然宕机,加款还没执行,会发生什么?你的钱凭空消失了 100 元。这显然是不可接受的。

为了让这种"要么全做、要么全不做"的操作变得可靠,数据库引入了**事务(Transaction)**这一抽象。而衡量一个事务是否"靠谱",业界总结出了四条性质:**ACID**。本文不只罗列定义,而是讲清楚每一条性质底层到底"靠什么"来保证——这正是面试和实战中最容易含糊的地方。

```sql
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1; -- 你
UPDATE account SET balance = balance + 100 WHERE id = 2; -- 朋友
COMMIT;
```

## ACID 四性:先把定义说清楚

| 特性 | 英文 | 核心含义 |
| --- | --- | --- |
| 原子性 | Atomicity | 事务内的操作要么全部成功,要么全部回滚,不存在"做了一半"的中间态 |
| 一致性 | Consistency | 事务执行前后,数据始终满足业务约束(账户总额不变、余额非负等) |
| 隔离性 | Isolation | 多个并发事务互不干扰,彼此感受不到对方的中间状态 |
| 持久性 | Durability | 事务一旦提交,其结果就永久落地,即使数据库崩溃也不丢失 |

这四条里有一个常被误解的关键点:**一致性是"目的",其余三性是"手段"**。原子性、隔离性、持久性合力工作,再加上业务层面的约束(如非空、外键、余额非负),最终才达成一致性。后面会专门解释这个关系。

下面逐条拆解 InnoDB(MySQL 默认存储引擎)是如何实现它们的。

## 原子性:靠 undo log 实现回滚

原子性要解决的问题是:执行到一半失败了,怎么把已经做的操作"撤销"掉?

InnoDB 的答案是 **undo log(回滚日志)**。在修改任何一条记录之前,InnoDB 会先把这条记录**修改前的旧值**记录到 undo log 中。

- 如果你执行 `UPDATE balance = balance - 100`,undo log 里会记下一条"反操作"——把 balance 加回 100。
- 如果是 `INSERT`,undo log 记录对应的 `DELETE`。
- 如果是 `DELETE`,undo log 记录对应的 `INSERT`(旧数据)。

这样一来,无论事务因为什么原因需要回滚(主动 `ROLLBACK`、SQL 报错、宕机),InnoDB 都能**沿着 undo log 反向执行**,把数据恢复到事务开始前的样子:

```sql
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1; -- undo: +100
-- 此处程序异常 / 主动 ROLLBACK
ROLLBACK; -- 依据 undo log 把 id=1 的余额加回 100
```

所以记住一句话:**原子性的本质是"可撤销",而撤销能力来自 undo log。** 顺带一提,undo log 不仅服务于回滚,它保存的"旧版本数据"还会被 MVCC 复用来实现一致性读,这一点我们在隔离性部分会再次提到。

## 持久性:靠 redo log + WAL 实现

持久性要解决的问题正好相反:事务**提交后**,怎么保证结果不丢?

最朴素的想法是——提交时直接把修改后的数据页刷到磁盘。但这有两个致命问题:

1. **慢**。InnoDB 以"页"(默认 16KB)为单位管理数据,一次小小的余额修改可能只改了几个字节,却要把整页随机写回磁盘,而磁盘随机 IO 极慢。
2. **不安全**。如果刷盘进行到一半就宕机,数据页会处于"写坏"的中间状态。

InnoDB 的解决方案是 **WAL(Write-Ahead Logging,预写日志)**:**先写日志,再写数据**。

具体来说,事务提交时,InnoDB 并不急着把数据页刷盘,而是先把"这次改了什么"以 **redo log(重做日志)** 的形式顺序追加写入磁盘。redo log 记录的是物理层面的修改(某个页的某个偏移量改成了什么值),写入是**顺序 IO**,远快于随机 IO。

```text
事务提交流程(简化):
1. 在内存(Buffer Pool)中修改数据页 —— 此时是脏页,还没落盘
2. 把修改写入 redo log,并 fsync 落盘  ←── 提交成功的关键
3. (稍后由后台线程)在合适时机把脏页慢慢刷回磁盘
```

关键在于第 2 步:只要 redo log 落盘成功,事务就可以安全地返回"提交成功"。即使紧接着宕机、内存中的脏页还没来得及刷回磁盘,也不要紧——**重启时,InnoDB 会扫描 redo log,把还没落盘的修改重新"重放"一遍**,这就是**崩溃恢复(crash recovery)**。

这里有个常被混淆的点,正好对照理解两种日志:

| 日志 | 记录内容 | 作用 | 服务的特性 |
| --- | --- | --- | --- |
| undo log | 数据的旧值(反操作) | 回滚未提交的事务、提供历史版本 | 原子性、隔离性(MVCC) |
| redo log | 数据的物理新值 | 重放已提交但未落盘的修改 | 持久性 |

一句话:**redo log 保证"已提交的不会丢",undo log 保证"没提交的不留痕"。** 崩溃恢复时两者配合——先用 redo log 重放所有记录,再用 undo log 回滚那些重放后发现尚未提交的事务。

## 隔离性:靠锁 + MVCC 实现

隔离性要解决的问题是:多个事务并发执行时,如何避免它们互相"看到对方的脏数据"或互相覆盖?

如果完全串行执行,隔离性自然完美,但性能极差。InnoDB 用**两套机制的组合**在性能和正确性之间取得平衡:**锁(Locking)** 和 **MVCC(多版本并发控制)**。

### 锁:解决写写冲突

当两个事务同时想修改同一行时,必须排队。InnoDB 用**行级锁**实现:

```sql
-- 事务 A
UPDATE account SET balance = balance - 100 WHERE id = 1; -- 给 id=1 这行加排他锁(X 锁)
-- 事务 B 此时也想改 id=1,必须等待 A 提交或回滚后释放锁
```

写操作会加**排他锁**,确保同一时刻只有一个事务能修改同一行,从而避免**更新丢失**。此外 InnoDB 还有间隙锁(Gap Lock)、临键锁(Next-Key Lock)等,用来在可重复读级别下防止幻读。

### MVCC:解决读写冲突,实现"读不加锁"

如果读操作也要等写锁释放,那读多写少的系统会被严重拖慢。MVCC 的思路是:**让读操作读取数据的"历史快照",而不是去争抢锁**。

它依赖三样东西:

- 每行记录隐藏的版本信息(事务 ID `DB_TRX_ID`、回滚指针 `DB_ROLL_PTR`);
- 前面提到的 **undo log**——旧版本数据就存在这里,通过回滚指针串成一条"版本链";
- **Read View**——事务在某一刻生成的"可见性快照",用来判断版本链上哪个版本对当前事务可见。

于是,一个事务的普通 `SELECT` 读到的是符合自己 Read View 的那个版本,**完全不阻塞其他事务的写入**,反之亦然。这就是 InnoDB 在"读已提交"和"可重复读"级别下高并发的秘密。

至于不同隔离级别(读未提交 / 读已提交 / 可重复读 / 串行化)下,Read View 的生成时机不同、能防住哪些并发问题(脏读、不可重复读、幻读),内容较多,我们留到下一篇《隔离级别与 MVCC》专门展开。

## 一致性:目标,由前三者 + 业务约束共同达成

讲完前三性,回头看一致性就清楚了:**一致性不是一种独立的底层机制,而是其他保障共同作用的结果。**

以转账为例,"一致"意味着:无论怎么转,**两个账户的总额始终不变**,且**没有人的余额变成负数**。要守住这条规则,需要:

- **原子性**保证"扣款"和"加款"不会只发生一半,总额不会凭空增减;
- **隔离性**保证两笔并发转账不会互相覆盖,导致总额对不上;
- **持久性**保证转账成功后结果不会因宕机而回退;
- **业务/约束层**保证语义正确——比如 `CHECK (balance >= 0)`、应用代码里的余额校验、外键约束等。

注意最后一点:即使 AID 三性都完美,如果你写了一条 `UPDATE balance = balance - 100`(扣了款却忘了给对方加),数据库照样会忠实执行,事务也能提交——但数据已经不一致了。**所以一致性最终还需要开发者保证业务逻辑本身正确。** 数据库只能保证"你定义的规则不被并发和故障破坏",不能保证"你定义的规则本身合理"。

## 四种机制如何协作:把流程串起来

把一次完整的转账事务从头到尾走一遍,你会看到四套机制是如何咬合的:

1. `START TRANSACTION` 开启事务。
2. 修改 id=1 余额前,先写 **undo log**(为原子性/MVCC 留后路),并加**行锁**(为隔离性)。
3. 在 Buffer Pool 中改数据页,同时记录 **redo log**(为持久性)。
4. 修改 id=2,重复上述步骤。
5. `COMMIT`:把 redo log **fsync 落盘**(WAL),提交即生效。脏页稍后由后台异步刷盘。
6. 期间若失败 → 用 undo log **回滚**;若宕机重启 → 用 redo log **重放** + undo log 回滚未提交事务。
7. 全程账户总额守恒、余额非负 → **一致性**达成。

这套"日志先行、锁与多版本并行、崩溃可恢复"的设计,本质上是工程上反复出现的可靠性范式。它的价值并不局限于数据库:在 **AI Agent 工程**里,一个 Agent 往往要执行多步操作(查库存 → 下单 → 扣款 → 发通知),如果中途某一步失败,我们同样需要"要么整体生效、要么整体回退"的能力——这就是为什么越来越多的 Agent 框架引入了类事务的 checkpoint、补偿(Saga)和幂等设计。理解 ACID,本质是理解"如何在不可靠的世界里构造可靠的多步操作",这种思维迁移到分布式系统和 Agent 编排中同样适用。

## 小结与后续

- **原子性** ← undo log(回滚 / 反操作)
- **持久性** ← redo log + WAL(先写日志、崩溃重放)
- **隔离性** ← 锁(写写冲突) + MVCC(读写冲突,基于 undo log 版本链 + Read View)
- **一致性** ← 目标,由以上三性 + 业务约束共同达成

可以看到,**undo log 和 redo log 是支撑整个事务体系的两根支柱**。如果你想继续深入,推荐顺着这两条线往下读:

- 《隔离级别与 MVCC》——脏读、不可重复读、幻读到底怎么发生,Read View 如何在不同级别下精确控制可见性;
- 《redo log / undo log / binlog》——三种日志的分工、两阶段提交如何保证 redo log 与 binlog 的一致,以及它们在主从复制中的角色。

把这三篇连起来读,你对 MySQL 事务的理解就能从"背定义"升级到"讲清楚为什么"。
