# 如何优化索引?最左前缀、覆盖索引与索引失效

某天凌晨,监控告警把你叫醒:一个原本几十毫秒返回的订单列表接口,突然慢到 3 秒以上,数据库 CPU 被打满。你登上服务器,把那条 SQL 拎出来跑了一遍,`EXPLAIN` 一看——明明表上有索引,可执行计划里赫然写着 `type=ALL`,扫描了上百万行。索引"在",却没"生效"。

这是后端工程师最常遇到的一类问题。本文不堆术语,而是把 MySQL(以最常见的 InnoDB 引擎、B+Tree 索引为背景)的索引优化讲清楚:**索引为什么这样设计、什么时候会失效、以及你该怎么读执行计划做判断**。理解这些之后,再看 `EXPLAIN` 就像医生看 X 光片一样有章可循。

## 先建立一个心智模型:B+Tree 与联合索引

InnoDB 的索引是 B+Tree:数据按索引列的值**有序**地组织在树的叶子节点上。理解一切优化的前提,就是记住"有序"两个字。

对于**联合索引**(也叫复合索引),比如 `INDEX idx_abc(a, b, c)`,它的排序规则是:先按 `a` 排,`a` 相同的再按 `b` 排,`b` 也相同的再按 `c` 排。就像电话簿先按姓排、姓相同按名排。这个排序方式直接决定了下面要讲的"最左前缀"。

## 最左前缀原则:为什么顺序这么重要

**最左前缀原则**是说:使用联合索引时,查询条件必须从索引的最左列开始,且不能跳过中间的列,索引才能被有效利用。

以 `idx_abc(a, b, c)` 为例:

| 查询条件 | 能用到索引的部分 | 原因 |
| --- | --- | --- |
| `WHERE a=1` | a | 从最左列开始 |
| `WHERE a=1 AND b=2` | a, b | 连续从左开始 |
| `WHERE a=1 AND b=2 AND c=3` | a, b, c | 完整命中 |
| `WHERE b=2 AND c=3` | 都用不上 | 跳过了最左列 a |
| `WHERE a=1 AND c=3` | 只有 a | 跳过了 b,c 无法定位 |

为什么跳过 `b` 后 `c` 就用不上?回到"电话簿"的比喻:整本簿子在"姓"相同的范围内才按"名"有序,但如果你只知道名不知道姓,整本簿子里的名是无序散落的,只能逐页翻。索引里的 `c` 同理——只有在 `a`、`b` 都确定的区间内,`c` 才是有序的。

### 范围列之后的列会失效

这是最容易踩的坑。**当某一列使用了范围查询(`>`、`<`、`BETWEEN`、`LIKE 'xx%'`)后,它右边的列无法再用索引排序定位。**

```sql
-- idx_abc(a, b, c)
SELECT * FROM t WHERE a = 1 AND b > 100 AND c = 5;
```

这里 `a` 等值、`b` 范围,索引能用到 `a` 和 `b`;但 `b > 100` 命中的是一个区间,这个区间内 `c` 是无序的,所以 `c = 5` 用不到索引,只能在结果里逐行过滤(后面会讲 ICP 对此的优化)。

**实战启示**:设计联合索引时,把等值查询的列放左边,范围查询的列放右边。如果业务上 `b` 总是范围、`c` 总是等值,那把索引建成 `(a, c, b)` 往往更优。

## 覆盖索引:避免回表

InnoDB 有两类索引:**聚簇索引**(主键索引,叶子节点存整行数据)和**二级索引**(普通索引,叶子节点只存索引列的值 + 主键值)。

走二级索引查询时,如果 `SELECT` 需要的列不在索引里,MySQL 拿到主键后还得再去聚簇索引查一次完整行——这个过程叫**回表**。回表是随机 IO,数据量大时代价很高。

**覆盖索引**指的是:一个查询所需的所有列,都能从某个索引中直接拿到,不需要回表。

```sql
-- 假设有 idx_name_age(name, age)
SELECT name, age FROM users WHERE name = '张三';   -- 覆盖,无需回表
SELECT * FROM users WHERE name = '张三';            -- 需要回表取其他列
```

在 `EXPLAIN` 的 `Extra` 列出现 **`Using index`**,就说明用上了覆盖索引(注意别和 `Using index condition` 混淆,后者是索引下推)。覆盖索引是性价比极高的优化手段:很多时候只要把高频查询里 `SELECT` 的几个小字段补进联合索引,就能消掉大量回表。

## 索引下推(ICP):把过滤下沉到存储引擎

**索引下推(Index Condition Pushdown,ICP)** 是 MySQL 5.6 引入的优化。回到前面"范围列之后失效"的例子:

```sql
-- idx_name_age(name, age)
SELECT * FROM users WHERE name LIKE '张%' AND age = 20;
```

`name LIKE '张%'` 是范围,后面的 `age` 无法用索引定位。在没有 ICP 的旧版本里,存储引擎把所有 `name` 以"张"开头的记录全部回表,再交给 Server 层用 `age = 20` 过滤,回表次数很多。

开启 ICP 后:虽然 `age` 不能用于树的定位,但它的值就**存在索引里**。存储引擎在遍历索引时顺手用 `age = 20` 先过滤一遍,只把真正满足条件的记录才回表。回表次数大幅减少。

`EXPLAIN` 的 `Extra` 里出现 **`Using index condition`** 就表示触发了 ICP。

## 索引失效的常见场景

下面这些写法会让"明明建了索引"却走全表扫描,逐一看清楚**为什么**。

### 1. 对索引列使用函数或运算

```sql
-- 失效:对列做了运算/函数,索引存的是 create_time 原值,不是 YEAR(create_time)
SELECT * FROM orders WHERE YEAR(create_time) = 2024;
SELECT * FROM orders WHERE id + 1 = 100;

-- 改写:让列保持"裸露"
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
SELECT * FROM orders WHERE id = 99;
```

索引按列的**原始值**有序排列。一旦套了函数或参与运算,引擎无法预知结果的有序性,只能挨个算、挨个比,索引自然废了。

### 2. 隐式类型转换

```sql
-- phone 是 varchar,却用数字去匹配
SELECT * FROM users WHERE phone = 13800138000;   -- 失效
SELECT * FROM users WHERE phone = '13800138000'; -- 正常
```

当字符串列和数字比较时,MySQL 会把列的值统统转成数字再比——相当于对列套了一层 `CAST()` 函数,等价于场景 1,索引失效。反过来,数字列用字符串查通常不会失效(转换发生在常量侧)。这个坑在手机号、订单号这类"长得像数字的字符串"上极其高发。

### 3. 前导 `%` 的模糊查询

```sql
SELECT * FROM users WHERE name LIKE '%三';   -- 失效
SELECT * FROM users WHERE name LIKE '张%';   -- 可用索引
```

索引是按前缀有序的,`张%` 能定位到"张"开头的连续区间;而 `%三` 不知道开头是什么,无从定位,只能全扫。若确有"包含/后缀"搜索需求,应考虑全文索引或 ES。

### 4. OR 连接了非索引列

```sql
-- name 有索引,remark 没有 → 整体退化为全表扫描
SELECT * FROM users WHERE name = '张三' OR remark = 'vip';
```

`OR` 要求两个条件都能走索引才可能用索引合并;只要有一边没索引,为了不漏数据就得全表扫描。可改为 `UNION` 拆成两条都能走索引的查询,或给 `remark` 也加索引。

### 5. 不等于与 NOT IN

`!=`、`<>`、`NOT IN`、`NOT EXISTS` 这类**否定条件**通常选择性很差(匹配的行太多),优化器往往判断走索引比全表扫还慢,于是放弃索引。这不算"bug",而是基于成本的合理选择。`IS NULL` / `IS NOT NULL` 是否走索引则取决于 NULL 值的分布。

## 用 EXPLAIN 看清执行计划

`EXPLAIN` 是诊断的核心工具,重点看这几列:

| 列 | 含义与关注点 |
| --- | --- |
| `type` | 访问类型,优劣排序:`system > const > eq_ref > ref > range > index > ALL`。出现 `ALL`(全表扫)或 `index`(全索引扫)通常要警惕 |
| `key` | 实际使用的索引,为 `NULL` 说明没用上索引 |
| `key_len` | 用到的索引长度,可借此判断联合索引用到了几列 |
| `rows` | 预估要扫描的行数,越小越好 |
| `Extra` | 关键附加信息 |

`Extra` 列几个高频信号:

- **`Using index`**:覆盖索引,无需回表,好事。
- **`Using index condition`**:触发了索引下推 ICP。
- **`Using where`**:Server 层还需额外过滤,常意味着有条件没用上索引。
- **`Using filesort`**:排序无法用索引完成,需额外排序,数据量大时是性能隐患,常可通过让 `ORDER BY` 的列对齐索引顺序来消除。
- **`Using temporary`**:用到了临时表(常见于 `GROUP BY`、`DISTINCT`),代价较高。

诊断开篇那个慢查询时,我会先看 `type` 是不是 `ALL`、`key` 是不是 `NULL`、`rows` 是不是大得离谱,再结合 `Extra` 判断有没有回表和 filesort。一套流程下来,病因基本就定位了。

## 区分度(选择性)与建索引的取舍

**选择性 = 不重复值的数量 / 总行数**,取值 0~1,越接近 1 区分度越高、越值得建索引。

```sql
SELECT COUNT(DISTINCT status) / COUNT(*) AS selectivity FROM orders;
```

- 像 `id`、`order_no` 这类几乎唯一的列,选择性接近 1,索引能精准定位极少的行,效果最好。
- 像 `gender`(男/女)、`status`(几种状态)这类低区分度的列,单独建索引意义不大:`gender = '男'` 可能命中一半的数据,优化器宁可全表扫。
- 联合索引中,应**把区分度高的列放在前面**(在满足最左前缀和业务查询模式的前提下),让索引尽快缩小范围。

对于很长的字符串列(如 URL、长文本),可以用**前缀索引**只索引前 N 个字符,在区分度和索引体积之间权衡:

```sql
ALTER TABLE t ADD INDEX idx_url(url(20));
```

## 什么时候不该建索引

索引不是越多越好。每个索引都要占用存储,并且在 `INSERT`/`UPDATE`/`DELETE` 时都要同步维护 B+Tree,**写入越频繁、索引越多,写性能损耗越大**。以下情况要克制:

- 写多读少、对写入延迟敏感的表(如日志、流水表)。
- 区分度极低的列(状态、性别、是否删除)。
- 数据量很小的表,全表扫本身就很快,索引反而是负担。
- 几乎不出现在查询条件里的列。
- 能被现有联合索引"最左前缀"覆盖的列,不必再单独建。

## 可执行的建索引原则清单

把上面的分析浓缩成一份能直接对照执行的清单:

1. **优先建联合索引而非多个单列索引**,并遵循"等值列在前、范围列在后、高频排序/分组列居中"。
2. **善用最左前缀**:一个 `(a, b, c)` 索引同时覆盖了 `(a)`、`(a, b)`、`(a, b, c)` 的查询,不必重复建。
3. **范围列放最右**,避免它后面的列失效。
4. **追求覆盖索引**:把高频查询 `SELECT` 的少量字段补进索引,消除回表,目标是看到 `Using index`。
5. **选择区分度高的列**建索引,低区分度列单独建意义有限。
6. **超长字符串用前缀索引**,在区分度和体积间取舍。
7. **保持索引列"裸露"**:不要在列上套函数、做运算,注意字符串列别和数字比较以免隐式转换。
8. **避免前导 `%`、慎用 `OR` 连接非索引列**,必要时拆成 `UNION`。
9. **控制索引数量**:写密集表和小表克制建索引,定期清理无人使用的索引。
10. **上线前必跑 `EXPLAIN`**:确认 `type` 不为 `ALL`、`key` 命中预期、`rows` 可控、`Extra` 没有意外的 `Using filesort`/`Using temporary`。

值得一提的是,这套"读执行计划→定位失效原因→改写或调整索引"的诊断思路,如今也越来越多地交给 AI/Agent 来辅助:把 `EXPLAIN` 的输出、表结构和慢查询一起喂给 Agent,它能快速给出"哪一列失效、为什么、如何改写"的初步判断。但真正能让你信任并采纳这些建议的,仍是你脑中那棵"有序的 B+Tree"——理解了为什么,工具才是助力而非黑盒。
