# MySQL 的锁机制:行锁、间隙锁与死锁

设想一个再常见不过的场景:双十一零点,一件库存只剩 1 的商品被两个用户同时下单。两个请求几乎在同一毫秒读到 `stock = 1`,各自判断"还有货",于是各自扣减,最终库存变成 `-1`,商家超卖。再看转账:A 给 B 转 100 元,系统需要先把 A 扣 100、再给 B 加 100,这两步之间如果有别的事务插进来读到了"钱已经从 A 扣了但还没到 B"的中间状态,账就乱了。

这两个问题的本质是一样的:**多个事务并发访问同一份数据时,如果不加以协调,就会互相踩踏**。数据库解决并发冲突的核心武器,就是锁。这篇文章我们就把 MySQL(特别是 InnoDB 存储引擎)的锁机制讲透:它有哪些粒度的锁、共享锁与排他锁怎么配合、InnoDB 那个让人头疼的"间隙锁"到底锁了什么、为什么它能在可重复读隔离级别下防住幻读,以及死锁是怎么发生、怎么排查、怎么避免的。

理解锁不只是为了应付面试。如果你在做 AI Agent 工程——比如让多个 Agent 并发写同一张任务表、或者用数据库做分布式任务队列的状态流转——锁的认知会直接决定你的系统会不会在高并发下出现脏数据或大面积死锁。这是绕不开的基本功。

## 按粒度划分:全局锁、表锁、行锁

锁可以锁"整个库"、锁"一张表",也可以只锁"一行"。粒度越大,实现越简单、开销越小,但并发度越低;粒度越小,并发越好,但加锁、检测死锁的成本越高。这是一个典型的权衡。

### 全局锁

全局锁锁住整个数据库实例,典型命令是:

```sql
FLUSH TABLES WITH READ LOCK;
```

执行后整个库进入只读状态,所有写操作(DML、DDL)和更新类事务的提交都会被阻塞。它的主要用途是**做全库逻辑备份**——为了让备份出来的数据处于一个一致的快照。不过对于 InnoDB 这种支持事务的引擎,更推荐用 `mysqldump --single-transaction`,它靠 MVCC 在一个一致性读事务里完成备份,不需要把全库锁死,对业务影响小得多。

### 表级锁

表锁是 MySQL 层面(而非存储引擎层)实现的锁,主要有两类。

第一类是**表锁**,通过 `LOCK TABLES ... READ/WRITE` 显式加,`UNLOCK TABLES` 释放。它会把整张表锁住,粒度太粗,InnoDB 下我们几乎不主动用它。

第二类更重要,叫 **元数据锁(MDL,Metadata Lock)**。MDL 不需要显式声明,访问一张表时会自动加上:做增删改查(DML)时加 MDL 读锁,做表结构变更(DDL,如 `ALTER TABLE`)时加 MDL 写锁。读锁之间兼容,读锁和写锁、写锁和写锁之间互斥。它的作用是保证你在查表时,别人不能同时改表结构。线上一个经典事故就是:给大表加字段时,如果前面有个长事务一直没提交占着 MDL 读锁,`ALTER` 就会被卡住申请 MDL 写锁,而它一卡,后面所有对这张表的查询又都被它挡住,瞬间雪崩。

### 行级锁

行锁由 InnoDB 在引擎层实现,只锁定被操作的行,并发度最高,也是我们日常打交道最多的锁。MyISAM 引擎不支持行锁,这也是 InnoDB 在高并发写场景下完胜 MyISAM 的关键原因之一。下面的篇幅,主要就围绕 InnoDB 的行锁展开。

| 粒度 | 实现层 | 加锁方式 | 并发度 | 典型场景 |
| --- | --- | --- | --- | --- |
| 全局锁 | Server 层 | `FLUSH TABLES WITH READ LOCK` | 最低 | 全库备份 |
| 表锁 / MDL | Server 层 | `LOCK TABLES` / 访问时自动 | 较低 | DDL 期间保护表结构 |
| 行锁 | InnoDB 引擎层 | DML 时自动按需加 | 最高 | 日常并发读写 |

## 共享锁 S 与排他锁 X

无论哪种粒度,锁在"读写互斥"这一层面上都分为两种基本模式。

- **共享锁(S 锁,Shared Lock)**:也叫读锁。多个事务可以同时持有同一行的 S 锁,彼此不冲突——大家都只是读。
- **排他锁(X 锁,Exclusive Lock)**:也叫写锁。一旦某事务持有某行的 X 锁,其他事务既不能再加 S 锁也不能加 X 锁,必须等待。

它们的兼容关系可以用一张表概括:

| | 请求 S 锁 | 请求 X 锁 |
| --- | --- | --- |
| 已持有 S 锁 | 兼容 | 冲突 |
| 已持有 X 锁 | 冲突 | 冲突 |

记忆要点:**只有"读读"兼容,"读写""写写"都互斥**。

在 InnoDB 中,`UPDATE`、`DELETE`、`INSERT` 会自动给涉及的行加 X 锁。普通的 `SELECT`(快照读)不加任何锁,靠 MVCC 读历史版本。如果你需要在读的时候加锁(当前读),可以显式声明:

```sql
SELECT ... LOCK IN SHARE MODE;   -- 加 S 锁,MySQL 8.0 推荐写法:FOR SHARE
SELECT ... FOR UPDATE;           -- 加 X 锁
```

回到开头的扣库存场景,正确做法就是用当前读把行锁住:

```sql
BEGIN;
SELECT stock FROM goods WHERE id = 100 FOR UPDATE;  -- 拿到 X 锁,后来者阻塞
UPDATE goods SET stock = stock - 1 WHERE id = 100 AND stock > 0;
COMMIT;
```

这样第二个事务在 `SELECT ... FOR UPDATE` 处就会被挡住,直到第一个事务提交,从根本上杜绝超卖。

## 意向锁 IS 与 IX

现在出现一个问题:假设事务 A 已经锁了表里的某几行(行锁),此时事务 B 想 `LOCK TABLES ... WRITE` 给整张表加表锁,它怎么知道表里有没有行被锁着?难道要逐行扫描成千上万条记录看有没有锁?那效率太低了。

InnoDB 的解法是**意向锁(Intention Lock)**,这是一种表级锁,但它不锁数据,只是一个"声明":

- **意向共享锁(IS)**:事务打算给某些行加 S 锁前,先在表上加 IS。
- **意向排他锁(IX)**:事务打算给某些行加 X 锁前,先在表上加 IX。

意向锁由 InnoDB 自动维护,你不用管。它的妙处在于:意向锁之间**互相兼容**(IS、IX 彼此都不冲突),只和真正的表级 S/X 锁冲突。这样一来,事务 B 想加表级 X 锁时,只需看一眼表上有没有 IX/IS,就立刻知道"有人在用行锁",无需扫描全表。意向锁本质上是一个高效的"标记位"。

## InnoDB 行锁的三种实现

很多人以为行锁就是"锁一行",但 InnoDB 的行锁实际上是加在**索引**上的,并且有三种形态。理解这三种,是搞懂 InnoDB 并发行为的关键。

### 记录锁(Record Lock)

最直接的一种,锁住索引上的一条具体记录。例如:

```sql
SELECT * FROM t WHERE id = 5 FOR UPDATE;  -- id 是主键
```

`id = 5` 这一行被精确锁住,其他事务无法修改或删除它,但 `id = 4`、`id = 6` 不受影响。

### 间隙锁(Gap Lock)

间隙锁锁住的不是某条记录,而是记录之间的**开区间**。假设表里现有 `id` 为 5、10、20 的三行,那么存在这样几个间隙:`(-∞, 5)`、`(5, 10)`、`(10, 20)`、`(20, +∞)`。当你执行:

```sql
SELECT * FROM t WHERE id BETWEEN 5 AND 15 FOR UPDATE;
```

InnoDB 会锁住相关的间隙,使得任何事务**无法向这些间隙中插入新行**。间隙锁的唯一目的就是阻止插入,从而防止幻读。一个反直觉但重要的点:**间隙锁之间是互相兼容的**——两个事务可以同时持有同一个间隙的 Gap 锁,因为它们都只是"禁止往里插",并不冲突。(这一点恰恰是后面某些死锁的根源。)

### 临键锁(Next-Key Lock)

临键锁是记录锁与间隙锁的组合,锁住一个**左开右闭的区间** `(上一条记录, 当前记录]`。它既锁住了当前这条记录本身,又锁住了它前面的间隙。**临键锁是 InnoDB 在可重复读隔离级别下行锁的默认形态**——平时你以为加的是行锁,其实大多时候 InnoDB 加的是 Next-Key Lock。

举例,表中有 `id` 为 5、10、20,执行 `WHERE id = 10 FOR UPDATE`(假设 id 是普通索引而非唯一索引),InnoDB 锁住的区间是 `(5, 10]`,同时往往还会延伸锁住下一个间隙 `(10, 20)`,既不让改 10,也不让在 `(5, 10)` 和 `(10, 20)` 之间插入新值。

## RR 隔离级别下,Next-Key Lock 如何防住幻读

幻读(Phantom Read)指的是:同一个事务里,前后两次执行**相同的范围查询**,第二次却多出了"幻影"行——因为期间别的事务插入了符合条件的新数据。

在可重复读(REPEATABLE READ,RR)下,InnoDB 用两套机制应对:

- **快照读**(普通 `SELECT`):靠 MVCC,整个事务读的都是事务开始时的一致性快照,自然看不到新插入的行。
- **当前读**(`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`):靠 **Next-Key Lock**。

来看一个例子。表 `account` 中 `balance` 列上有索引,现有余额为 100、200、300 的三行:

```sql
-- 事务 A
BEGIN;
SELECT * FROM account WHERE balance BETWEEN 100 AND 300 FOR UPDATE;
-- 这一刻,InnoDB 用 Next-Key Lock 锁住了 (100,200]、(200,300] 等区间,
-- 还会锁住 (300, 下一条) 的间隙
```

此时事务 B 想插入一条 `balance = 250` 的新行:

```sql
-- 事务 B
INSERT INTO account (balance) VALUES (250);  -- 被阻塞!
```

因为 250 落在被事务 A 用临键锁锁住的 `(200, 300]` 区间里,插入操作申请插入意向锁时会与间隙锁冲突,只能等待。这样,当事务 A 再次执行同样的范围查询时,绝不会凭空多出 250 这一行——幻读被彻底挡住。**这正是间隙锁存在的全部意义:不是为了锁已有数据,而是为了锁"还不存在但可能被插入"的数据**。

值得一提的是,如果是在读已提交(READ COMMITTED,RC)隔离级别下,InnoDB 会退化掉间隙锁(基本只用记录锁),所以 RC 下并发更高、更少死锁,但需要靠业务自身或 binlog 的 row 格式来应对幻读问题。

## 死锁:怎么发生

死锁是两个或多个事务互相持有对方需要的锁,谁也不肯松手,陷入循环等待。最经典的就是**交叉加锁**。回到转账场景,两个事务分别要给 A、B 两个账户加锁,但加锁顺序相反:

```sql
-- 事务 1                              -- 事务 2
BEGIN;                                 BEGIN;
UPDATE acct SET bal=bal-100            UPDATE acct SET bal=bal-50
  WHERE id = 'A';  -- 锁住 A             WHERE id = 'B';  -- 锁住 B
-- 接着想锁 B                            -- 接着想锁 A
UPDATE acct SET bal=bal+100            UPDATE acct SET bal=bal+50
  WHERE id = 'B';  -- 等待事务2释放 B     WHERE id = 'A';  -- 等待事务1释放 A
```

事务 1 拿着 A 等 B,事务 2 拿着 B 等 A,形成闭环。InnoDB 内置死锁检测,会主动发现这个环,并**回滚其中一个代价较小的事务**(报 `Deadlock found when trying to get lock`),让另一个得以继续,从而打破僵局。注意:被回滚的事务需要应用层捕获错误并重试。

## 死锁:怎么排查

死锁发生后,第一手资料来自:

```sql
SHOW ENGINE INNODB STATUS;
```

输出里的 `LATEST DETECTED DEADLOCK` 段落会记录最近一次死锁:两个事务各自执行的 SQL、各自持有(HOLDS THE LOCK)和等待(WAITING FOR THIS LOCK)的锁、以及 InnoDB 最终回滚了哪个(WE ROLL BACK)。读懂这一段,通常就能定位是哪两条 SQL、按什么顺序加锁导致的。

对于"正在发生"的锁等待(还没演变成死锁),可以查 `information_schema`(MySQL 8.0 中 `INNODB_LOCKS`/`INNODB_LOCK_WAITS` 已迁移到 `performance_schema`):

```sql
-- MySQL 8.0:当前的锁等待关系
SELECT * FROM performance_schema.data_lock_waits;
-- 当前持有/等待的锁明细
SELECT * FROM performance_schema.data_locks;
-- 谁在等谁、被阻塞的线程
SELECT * FROM sys.innodb_lock_waits;
```

配合 `SHOW PROCESSLIST` 看哪个连接卡住、`information_schema.INNODB_TRX` 看长事务,基本能还原现场。日常还可以开启 `innodb_print_all_deadlocks = ON`,把每次死锁都打进错误日志,便于事后复盘。

## 死锁:怎么减少

死锁无法 100% 杜绝,但可以大幅降低概率:

1. **统一加锁顺序**。这是最有效的一招。所有事务都按同一规则(比如按主键 id 从小到大)申请锁,交叉等待的环就形不成。上面的转账例子,只要两个事务都"先锁 id 小的账户",死锁立刻消失。
2. **缩短事务、尽早提交**。事务越短,持锁时间越短,冲突窗口越小。不要在事务里夹杂 RPC 调用、等待用户输入这类慢操作。
3. **减少锁范围**。让 `WHERE` 走索引精确命中,避免全表扫描把大量行甚至间隙都锁上;能用记录锁就别触发大范围临键锁。
4. **降低隔离级别**(谨慎)。在业务能接受的前提下用 RC,可消除间隙锁带来的大量死锁。
5. **必备的重试机制**。应用层捕获死锁错误(MySQL 错误码 1213),自动重试若干次。死锁回滚是 InnoDB 的正常自愈行为,业务必须能优雅地接住它。

## 加锁实战建议

最后给一份能直接落地的清单:

- **加锁字段务必走索引**。InnoDB 行锁加在索引上,如果查询没用上索引,会退化成给所有扫描到的记录加锁,几乎等同于锁表,并发瞬间归零。
- **能快照读就别当前读**。只是查询用途的 `SELECT` 别随手加 `FOR UPDATE`,把锁留给真正需要写一致性的地方。
- **写操作把过滤条件写精确**。`UPDATE ... WHERE id = ?` 命中唯一索引时只加记录锁,而范围条件会引入临键锁,影响面大得多。
- **用唯一索引/数据库约束兜底**。扣库存时 `WHERE stock > 0` 配合 `stock` 非负约束,即使锁的逻辑有疏漏,数据库层面也不会让库存变负。
- **批量操作注意顺序**。批量 `UPDATE`/`DELETE` 时,让 SQL 按确定顺序(如主键序)处理,避免多个批量任务交叉加锁。

锁的世界里没有银弹,它本质上是在**数据正确性**和**并发性能**之间反复权衡。把粒度、S/X、意向锁、三种行锁的关系想清楚,你就能在面对超卖、转账、队列状态流转这类问题时,准确判断"该锁什么、锁多大、锁多久",写出既正确又不至于把数据库压垮的代码。
