# Redis 的过期删除与内存淘汰策略

你给一批 key 设置了过期时间，按理说它们到点就该消失，可你盯着 `used_memory` 一看——内存还在涨。是 Redis 没删吗？还是删得不够快？这个看似简单的现象，背后牵扯出 Redis 两套相互独立又彼此配合的机制：**过期删除**决定"一个设了 TTL 的 key 什么时候真正从内存里消失"，**内存淘汰**决定"当内存逼近上限时，Redis 拿哪些 key 开刀"。把这两件事分清楚，你才能理解为什么过期了内存还在涨，也才能在生产环境里把 Redis 配对。

本文从"过期了为什么内存还在"讲起，逐层拆解。

## 一、过期 key 是怎么被删掉的

Redis 在主库里用一个独立的字典记录每个带过期时间的 key 及其到期时间戳（毫秒）。但"记下了到期时间"不等于"到期就立刻删"。摆在 Redis 面前有三种思路。

### 定时删除：精确但烧 CPU

给每个设了过期时间的 key 都挂一个定时器，时间一到回调函数就把它删掉。这种做法对内存最友好——key 一过期就立即释放。但代价是 CPU：如果同一时刻有几十万个 key 到期，Redis 主线程就得在这一瞬间集中处理几十万次删除，而 Redis 是单线程处理命令的，这段时间它没法响应正常的读写请求，造成明显的延迟尖刺。维护海量定时器本身也有开销。

对一个以"低延迟、高吞吐"为卖点的内存数据库来说，用宝贵的 CPU 时间去保证"过期 key 一秒都不多留"，性价比太低。所以 **Redis 不用定时删除**。

### 惰性删除：省 CPU 但可能漏

惰性删除（lazy expiration）的思路是：平时不管，等到有人来访问这个 key 时，再检查它是否过期。如果过期了，就在这次访问中顺手删掉，并对调用方返回"不存在"。

这样 CPU 只在真正需要时才花，但缺点很明显：**如果一个过期的 key 之后再也没人访问，它就永远赖在内存里**。这正是"key 设了过期、内存却在涨"的第一个原因——那些过期但无人问津的 key，惰性删除根本碰不到它们。

### 定期删除：兜底的扫描

为了清理这些"漏网之鱼"，Redis 又加了一层**定期删除**（active expiration）。它按一定频率（由 `hz` 配置影响，默认每秒约 10 次）启动一个后台任务，从设置了过期时间的 key 里**随机抽样一批**来检查，删掉其中已过期的。

注意这里的关键词是"随机抽样"，不是"全量扫描"。全量扫描海量 key 同样会阻塞主线程。Redis 的定期删除采用一个自适应的循环：每次从过期字典里随机取一批 key（默认每批 20 个），删掉其中过期的；如果这批里过期比例超过某个阈值（约 25%），说明过期 key 还很多，就立即再来一轮；低于阈值才停手。同时整个过程有时间上限，避免单次占用主线程太久。

### 组合的智慧

所以 Redis 的过期删除是 **惰性删除 + 定期删除** 的组合：

| 策略 | CPU 开销 | 内存友好度 | Redis 是否采用 |
| --- | --- | --- | --- |
| 定时删除 | 高（集中爆发） | 最好 | 否 |
| 惰性删除 | 极低 | 差（可能永久残留） | 是 |
| 定期删除 | 可控（抽样 + 限时） | 中 | 是 |

惰性删除保证"被访问到的过期 key 一定不会被误用"，定期删除则以可控的代价持续回收那些没人访问的过期 key。两者都是概率性、尽力而为的——这意味着**任意时刻内存中都可能存在一部分已过期但尚未被删除的 key**，这是设计上的有意权衡，不是 bug。

## 二、当内存撞上 `maxmemory`

过期删除解决的是"该走的 key 慢慢走"。但如果你写入速度极快，或者很多 key 压根没设过期时间，内存迟早会逼近物理上限。为此 Redis 提供了 `maxmemory` 来设上限：

```conf
# 设置最大可用内存为 4GB
maxmemory 4gb
# 达到上限后的淘汰策略
maxmemory-policy allkeys-lru
```

一旦 `used_memory` 触及 `maxmemory`，Redis 在执行会增加内存的写命令前，会先按 `maxmemory-policy` 淘汰一些 key 腾出空间。这套淘汰逻辑共有 **8 种策略**，可以按两个维度来记：**淘汰范围**（全体 key / 仅设了过期时间的 key）× **淘汰依据**（LRU / LFU / random / TTL）。

| 策略 | 淘汰范围 | 依据 | 典型场景 |
| --- | --- | --- | --- |
| `noeviction` | 不淘汰 | —— | 写满即报错，数据绝不能丢的场景 |
| `allkeys-lru` | 所有 key | 最近最少使用 | 纯缓存，访问有热点 |
| `allkeys-lfu` | 所有 key | 最不经常使用 | 纯缓存，热点稳定、有长期高频 key |
| `allkeys-random` | 所有 key | 随机 | 各 key 访问概率均匀 |
| `volatile-lru` | 设了过期时间的 key | 最近最少使用 | 缓存与持久数据混存 |
| `volatile-lfu` | 设了过期时间的 key | 最不经常使用 | 同上，按频率淘汰 |
| `volatile-random` | 设了过期时间的 key | 随机 | 混存且无明显热点 |
| `volatile-ttl` | 设了过期时间的 key | 剩余 TTL 最短优先 | 希望快到期的先走 |

几个要点：

- **`noeviction` 是早期版本的默认值**（新版本默认值随版本演进，请以实际配置为准）。它什么都不淘汰，内存满后所有写命令直接返回错误（OOM），但读命令仍可用。适合那些把 Redis 当主存储、绝不容许悄悄丢数据的场景。
- **`volatile-*` 系列只在"设了过期时间的 key"里挑**。如果你用的是 `volatile-lru` 但所有 key 都没设 TTL，那么当内存满时它**找不到可淘汰的 key，行为退化得和 `noeviction` 一样**——写命令报错。这是一个常见的线上坑：以为配了淘汰策略就高枕无忧，结果因为没给 key 设过期时间，照样写不进去。

## 三、近似 LRU 与近似 LFU：为什么是"近似"

教科书里的 LRU 需要一个双向链表把所有 key 按访问时间串起来，每次访问就把 key 移到头部，淘汰时取链表尾部。但在 Redis 里给几百万个 key 维护这样一条精确链表，内存和指针维护成本都很高。Redis 选择了**近似 LRU**。

### 近似 LRU：靠采样逼近

Redis 在每个对象头里存了一个 24 位的"最近访问时间"字段（LRU clock），每次访问该 key 时更新它。淘汰时，Redis 并不遍历全部 key 找最旧的，而是**随机采样若干个 key**（数量由 `maxmemory-samples` 控制，默认 5），从这几个里挑出访问时间最旧的淘汰掉。

```conf
# 采样数越大，越接近精确 LRU，但 CPU 开销也越高
maxmemory-samples 5
```

采样数越大，结果越接近真实 LRU，但每次淘汰要比较的对象越多。Redis 还引入了一个"淘汰池"（eviction pool）来记住历次采样中比较旧的候选者，让多次采样的效果累积，使近似 LRU 的质量进一步接近精确 LRU。`maxmemory-samples=10` 时的效果已经和理论 LRU 相当接近，而默认 5 在精度和性能间取得了不错的平衡。

### 近似 LFU：计数 + 衰减

LRU 只看"最近有没有被访问"，有个软肋：一个 key 被偶尔扫一次（比如某个全量遍历），就会被误判为"刚刚很热"，从而逃过淘汰。LFU（Least Frequently Used）改看**访问频率**，更适合识别真正长期的热点。

Redis 的 LFU 同样复用对象头里那 24 位，但把它拆成两部分：高 16 位存"上次访问时间"，低 8 位存一个**访问计数器**。这个 8 位计数器最大才 255，显然不能直接当真实访问次数，所以 Redis 用了两个巧思：

- **对数式自增**：计数器不是每次访问 +1，而是以越来越小的概率递增——计数越高，再涨一次越难。这让一个 8 位的小计数器能映射到很大的访问频率范围。递增概率由 `lfu-log-factor` 控制。
- **计数衰减**：如果只增不减，曾经的热点 key 即使早已凉透，凭着历史高分也永远不会被淘汰。所以 Redis 让计数器随时间衰减：一个 key 越久没被访问，它的计数就会按 `lfu-decay-time`（单位分钟）逐步减小。

```conf
# 计数递增的对数因子，越大则计数器增长越慢，可覆盖更大频率范围
lfu-log-factor 10
# 多少分钟无访问衰减一个计数单位
lfu-decay-time 1
```

正是"对数自增 + 时间衰减"这套组合，让 8 个比特就能近似刻画出 key 的访问热度，既区分得出冷热，又不会被陈旧的历史高频锁死。

## 四、过期 key 在持久化与主从下的处理

过期删除是概率性的，那么这些"已过期但还没被物理删除"的 key，在持久化和主从复制时会不会被错误地保留？Redis 在这些链路上都做了正确性兜底。

- **RDB**：生成 RDB 快照时，已过期的 key **不会被写入**文件；加载 RDB 时，主库会跳过已过期的 key，从库则会照常载入（载入后由主库下发的删除指令来清理）。
- **AOF**：当一个 key 被过期删除（无论惰性还是定期触发），Redis 会向 AOF 追加一条显式的 `DEL`（新版本为 `UNLINK`）命令，确保 AOF 重放时该 key 同样被删。AOF 重写时也会自动忽略已过期的 key。
- **主从复制**：这里是最容易出错、也最关键的一点——**从库不会自己主动删除过期 key**。过期判定统一由**主库**做：主库删除一个过期 key 后，会向所有从库下发一条 `DEL` 命令，由它驱动从库删除。这样设计是为了保证主从数据一致、避免从库各自为政导致同一个 key 在主从上状态不一致。
- 那么在主库还没下发 `DEL` 之前，从库被读到一个逻辑上已过期的 key 怎么办？从库在响应读请求时会**逻辑上判断它已过期**，对客户端返回"不存在"，但**并不真正删除底层数据**，依旧等主库的删除指令来物理回收。

理解这点很重要：从库上看似"还在"的过期 key，对客户端其实是不可见的，数据正确性有保障，只是物理内存的回收要等主库统一调度。

## 五、实战配置建议

把上面的机制落到配置上，按用途选型：

- **纯缓存场景**（Redis 里的数据丢了可以从数据库回源）：用 `allkeys-lru`。它对所有 key 生效，能在内存满时优雅地淘汰冷数据，是最常用、最稳妥的选择。如果你的热点非常稳定、希望长期高频 key 优先保留，可考虑 `allkeys-lfu`。
- **缓存 + 持久数据混存**：给可丢弃的缓存 key 设过期时间，给重要数据不设，再用 `volatile-lru` / `volatile-lfu`，这样淘汰只会落在缓存上，保护无 TTL 的重要数据。但务必确认确实有 key 设了 TTL，否则会退化成写报错。
- **Redis 作主存储、绝不能丢数据**：`noeviction` + 完善的容量监控告警，宁可写失败也不丢数据，由上层做好降级。
- 始终配 `maxmemory`，否则前面这些淘汰策略根本不会触发，内存可能一路涨到把机器吃满、被 OS 的 OOM Killer 杀掉。`maxmemory-samples` 一般保持默认 5 即可，对淘汰精度敏感且 CPU 有余量时可调到 10。

最后落到一个当下很现实的场景：在 AI / Agent 工程里，我们常用 Redis 缓存大模型的回答、向量召回结果、会话上下文或工具调用结果——这类数据天然"可重算、有冷热、量还不小"。给它们设上合理的过期时间，再配 `maxmemory` + `allkeys-lru`（或对长期热点用 `allkeys-lfu`），就能在固定内存预算内自动淘汰冷门的历史会话、留住高频复用的结果，把 Redis 用成一个稳定可控的 LLM 缓存层。理解过期与淘汰这两套机制，正是把缓存"用对"的前提。
