# 如何用 Redis 实现分布式锁?

设想这样一个场景:你的服务部署了 4 个实例,每个实例上都跑着一个定时任务,每天凌晨给用户结算账单。某天凌晨,这 4 个实例几乎在同一毫秒被各自的定时器唤醒,同时去读取「待结算订单」,同时开始扣款。结果就是同一笔订单被结算了 4 次,用户被多扣了钱。

这不是危言耸听,而是分布式系统里最常见的「并发抢占同一资源」问题。在单机时代,我们用 `synchronized`、`ReentrantLock` 就能让多个线程排队访问临界区;但这些锁的作用域只在单个 JVM 进程内,跨进程、跨机器时它们完全失效。我们需要一把所有实例都能看见、都认可的「外部锁」——这就是分布式锁。

在 AI/Agent 工程里这个需求同样普遍:一个任务队列被多个 worker 消费,如果不加约束,同一个推理任务可能被多个 worker 同时拉取并重复执行,既浪费昂贵的 GPU/Token,又可能产生重复的副作用(比如重复发邮件、重复写库)。给「正在处理某任务」这件事加一把分布式锁,就能保证「同一时刻只有一个 worker 在处理该任务」。

本文我们从零开始,一步步把一把正确的 Redis 分布式锁搭起来,并讲清每一步「为什么必须这么做」。

## 一、为什么选 Redis 来做锁

实现分布式锁的本质,是需要一个「所有节点都能访问的、支持原子操作的共享存储」。常见选项有数据库、ZooKeeper、Redis。Redis 之所以流行,原因有三:

- **快**:纯内存操作,加锁/解锁延迟在微秒级,对业务性能影响极小。
- **天然支持过期**:可以给 key 设 TTL,即使持锁进程崩溃,锁也会自动释放,不会永久死锁。
- **单线程命令模型**:Redis 处理命令是串行的,这为「原子性」提供了底层保证。

锁的核心语义很简单:**用一个 key 的存在与否表示「锁被持有」**。谁能把这个 key 创建成功,谁就拿到了锁;用完后把 key 删掉,就是释放锁。

## 二、最朴素的实现:为什么不能分两步

最直觉的写法是先判断 key 在不在,不在就设置:

```bash
# 错误示范:两条命令不是原子的
SETNX lock:order:1001 1   # 不存在才设置,拿到锁
EXPIRE lock:order:1001 30 # 再设置 30 秒过期
```

这里有个致命缺陷:`SETNX` 成功之后、`EXPIRE` 执行之前,如果进程恰好崩溃(或被 OOM Kill),那么这个 key 就**永远没有过期时间**了。后续所有请求都会卡在这把永不释放的死锁上。

问题的根源是「加锁」和「设过期」被拆成了两步,中间存在窗口。要解决它,必须让这两件事在一条命令里原子完成。

## 三、正确的加锁:SET ... NX PX 一条命令搞定

从 Redis 2.6.12 起,`SET` 命令支持了扩展参数,可以一次性完成「不存在才设置 + 设置过期时间」:

```bash
SET lock:order:1001 <unique_value> NX PX 30000
```

逐项拆解:

| 参数 | 含义 |
| --- | --- |
| `NX` | Not eXists,只有 key 不存在时才设置成功,返回 `OK`;否则返回 `nil` |
| `PX 30000` | 设置过期时间为 30000 毫秒(也可用 `EX` 设秒) |
| `<unique_value>` | 锁的「持有者标识」,必须全局唯一 |

`NX` 保证了「同一时刻只有一个客户端能设置成功」,即互斥;`PX` 保证了「即使持锁者崩溃,锁也会在 30 秒后自动失效」,即防死锁。两者在一条命令里完成,杜绝了上一节的崩溃窗口。

加锁成功返回 `OK`,意味着拿到了锁;返回 `nil`,意味着锁被别人占着,本次抢锁失败,业务可以选择重试或放弃。

## 四、为什么 value 必须是唯一标识

很多人加锁时把 value 随便填成 `1`,这会埋下一个隐蔽的大坑。看下面这个时序:

1. 客户端 A 拿到锁,TTL 30 秒;
2. A 的业务因为 GC 或慢调用,执行了 35 秒;
3. 第 30 秒时,锁自动过期,A 其实**已经不再持有锁**;
4. 客户端 B 此时成功抢到锁,开始执行;
5. 第 35 秒,A 业务执行完,执行「释放锁」——它删掉了 key;
6. 但这个 key 现在是 **B 的锁**!A 把 B 的锁误删了;
7. 客户端 C 又趁机抢到锁……锁的互斥性彻底崩塌。

根因是:A 在释放时无法判断「现在这把锁还是不是我的」。解决办法是给每个客户端分配一个唯一的 value(比如 `UUID` 或 `线程ID + 时间戳`),释放锁时**先比对 value 是不是自己当初设置的,是自己的才删**。

## 五、释放锁:为什么必须用 Lua 保证原子

按上面的思路,释放逻辑是「先 GET 比对,再 DEL」:

```bash
# 错误示范:GET 和 DEL 之间又有窗口
GET lock:order:1001          # 取出 value 比对
DEL lock:order:1001          # 确认是自己的才删
```

但「比对」和「删除」又是两步!在 GET 返回「是我的锁」之后、DEL 执行之前,锁可能恰好过期并被别人抢走,这时 DEL 删的还是别人的锁。我们又回到了第四节的误删问题。

要让「判断归属 + 删除」原子化,就要借助 Redis 的 Lua 脚本能力——Redis 执行 Lua 脚本时是单线程、不可被打断的,整段脚本要么全做完要么全不做:

```lua
-- KEYS[1] 是锁的 key,ARGV[1] 是当前客户端的唯一标识
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

通过 `EVAL` 执行它:

```bash
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end" 1 lock:order:1001 <unique_value>
```

只有当锁的 value 等于自己的标识时才删除,且整个「判断 + 删除」在一次执行中完成,中间不会被其他命令插入。至此,我们有了一把「加锁原子、释放原子、不会误删、能防死锁」的基础锁。

## 六、锁过期了但业务还没跑完:看门狗续期

第四节暴露了另一个独立问题:**TTL 设短了,业务还没执行完锁就过期了**。这会破坏互斥——别人会在你还在干活时抢到锁。

那把 TTL 设很长行不行?也不行。如果持锁进程崩溃,这把超长的锁要等很久才释放,期间所有人都被阻塞,失去了「自动恢复」的意义。

真正的解法是**自动续期(看门狗,Watchdog)**:加锁时给一个合理的初始 TTL(比如 30 秒),同时启动一个后台线程,每隔 TTL/3(约 10 秒)检查一次——只要业务还在执行且锁还属于自己,就把 TTL 重置回 30 秒。这样:

- 业务正常运行时,锁被不断续命,永远不会中途过期;
- 一旦进程崩溃,后台线程也随之消失,不再续期,锁会在最多一个 TTL 周期后自动释放。

这正是 **Redisson** 的核心设计。用 Redisson 写一把可重入锁非常简洁:

```java
RLock lock = redisson.getLock("lock:order:1001");
lock.lock();                 // 默认 30 秒 TTL,自动开启看门狗续期
try {
    // 临界区:处理订单
} finally {
    lock.unlock();           // 释放锁,同时停止续期
}
```

注意一个细节:如果你调用的是 `lock.lock(10, TimeUnit.SECONDS)` 显式指定了租约时间,Redisson 就**不会启动看门狗**——因为你已经明确表达了「锁最多持有 10 秒」的意图。续期机制只在你不指定租约时间时才生效。续期同样靠 Lua 脚本保证「判断归属 + 重置 TTL」的原子性。

## 七、RedLock:多主节点方案及其争议

到这里,我们的锁都建立在「单个 Redis 节点」之上。但若该节点是主从架构,会有一个隐患:主节点写入锁后、还没来得及同步给从节点就宕机了,从节点被提升为主,新主上根本没有这把锁,于是别人又能加锁成功——**互斥被打破**。

为应对这个问题,Redis 作者提出了 **RedLock(红锁)** 算法,思路是部署 N 个(通常 5 个)相互独立的 Redis 主节点(不做主从复制),加锁时:

1. 记录当前时间;
2. 依次向所有 N 个节点用相同 key/value 申请锁,每次申请设很短的超时;
3. 只有在**超过半数(N/2+1)节点加锁成功**、且总耗时小于锁的有效期时,才算加锁成功;
4. 锁的实际有效时间 = 初始 TTL 减去加锁过程的总耗时;
5. 若加锁失败,向所有节点(包括没成功的)发起释放。

它靠「多数派」来容忍少数节点故障。但 RedLock 引发了分布式领域一场著名争论。**Martin Kleppmann**(《数据密集型应用系统设计》作者)提出了几点质疑:

- **依赖系统时钟**:RedLock 的正确性建立在各节点时钟以大致相同速率前进的假设上。一旦发生时钟跳变(NTP 校时、管理员手动改时间),节点可能提前认为锁过期,导致两个客户端同时持锁。
- **GC/网络停顿无法防御**:客户端拿到锁后,若发生长时间 STW GC 暂停或网络延迟,锁可能已经过期被别人抢走,而该客户端「醒来」后仍以为自己持锁并去操作共享资源。这个问题对**任何**带 TTL 的锁都存在,RedLock 也不例外。
- **真正需要强正确性时**,他主张用带 **fencing token(单调递增的令牌)** 的方案:每次加锁返回一个递增编号,被保护的资源端校验该编号,拒绝处理「编号比已见过的更小」的请求,从而即使旧持锁者复活也无法造成破坏。

Redis 作者 antirez 随后做了回应,双方各执一词。**结论上的共识是**:如果你的场景对锁失效只是「效率问题」(偶尔重复执行一次,代价可接受),单 Redis 锁足够;如果是「正确性问题」(绝不允许两个客户端同时操作,否则数据损坏或资损),那么不要只依赖任何基于 TTL 的锁,而应配合 fencing token 或具备强一致语义的协调系统。

## 八、和数据库锁、ZooKeeper 锁的对比

| 维度 | Redis 锁 | 数据库锁 | ZooKeeper 锁 |
| --- | --- | --- | --- |
| 性能 | 最高(内存) | 较低(磁盘 IO) | 中等 |
| 实现复杂度 | 中(需处理过期/续期) | 低(唯一索引或 `SELECT ... FOR UPDATE`) | 中(临时顺序节点) |
| 可靠性 | 主从切换有丢锁风险 | 强(依赖事务) | 强(ZAB 一致性协议) |
| 死锁防护 | TTL 自动过期 | 需靠连接断开或超时 | 临时节点,会话断开自动删除 |
| 锁释放感知 | 需轮询重试 | 需轮询 | Watch 机制,可被动通知 |

简单说:

- **数据库锁**适合并发不高、已有数据库依赖、追求实现简单和强一致的场景。唯一索引插入或行锁都能实现,但磁盘 IO 让它扛不住高并发。
- **ZooKeeper 锁**用临时顺序节点 + Watch 实现,会话断开锁自动消失(无需依赖 TTL),且支持「排队 + 通知」避免惊群,一致性强。代价是部署运维更重、吞吐不如 Redis。
- **Redis 锁**性能最佳,适合高并发、能容忍极端情况下偶发失效的场景,是互联网业务最常见的选择。

## 九、常见坑与适用场景

实践中容易踩的坑:

1. **不设过期时间**:进程崩溃直接造成永久死锁。务必用 `SET ... NX PX`。
2. **value 不唯一 / 释放不校验归属**:导致误删别人的锁,前文已详述。
3. **释放用 GET + DEL 而非 Lua**:破坏原子性。
4. **业务耗时超过 TTL 又没续期**:互斥失效,要么用看门狗,要么确保 TTL 足够覆盖业务。
5. **不可重入**:同一线程二次加锁会失败,需要可重入设计(Redisson 用 Hash 结构记录重入次数)。
6. **把它当强一致锁用在资损场景**:见第七节,这类场景要上 fencing token。

适用场景:防止定时任务重复执行、避免缓存击穿时的并发重建、限制某资源同一时刻只有一个处理者(如前文 AI worker 去重消费任务)、秒杀/库存扣减的串行化等。

## 小结

一把正确的 Redis 分布式锁,关键就在于把每一步都做成**原子**的:用 `SET key unique_value NX PX ms` 让「加锁 + 设过期」一步完成;用唯一 value 标识持有者;用 Lua 脚本让「判断归属 + 删除/续期」一步完成;用看门狗解决业务超时;清醒认识到它在主从切换、时钟漂移、进程停顿下的局限,并在真正需要强正确性时引入 fencing token。理解了这些「为什么」,你就不只是会调 API,而是能在任何语言、任何客户端里亲手写出一把可靠的分布式锁。
