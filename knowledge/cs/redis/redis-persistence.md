# Redis 持久化:RDB 与 AOF 怎么选?

先做一个小实验:你往 Redis 里写了一批数据,然后把进程 `kill` 掉再重启,数据还在吗?

答案是:**取决于你有没有开启持久化**。Redis 是一个内存数据库,所有数据默认只活在内存里。一旦进程退出、机器重启、容器被销毁,内存被回收,数据就灰飞烟灭。"重启后数据是否还在"这个问题,本质上就是在问:**Redis 有没有把内存里的数据落到磁盘上,以及落了多少**。

Redis 给出了两套答案:RDB 和 AOF。理解它们的区别,不是为了背配置项,而是为了能回答一个更现实的问题——"我能接受丢多少数据"。

## 为什么内存数据库也需要落盘

Redis 之所以快,正是因为它绕开了磁盘,直接在内存里读写。但内存有两个天生的缺陷:**易失**(断电即丢)和**有限**(容量受限于物理内存)。持久化解决的是第一个问题:让重启后能恢复数据。

落盘有两条根本不同的思路:

- **保存"结果"**:某一时刻内存里长什么样,就照原样拍一张快照存下来。这是 RDB。
- **保存"过程"**:把每一条修改数据的命令都记下来,重启时重新执行一遍,把数据"演算"回来。这是 AOF。

一个存状态,一个存日志。后面所有的取舍,都从这个根本差异里长出来。

## RDB:内存快照

RDB(Redis Database)是把某个时间点的全部内存数据,序列化成一个紧凑的二进制文件(默认 `dump.rdb`)。它是一张"照片"。

### 怎么触发

```conf
# redis.conf:在 900 秒内有至少 1 次写,则触发快照
save 900 1
save 300 10
save 60 10000
```

这几行是"或"的关系:满足任意一条就触发。手动触发则有两个命令:

```bash
# 同步保存:主进程直接干活,期间阻塞所有请求(生产慎用)
redis-cli save

# 后台保存:fork 子进程去做,主进程几乎不受影响(推荐)
redis-cli bgsave
```

`save` 由主线程亲自序列化,数据量大时会卡死整个实例;`bgsave` 才是日常自动快照真正走的路径。

### fork 与写时复制(COW):RDB 的灵魂

`bgsave` 怎么做到"不阻塞主进程"?靠的是操作系统的 `fork()`。

`fork` 出一个子进程,子进程**拥有和父进程一模一样的内存视图**。子进程负责把这份内存写进 RDB 文件,主进程继续对外服务。关键问题是:fork 之后主进程还在改数据,子进程看到的快照岂不是会被污染?

不会。这里依赖 **写时复制(Copy-On-Write,COW)**:fork 时父子进程共享同一份物理内存页,只有当某个页**被写入**时,内核才为修改方复制出一份新页。于是子进程始终看到的是 fork 那一刻的"冻结"视图,而主进程的新写入落在新复制的页上,互不干扰。

这也解释了 RDB 的一个隐性成本:**fork 期间如果写入很频繁,会触发大量页复制,内存占用可能翻倍,fork 本身在大内存实例上也会有明显延迟**。这是评估 RDB 时容易被忽略的点。

### RDB 的优劣

- 优点:文件是**紧凑的二进制**,体积小;恢复时直接把文件加载进内存即可,**恢复速度快**,适合做冷备和灾难恢复;对主进程性能影响小(借助 fork)。
- 缺点:它是**周期性**快照,两次快照之间的数据是裸露的。如果在下一次 `bgsave` 之前宕机,**这段时间的写入全部丢失**。按上面的 `save 900 1`,最坏情况可能丢掉接近 15 分钟的数据。

一句话:RDB 牺牲"数据新鲜度"换取"恢复效率"。

## AOF:命令追加日志

AOF(Append Only File)换了个思路:不存结果,存过程。每执行一条会修改数据的写命令,就把这条命令**追加**到 AOF 文件末尾。重启时,Redis 重新执行一遍文件里的所有命令,数据就回来了。

```conf
appendonly yes
appendfilename "appendonly.aof"
```

### appendfsync:三种刷盘策略,安全与性能的核心权衡

写命令追加到 AOF,并不是直接写到磁盘——它先进入操作系统的内核缓冲区,内核会择机真正落盘。什么时候调用 `fsync()` 强制落盘,就是 AOF 数据安全性的关键开关:

```conf
appendfsync everysec
```

| 策略 | 含义 | 最多丢失 | 性能 |
| --- | --- | --- | --- |
| `always` | 每条写命令都 fsync | 几乎不丢 | 最慢,每次写都等磁盘 |
| `everysec` | 每秒 fsync 一次(后台线程) | 约 1 秒数据 | 折中,**默认推荐** |
| `no` | 交给操作系统决定何时刷 | 取决于内核(可能数十秒) | 最快 |

这张表就是"数据安全 vs 性能"权衡的浓缩版。`always` 给你近乎不丢的保证,但把 Redis 拖成了同步磁盘 IO 的速度;`no` 跑得最快却把命运交给了内核;`everysec` 是绝大多数场景的最优解——最坏丢 1 秒,性能损耗可控。

### AOF 重写(rewrite):防止文件无限膨胀

AOF 记录的是"过程",问题随之而来:对同一个 key 反复操作,日志会越积越长。比如对一个计数器 `INCR` 一万次,AOF 里就有一万行,但其实只需要一条 `SET counter 10000` 就能表达最终状态。

**AOF 重写**就是干这件事:它不读旧的 AOF 文件,而是**直接遍历当前内存数据**,生成一份能产生等价数据的最小命令集,写成新 AOF 替换旧文件。重写同样通过 fork 子进程在后台进行,主进程在重写期间产生的新写命令会先缓存,重写完成后再追加,保证不丢。

```conf
# 当 AOF 文件比上次重写后大了一倍,且超过 64MB 时自动重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

也可以手动触发:

```bash
redis-cli bgrewriteaof
```

### AOF 的优劣

- 优点:**数据安全性高**(`everysec` 最多丢 1 秒);文件是命令文本,可读性强,误操作时甚至能手动编辑修复(比如删掉一条误执行的 `FLUSHALL`)。
- 缺点:相同数据集下**文件通常比 RDB 大**;恢复时要逐条重放命令,**恢复速度比 RDB 慢**;`always` 模式下写性能下降明显。

## 混合持久化:4.0+ 的最优解

RDB 恢复快但易丢数据,AOF 安全但恢复慢、文件大。能不能两者兼得?Redis 4.0 给出了**混合持久化**:

```conf
aof-use-rdb-preamble yes
```

开启后,**AOF 重写时生成的文件不再是纯命令文本,而是"RDB 格式的全量数据 + 增量的 AOF 命令"**:文件头部是一段 RDB 二进制快照(代表重写那一刻的全量数据),尾部是重写之后新产生的写命令(AOF 格式)。

这样恢复时:先以 RDB 速度把头部全量数据快速加载,再重放尾部少量增量命令。**既拿到了 RDB 的快速恢复,又保留了 AOF 的低丢失**。Redis 7.0 之后默认就是开启的,这也是当前生产环境的主流推荐配置。

## RDB vs AOF 横向对比

| 维度 | RDB | AOF |
| --- | --- | --- |
| 存什么 | 数据快照(结果) | 写命令日志(过程) |
| 文件体积 | 小、紧凑 | 大(重写后改善) |
| 数据安全 | 可能丢分钟级数据 | `everysec` 丢约 1 秒 |
| 恢复速度 | 快 | 慢(逐条重放) |
| 对性能影响 | fork 时有瞬时压力 | 持续的 fsync 开销 |
| 适用场景 | 冷备、灾备、可容忍丢失 | 对数据完整性要求高 |

## 选型建议

不要陷入"二选一"的思维,真正的问题是**你的业务能容忍丢多少数据**:

- **缓存场景、数据可重建**:数据丢了能从数据库重新加载,RDB 足矣,甚至可以关掉持久化追求极致性能。
- **不能丢、或丢失代价高**:开启 AOF + `everysec`,并打开混合持久化。
- **追求高可用且不想丢数据**:**RDB 与 AOF 同时开启**。Redis 重启时若两者都存在,会优先用 AOF 恢复(因为更完整),RDB 则作为快速冷备和兜底。这是兼顾安全与运维便利的常见组合。

## 主从 + 哨兵架构下的取舍

在单机之外,持久化的角色会发生微妙的变化。

主从复制中,**全量同步本身就依赖 RDB**:主节点 `bgsave` 生成 RDB 发给从节点。所以即使你"不打算用 RDB 恢复",主从场景下 RDB 也常常是必需的。

一个高可用集群里,数据安全的第一道防线其实是**副本**而非磁盘——主挂了,哨兵会把从节点提升为新主。这带来一个常见的优化:**让主节点关闭或弱化持久化(减少 fork 抖动、提升吞吐),由从节点专门负责持久化(开 AOF/RDB)**。

但这里有个**致命陷阱**:如果主节点完全关闭持久化,主进程重启后内存为空,此时哨兵尚未察觉、复制链路自动重连,**从节点会被主节点的空数据集覆盖,全集群数据清零**。所以即便依赖副本,主节点也建议至少保留 RDB,或确保自动重启前有人工介入。安全性永远不该只压在一个机制上。

## 一点工程联想

在 AI/Agent 工程里,Redis 常被用作对话上下文缓存、向量检索的元数据存储、或多 Agent 间的任务队列与共享状态。这类数据有个特点:**重建成本不对称**。重新调用一次大模型生成上下文,既花钱又花时间;而一段被截断的对话历史一旦丢失,Agent 的"记忆"就断了。这种场景下,持久化策略就不该用对待普通缓存的随意态度——开启 AOF + 混合持久化,把"最多丢 1 秒"作为底线,往往比事后重算更划算。

## 小结

回到开头那个问题——重启后数据还在不在,取决于你怎么配:

- **RDB** 存快照,靠 fork + COW 不阻塞主进程,恢复快但可能丢分钟级数据。
- **AOF** 存命令日志,`appendfsync` 三档决定安全与性能的平衡,靠 rewrite 压缩体积。
- **混合持久化**(`aof-use-rdb-preamble`)是当前最优解:RDB 头部快速恢复 + AOF 尾部低丢失。
- 选型的核心是"能容忍丢多少数据",而非追求某个机制的优劣。
- 主从 + 哨兵下,副本是第一道防线,但主节点别裸奔,否则一次重启可能拖垮整个集群。

持久化配置从来不是越安全越好,而是在数据安全和性能之间,找到匹配你业务风险的那个点。
